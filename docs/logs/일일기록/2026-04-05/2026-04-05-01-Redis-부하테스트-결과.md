## 분석

### 응답 시간

- p50: Redis 있을 때 절반 수준으로 단축 (38.5 ms → 19.8 ms)
- p95: 절반 이상 단축 (189.8 ms → 92.8 ms)
- 최대값이 핵심: Redis 없음에서 **30,582 ms** 발생 — DB 커넥션 풀 경쟁으로 일부 요청이 수십 초 대기. Redis 있음에서는 최대 **249 ms**로 완전히 억제됨.

### 에러율

- Redis 없음: 40건 실패 (0.27%) — 고부하 구간(100~200 VU)에서 DB 타임아웃으로 발생한 것으로 추정
- Redis 있음: 에러 0건 — 메모리 캐시이므로 커넥션 경쟁 없음

### RPS (처리량)

- Redis 있을 때 28% 더 많은 요청 처리 (47.0 → 60.1 req/s)
- DB 조회가 없으므로 각 요청의 처리 시간이 짧아져 동일 시간 내 더 많은 요청 소화

### k6 threshold 결과

두 시나리오 모두 `p(95)<2000` threshold는 통과했으나, `rate<0.05` threshold는 시나리오 A에서 실패로 기록됨.

## k6란 무엇인가

k6는 **부하 테스트 도구**다. 실제 사용자가 서버에 요청을 보내는 상황을 코드로 시뮬레이션해서, 서버가 얼마나 잘 버티는지 측정한다.

JavaScript로 시나리오를 작성하면 k6가 그 시나리오를 수백 개의 가상 사용자(VU, Virtual User)가 동시에 실행하는 것처럼 서버에 요청을 보낸다.

### k6가 측정하는 것

k6는 HTTP 요청/응답 과정을 단계별로 쪼개서 각각의 시간을 측정한다.

```
요청 전체 시간 (http_req_duration)
├── http_req_blocked         — 커넥션 풀이 꽉 차서 기다린 시간
├── http_req_connecting      — TCP 연결 수립 시간
├── http_req_tls_handshaking — TLS(HTTPS) 핸드셰이크 시간
├── http_req_sending         — 요청 데이터를 서버로 전송한 시간
├── http_req_waiting         — 서버가 처리하고 첫 바이트 응답이 올 때까지 기다린 시간 (TTFB)
└── http_req_receiving       — 응답 데이터를 다 받는 데 걸린 시간
```

그 외에도:
- **RPS** — 1초에 처리한 요청 수
- **에러율** — 실패한 요청 비율
- **VU** — 현재 동시 접속 중인 가상 사용자 수
- **데이터 송수신량**

### 핵심 지표 (일반)

k6 테스트에서 항상 먼저 봐야 하는 지표 4가지:

| 지표 | 설명 | 보는 방법 |
|---|---|---|
| `http_req_duration` | 요청 전체 응답 시간 | p95, p99로 확인. 평균은 느린 요청을 숨기므로 신뢰하기 어려움 |
| `http_req_failed` | 에러율 | 부하가 올라갈 때 에러율이 튀는 지점이 서버 한계 |
| `http_reqs` (RPS) | 1초당 처리 요청 수 | RPS가 더 이상 늘지 않고 에러율이 오르면 포화 상태 |
| `http_req_waiting` | 순수 서버 처리 시간 (TTFB) | `http_req_duration`에서 네트워크/TLS를 제외한 값. 병목이 서버인지 네트워크인지 구분할 때 사용 |

나머지(`http_req_sending`, `http_req_receiving`, `http_req_connecting` 등)는 이상값이 보일 때만 들여다보는 보조 지표다.

#### `http_reqs` 상세

전체 테스트 동안 보낸 요청의 총 횟수와 초당 평균 처리 수다.

```
"http_reqs": {
    "count": 14568,  — 총 요청 수
    "rate": 46.99    — RPS (평균)
}
```

이 RPS는 k6가 보낸 속도이지 서버가 처리한 속도가 아니다. 서버가 느려지면 VU들이 응답을 기다리느라 다음 요청을 못 보내게 되어 RPS가 낮아진다. RPS가 낮다 = 서버가 느리다는 신호다.

#### `http_req_waiting` 상세

`http_req_waiting`은 VU 1명 기준으로 측정한다. VU 20명이면 각자 독립적으로 요청을 보내고, 각 VU마다 개별적으로 시간을 측정한 뒤 집계한다.

```
VU가 요청을 서버로 전송 완료
           ↓
   ← http_req_waiting 시작 →
           ↓
서버가 응답의 첫 번째 바이트를 보냄    ← 서버 처리 완료 시점
           ↓
   ← http_req_receiving 시작 →
           ↓
      응답 전체 수신 완료
```

서버가 DB 조회 등 처리를 다 끝내야 응답을 시작할 수 있으므로, waiting = 서버가 처리하느라 걸린 시간이다. 20명이 동시에 요청하면 서버에 요청 20개가 한꺼번에 들어오기 때문에 각 VU의 waiting이 길어지고, 이 값들의 평균/p95/p99가 결과에 집계된다.

### 이번 테스트에서의 의미

결과에서 `http_req_waiting`이 `http_req_duration`과 거의 같았다. 이는 병목이 **서버 처리 시간**에 있었다는 의미다. 네트워크나 TLS는 문제가 없었고, Redis 없을 때 서버가 DB 조회를 처리하는 데 시간이 걸린 것이다.

## 시나리오 A 에러 40건 원인 분석

### DB 커넥션 풀 고갈

FastAPI는 DB 연결을 미리 만들어둔 커넥션 풀에서 꺼내 씁니다. `backend/app/db/session.py`에서 `create_engine(settings.DATABASE_URL)` — 별도 옵션 없이 SQLAlchemy 기본값을 사용하고 있다.

| 항목 | 기본값 |
|---|---|
| 풀 크기 | 5개 |
| 최대 overflow | 10개 |
| 풀 타임아웃 | **30초** |

동시 요청이 **15개(5+10)를 초과**하면 그 이후 요청들은 커넥션을 기다리고, 30초 대기 후에도 커넥션을 받지 못하면 예외가 발생한다.

```
커넥션 풀 (최대 15개)
├── 요청 1~15 → 커넥션 획득 → DB 조회 중...
└── 요청 16~ → 커넥션 없음 → 대기 → 30초 초과 → 에러
```

### 이번 테스트와의 연결

시나리오 A의 최대 응답시간이 **30,582 ms (~30초)** 였던 것이 이 타임아웃과 일치한다. 커넥션 풀 대기 30초 후 예외가 터진 것이다. Redis가 있으면 커넥션 풀을 아예 사용하지 않으므로 이 문제 자체가 발생하지 않는다.

---

## 결론

Redis 캐시가 응답 속도, 처리량, 안정성 모두에서 유의미한 차이를 만들었다.
특히 **최대 응답시간(30초 → 0.25초)** 과 **에러 완전 제거**는 고부하 상황에서 Redis의 핵심 역할을 수치로 입증한다.
