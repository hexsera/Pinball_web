## 목표

- 핀볼 게임 음악/이미지 등 큰 정적 리소스를 첫 게임 시작 시 다운로드 후 캐시
- 파일이 바뀔 때만 재다운로드, 그 외에는 캐시 재사용
- 불필요한 트래픽 절감

## 현재 상태

- 정적 리소스는 `frontend/public/audio/`, `frontend/public/images/` 에 위치
- 빌드 시 `dist/`에 그대로 복사됨 (번들에 포함되지 않음 — 구조는 올바름)
- nginx.conf에 `Cache-Control` 헤더 없음 → 브라우저가 매번 재다운로드

### 주요 파일 크기

| 파일 | 크기 |
|------|------|
| audio/back_bgm.mp3 | 4.3MB |
| images/pinball_back.png | 2.2MB |
| audio/ball_hit.wav | 564KB |
| images/newbg.jpg | 564KB |
| audio/lifedown.wav | 368KB |
| 기타 | 소규모 |

## 구현 방향

### 1단계 — nginx Cache-Control 설정

```nginx
# index.html — 항상 최신 확인
location = /index.html {
    add_header Cache-Control "no-cache";
}

# JS/CSS 번들 — Vite가 해시를 파일명에 붙이므로 장기 캐시 안전
location ~* \.(js|css)$ {
    add_header Cache-Control "max-age=31536000, immutable";
}

# 정적 리소스 (audio, images) — 장기 캐시
location ~* \.(mp3|wav|ogg|png|jpg|webp|gif|svg)$ {
    add_header Cache-Control "max-age=31536000, immutable";
}
```

**캐시 무효화 전략:**
- JS/CSS: Vite 빌드 시 파일명에 해시 자동 삽입 → 변경 시 파일명이 달라져 자동 무효화
- audio/images: `public/`의 파일은 해시 미적용 → 파일 변경 시 파일명에 버전 수동 추가 필요 (예: `back_bgm_v2.mp3`)

### 2단계 — Lazy Load (게임 시작 시점에만 다운로드)

현재는 브라우저가 페이지 진입 시 자동 로드. 게임 시작 버튼 클릭 시점에 명시적으로 fetch하여 캐시에 올리는 방식으로 변경.

```js
// 게임 시작 버튼 클릭 핸들러에서
const preloadAssets = async () => {
  const assets = ['/audio/back_bgm.mp3', '/images/pinball_back.png', ...]
  await Promise.all(assets.map(url => fetch(url)))
}
```

첫 다운로드 이후에는 브라우저 캐시에서 즉시 로드.

## 적용 대상 nginx 파일

- 실제 배포: `nginx.conf` (루트) — Docker nginx 컨테이너에 마운트됨
- 로컬 테스트용: `nginx/nginx.conf`

## 동작 원리

### 정적 파일 GET 엔드포인트

별도 API 코드 없이 nginx가 자동으로 파일 경로를 GET 엔드포인트로 노출한다.

```
nginx root → /etc/nginx/html (dist/ 파일들이 복사된 곳)

dist/audio/back_bgm.mp3       →  GET /audio/back_bgm.mp3
dist/images/pinball_back.png  →  GET /images/pinball_back.png
dist/assets/main.a3f7c2.js    →  GET /assets/main.a3f7c2.js
```

FastAPI 등 백엔드 코드 수정 불필요. nginx.conf에 Cache-Control 헤더만 추가하면 됨.

### 음악/이미지가 핀볼 페이지 진입 시 로드되는 이유

JS 번들과 정적 파일은 로드 시점이 다르다.

- **JS 번들**: `import` 문으로 번들에 포함 → 첫 접속 시 전부 다운로드
- **audio/images**: JS 코드가 실행될 때 브라우저가 필요한 시점에 서버에 요청

```
핀볼 페이지 진입
    → React가 Pinball 컴포넌트 마운트
    → JS 코드 실행: new Audio('/audio/back_bgm.mp3')
    → 브라우저: GET /audio/back_bgm.mp3 요청
    → nginx가 파일 응답
```

캐시 미적용 시 핀볼 페이지 진입마다 위 흐름 반복.

### Cache-Control 적용 후 동작

```
첫 방문: 서버에서 파일 받음 → 브라우저 로컬에 저장
재방문: 로컬 캐시 유효 → 서버 요청 없이 즉시 로드
```

Network 탭에서 캐시 재사용 시 파일 옆에 `(disk cache)` 또는 `(memory cache)` 표시됨.
`max-age=31536000` = 브라우저에게 "1년간 유효"라고 알려주는 것.

## 현재 nginx 설정 상태

컨테이너 내부에 두 개의 설정이 동시에 존재:
- `/etc/nginx/nginx.conf` — 프로젝트에서 만든 설정. `root /etc/nginx/html` (dist 파일 실제 서빙 위치)
- `/etc/nginx/conf.d/default.conf` — nginx 이미지 기본 설정. `root /usr/share/nginx/html`

둘 다 `Cache-Control` 헤더 없음 → 브라우저 휴리스틱 캐시에 맡기는 상태.
Cache-Control 설정은 `nginx.conf`에 추가하면 됨.

## 브라우저 휴리스틱 캐시

nginx에 `Cache-Control`을 설정하지 않아도 브라우저가 자체적으로 캐시를 적용하는 동작.

**작동 원리:**
서버가 `Last-Modified` 헤더를 응답에 포함하면 브라우저가 아래 공식으로 캐시 유효기간을 자체 계산:

```
캐시 유효기간 = (현재시각 - Last-Modified) × 0.1
```

파일이 오래전에 마지막으로 수정됐을수록 "자주 안 바뀌는 파일"로 판단 → 더 오래 캐시.

**audio/images가 잘 캐시되는 이유:**
정적 파일이라 `Last-Modified`가 오래됨 → 브라우저가 길게 캐시.

**휴리스틱 캐시의 문제:**
- 브라우저 재량이라 동작이 일관적이지 않음 (`Ctrl+Shift+R`에도 캐시 유지되는 등)
- 브라우저마다 동작이 다를 수 있음
- 서버가 캐시 만료 시점을 제어할 수 없음

**Cache-Control 명시 설정의 이유:**
`Cache-Control: max-age=31536000` 설정 시 "1년간 유효"를 서버가 직접 보장 → 동작 예측 가능, 일관성 확보.

## 휴리스틱 캐시 확인 — ball_hit.wav

크롬 개발자 도구 Network 탭에서 핀볼 음악 파일(`/audio/ball_hit.wav`) Response Headers 직접 확인.

```
date:          Mon, 18 May 2026 11:34:02 GMT
etag:          "6a0af4a8-8c638"
last-modified: Mon, 18 May 2026 11:14:48 GMT
server:        nginx/1.29.4
Cache-Control: (없음)
```

- `Cache-Control` 없음 → 휴리스틱 캐시 작동 중 확인
- `Last-Modified` + `ETag` 있음 → 캐시 만료 후 조건부 요청에 사용됨
  - 브라우저가 서버에 `if-modified-since`, `if-none-match` 헤더로 "바뀌었어?" 질문
  - 안 바뀌면 `304 Not Modified` (바디 없음), 바뀌면 `200 OK` + 새 파일

**Cache-Control 명시 후 달라지는 점:**
`max-age=31536000` 설정 시 브라우저가 유효기간 내에는 서버에 조건부 요청 자체를 보내지 않음 → ETag 확인 트래픽도 완전히 제거됨.

## 실험 기록

### Before (캐시 적용 전)

**측정 방법:** 크롬 개발자 도구 Network 탭, `Ctrl+Shift+R` (강제 새로고침, 캐시 무시)

| 페이지 | requests | transferred | resources | Finish |
|--------|----------|-------------|-----------|--------|
| 메인 페이지 (`/`) | 9 | 1.9MB | 2.2MB | 463ms |
| 핀볼 페이지 (`/pinball`) | 16 | 1.9MB | 6.6MB | - |
| 핀볼 페이지 — Disable cache | 18 | 6.5MB | 6.7MB | 825ms |

**분석:**
- 메인과 핀볼의 transferred는 같지만 resources 차이(2.2MB vs 6.6MB)가 핀볼 진입 시 audio/image 파일이 추가 로드됨을 의미
- transferred < resources 격차는 일부 파일에 gzip 압축이 적용되고 있음을 의미
- `Ctrl+Shift+R` 기준이므로 현재 캐시 설정 없이 매 접속마다 이만큼 재다운로드 중
- **캐시 효과: transferred 기준 6.5MB → 1.9MB, 약 4.6MB 절감**

**측정 2: 브라우저 기본 캐시 동작 확인**

조건: 핀볼 페이지 이전 방문으로 audio/images 캐시된 상태 → 메인 페이지 `Ctrl+Shift+R` → 핀볼 페이지 이동

| requests | transferred | resources |
|----------|-------------|-----------|
| 12 | 569B | 4.4MB |

**분석:**
- transferred 569B — JS/HTML 등 일부만 서버에서 받음
- resources 4.4MB — audio/images는 브라우저 캐시에서 로드 (서버 요청 없음)
- nginx에 Cache-Control 미설정이지만 브라우저가 자체적으로 캐시를 유지하고 있음
- `Ctrl+Shift+R`은 현재 페이지 캐시만 무효화하고, 다른 페이지에서 캐시된 리소스는 유지됨
- nginx Cache-Control 설정 시 이 동작을 **명시적으로 제어**할 수 있게 됨 (만료 기간, 무효화 타이밍 등)

### 실험 1 — 메인 페이지 컴포넌트 수정 후 핀볼 페이지 리소스 영향 확인

**조건:**
- HomePage.jsx 텍스트 수정 (핀볼 로직/리소스 경로 무관)
- 빌드 후 nginx 배포
- 핀볼 페이지에서 `Ctrl+Shift+R`

**빌드 해시 변화:**
```
수정 전: index-B_HNKD-y.js
수정 후: index-DmxjqMDJ.js  ← JS 해시 바뀜
CSS:     index-Dww-4alG.css  ← 그대로 (CSS 변경 없음)
```

**측정 결과:**

| requests | transferred | resources |
|----------|-------------|-----------|
| 14 | 940B | 4.4MB |

**분석:**
- transferred 940B — 새 JS 번들(해시 변경)만 새로 받음
- resources 4.4MB — audio/images는 URL 변경 없으므로 캐시 재사용
- 메인 페이지만 수정해도 JS 번들 해시가 바뀌어 번들은 재다운로드됨
- 단 정적 리소스(audio/images)는 영향 없음 → 캐시 유지

**부수 관찰:**
- 배포 후 `F5`로 변경사항이 반영되는 경우도 있고 안 되는 경우도 있음
- nginx에 Cache-Control 미설정 상태라 브라우저 재량 캐시가 작동 중 → 일관성 없는 동작의 원인
- `Ctrl+Shift+R`은 항상 최신 내용을 받으므로 확실한 확인 방법

### 실험 2 — 핀볼 컴포넌트 수정 후 핀볼 페이지 리소스 영향 확인

**조건:**
- Pinball.jsx 텍스트 수정 + 의미없는 JS 코드 한 줄 추가 (게임 로직/리소스 경로 무관)
- 빌드 후 nginx 배포
- 핀볼 페이지에서 `Ctrl+Shift+R`

**빌드 해시 변화:**
```
수정 전: index-DmxjqMDJ.js
수정 후: index-G5BvzhjO.js  ← JS 해시 바뀜
```

**측정 결과:**

| requests | transferred | resources | Finish | Load |
|----------|-------------|-----------|--------|------|
| 22 | 1.9MB | 6.6MB | 1.5min | 314ms |

**분석:**
- transferred 1.9MB — 핀볼 페이지에서 `Ctrl+Shift+R`했으므로 새 JS 번들 포함해서 재다운로드
- resources 6.6MB — baseline과 동일. audio/images는 캐시 재사용
- Finish 1.5min은 음악 파일 로드가 느린 게 아니라 게임 시작 대기 상태에서 측정된 것 (Load 314ms가 실제 로드 시간)
- 실험 1과 transferred 차이(940B vs 1.9MB)는 측정 위치 차이: 실험 1은 메인→핀볼 이동(JS 캐시 일부 유지), 실험 2는 핀볼 페이지에서 강제 새로고침(JS 전체 재다운로드)

**결론:**
- 어느 컴포넌트를 수정하든 JS 번들 해시는 바뀜 → JS는 재다운로드
- audio/images URL은 변경 없으므로 항상 캐시 재사용
- nginx Cache-Control 설정 전에도 브라우저 재량 캐시로 정적 리소스는 보호되고 있음

**측정 방법 보완:**
- `Ctrl+Shift+R`은 현재 페이지 캐시를 무시하고 요청을 보내지만, 서버 응답을 다시 캐시에 저장하므로 audio/images가 항상 캐시된 것처럼 보이는 현상이 있음
- 정확한 "캐시 없을 때" baseline은 개발자 도구 Network 탭 → `Disable cache` 체크 후 측정해야 함

## 음악/이미지 리소스 서빙 방식

**소규모 게임 (현재 프로젝트)**
- `public/audio`에 파일을 두고 nginx가 직접 서빙
- 별도 API 불필요, 구현 단순
- 현재 프로젝트 규모에 적합한 방식

**대규모 게임 / 상용 서비스**
- S3 같은 오브젝트 스토리지(버킷)에 파일 보관
- CDN(CloudFront, Cloudflare 등)을 앞에 두고 서빙
- API는 파일 URL만 반환, 실제 파일은 CDN에서 직접 전달

```
브라우저 → API 서버: "BGM URL 줘"
API 서버 → 브라우저: "https://cdn.example.com/audio/bgm.mp3"
브라우저 → CDN: 파일 직접 다운로드
```

**API로 파일 자체를 스트리밍하는 경우**
- 스포티파이, 유튜브 뮤직 등 음악 서비스
- 저작권 보호, 사용자별 접근 제어가 필요할 때
- 게임에서는 거의 사용하지 않음
