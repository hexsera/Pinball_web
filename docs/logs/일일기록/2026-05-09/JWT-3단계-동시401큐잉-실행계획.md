# JWT 3단계 — 동시 401 큐잉 실행계획

## 요구사항 요약

**요구사항**: 페이지 로드 시 여러 API 요청이 거의 동시에 401(만료된 Access Token)을 받을 때, `/auth/refresh` 호출이 중복으로 나가지 않도록 첫 요청 1건만 갱신을 수행하고 나머지는 큐에 대기시킨 뒤 새 토큰으로 한 번에 재시도한다.

**목적**: 2단계 인터셉터는 401 한 건씩 독립적으로 갱신을 호출한다. 동시 다발 401이 발생하면 갱신 요청이 중복되어 자원이 낭비되고, 4단계 Refresh Rotation 도입 시 두 번째 갱신이 "이미 사용된 Refresh"로 판정되어 정상 사용자가 강제 로그아웃되는 사고가 난다. 4단계 Rotation의 안전한 도입을 위한 전제 조건이다.

---

## 현재상태 분석

- **`frontend/src/lib/api.js`**: 응답 인터셉터가 401 수신 시마다 독립적으로 `/auth/refresh`를 호출한다. 동시성 제어 없음.
- **재시도 가드**: `original._retry` 플래그로 같은 요청의 무한 루프는 막지만, 서로 다른 요청들이 동시에 401을 받는 경우는 처리하지 않는다.
- **백엔드**: 변경 불필요. 큐잉은 브라우저 한 개 안에서 발생하는 동시성 문제로, 백엔드 입장에서는 정상적인 갱신 요청 다수로 보일 뿐 묶을 단서가 없다.
- **테스트 환경**: `frontend/` 디렉토리에 Vitest 설치됨 (`npm run test:run`).

---

## 구현 방법

- **모듈 스코프 변수 두 개** 사용: `isRefreshing`(갱신 진행 플래그)과 `failedQueue`(대기 중 요청 콜백 배열).
- **첫 401**: `isRefreshing`을 `true`로 바꾸고 `/auth/refresh`를 호출한다.
- **이후 들어오는 401**: 갱신 결과를 기다리는 Promise를 만들어 콜백을 큐에 넣고 반환한다.
- **갱신 성공**: 큐에 쌓인 콜백들을 새 토큰으로 깨워 원 요청을 재시도시킨다.
- **갱신 실패**: 큐의 콜백들도 모두 reject 처리하고 로그아웃 화면으로 이동한다.
- **공통 마무리**: `finally`에서 `isRefreshing`을 `false`로 되돌리고 큐를 비운다.

## 구현 단계

### 1. 모듈 스코프 상태 변수 선언

```javascript
// frontend/src/lib/api.js 상단에 추가
let isRefreshing = false;
let failedQueue = [];

const processQueue = (error, token = null) => {
  failedQueue.forEach((prom) => {
    if (error) prom.reject(error);
    else prom.resolve(token);
  });
  failedQueue = [];
};
```

- **무엇을 하는가**: 갱신 진행 상태와 대기 큐를 인터셉터 바깥에 두어 모든 401 콜백이 같은 변수를 공유하게 만든다.
- `isRefreshing`: 현재 누군가 `/auth/refresh`를 호출 중인지 표시하는 깃발.
- `failedQueue`: 갱신 완료를 기다리는 요청들의 `{ resolve, reject }` Promise 핸들러 모음.
- `processQueue`: 갱신 결과를 큐의 모든 대기자에게 한 번에 통지하는 헬퍼. 성공 시 새 토큰을 전달, 실패 시 에러를 전파한다.

### 2. 응답 인터셉터에 큐잉 분기 추가

```javascript
api.interceptors.response.use(
  (res) => res,
  async (error) => {
    const original = error.config;
    if (error.response?.status !== 401 || original._retry) {
      return Promise.reject(error);
    }

    if (isRefreshing) {
      // 갱신이 이미 진행 중 → 큐에 등록하고 결과를 기다림
      return new Promise((resolve, reject) => {
        failedQueue.push({ resolve, reject });
      })
        .then((token) => {
          original.headers.Authorization = `Bearer ${token}`;
          return api(original);
        })
        .catch((err) => Promise.reject(err));
    }

    original._retry = true;
    isRefreshing = true;

    try {
      const { data } = await axios.post('/api/v1/auth/refresh', {}, { withCredentials: true });
      useAuthStore.getState().setAccessToken(data.access_token);
      processQueue(null, data.access_token);
      original.headers.Authorization = `Bearer ${data.access_token}`;
      return api(original);
    } catch (refreshError) {
      processQueue(refreshError, null);
      useAuthStore.getState().clearAccessToken();
      window.location.href = '/login';
      return Promise.reject(refreshError);
    } finally {
      isRefreshing = false;
    }
  }
);
```

- **무엇을 하는가**: 401 수신 시 첫 요청만 갱신을 수행하고 나머지는 큐에 줄 세웠다가 갱신 완료 후 일괄 재시도하도록 인터셉터를 재구성한다.
- **첫 분기 (`isRefreshing === true`)**: 새 Promise를 만들어 `resolve`/`reject` 핸들러를 큐에 넣고 반환한다. 이 Promise는 `processQueue`가 호출될 때까지 보류된다. 깨어나면 `then`에서 새 토큰을 헤더에 부착한 뒤 `api(original)`로 원 요청을 재시도한다.
- **두 번째 분기 (첫 401)**: `isRefreshing = true`로 깃발을 올린 뒤 `/auth/refresh`를 호출한다. 성공하면 `processQueue(null, token)`로 큐의 대기자들을 깨우고, 실패하면 `processQueue(err, null)`로 모두 reject한 뒤 로그아웃한다.
- **`finally`**: 성공·실패 관계없이 `isRefreshing`을 반드시 `false`로 되돌려 다음 만료 사이클에서 갱신이 다시 시작될 수 있게 한다.
- **`original._retry` 가드**: 갱신 후 재시도된 요청이 또 401을 받으면(서버 측 비정상 상태) 무한 루프 없이 즉시 reject된다.

### 3. (선택) Vitest 단위 테스트 추가

```javascript
// frontend/src/lib/api.test.js
import { describe, it, expect, vi, beforeEach } from 'vitest';
import MockAdapter from 'axios-mock-adapter';
import axios from 'axios';
import api from './api';
import { useAuthStore } from '../store/authStore';

describe('api 인터셉터 — 동시 401 큐잉', () => {
  let mock;
  beforeEach(() => {
    mock = new MockAdapter(axios);
    useAuthStore.getState().setAccessToken('expired-token');
  });

  it('동시 401 요청 4건에 대해 /auth/refresh는 1번만 호출된다', async () => {
    let refreshCallCount = 0;
    mock.onPost('/api/v1/auth/refresh').reply(() => {
      refreshCallCount += 1;
      return [200, { access_token: 'new-token', token_type: 'bearer' }];
    });
    mock.onGet(/\/api\/v1\/.*/).reply((config) => {
      const auth = config.headers.Authorization;
      return auth === 'Bearer new-token' ? [200, { ok: true }] : [401];
    });

    const results = await Promise.all([
      api.get('/users/me'),
      api.get('/scores/monthly'),
      api.get('/scores/high'),
      api.get('/friends'),
    ]);

    expect(refreshCallCount).toBe(1);
    expect(results.every((r) => r.data.ok)).toBe(true);
  });
});
```

- **무엇을 하는가**: `axios-mock-adapter`로 4개 요청이 동시에 401을 받는 상황을 재현해 `/auth/refresh`가 정확히 1회만 호출되고 4개 요청이 모두 새 토큰으로 성공하는지 검증한다.
- 설치 필요: `npm install -D axios-mock-adapter`.
- 테스트가 부담스러우면 이 단계는 생략하고 수동 검증만 진행해도 된다.
