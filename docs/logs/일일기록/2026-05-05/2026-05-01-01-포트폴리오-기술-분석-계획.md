## 목적

취업 포트폴리오를 위해 이 프로젝트에서 어필할 수 있는 기술 스택, 아키텍처 설계, 개발 방법론을 체계적으로 정리한다.

## 1. 현황 요약 (분석 완료)

프로젝트를 전수 분석한 결과, 포트폴리오로 어필 가능한 항목을 인프라 / 프론트엔드 / 백엔드 3개 영역으로 분류했다.

## 2. 인프라

### 2-1. 기술 스택

| 기술 | 버전/비고 |
|------|-----------|
| Docker + Docker Compose | 컨테이너 기반 배포 |
| Traefik | SSL/TLS, Cloudflare DNS-01 챌린지 |
| Nginx | SPA 정적 서빙, HTML5 폴백 |
| PostgreSQL | 16, 영속 데이터 |
| Redis | 8, 게임 세션 (TTL 2시간) |
| MySQL | 8, 레거시 백업용 |

### 쓸만한것
Docker + Traefik + Nginx + PostgreSQL + Redis

### 2-2. 아키텍처

#### (M) 컨테이너 기반 배포 아키텍처
```
브라우저
  → Traefik (80/443, SSL termination, Cloudflare DNS-01)
      → Nginx (정적 SPA, /api 제외)
      → FastAPI (PathPrefix /api)
          → PostgreSQL
          → Redis
```
- Traefik 라벨 기반 라우팅으로 설정 파일 없이 서비스 디스커버리
- 환경변수를 `.env` 파일로 분리하여 민감정보 소스코드 제외
- 근거 파일: [docker-compose.yml](../../docker-compose.yml)

#### (E) Mixed Content / HTTPS 이슈 해결
- FastAPI `redirect_slashes=False` + 라우터 경로 후행 슬래시 제거로 307 리다이렉트 차단
- HTTPS 환경에서 HTTP 리다이렉트 발생 시 브라우저가 Mixed Content 오류 발생하는 문제 직접 해결
- 근거: [CLAUDE.md](../../CLAUDE.md) 주요 설계 결정 섹션

#### (F) 혼합 스토리지 전략 (Polyglot Persistence)
- PostgreSQL: 영속 데이터 (유저, 점수, 친구 관계)
- Redis: 휘발성 게임 세션 (TTL 2시간, 게임 중 상태 저장)
- IndexedDB: 브라우저 로컬 공지 캐시 (서버 요청 없이 읽기)
- 근거 파일: [backend/app/api/v1/game_sessions.py](../../backend/app/api/v1/game_sessions.py)

### 3-2. 아키텍처

#### (I) React Context API를 이용한 전역 인증 상태 관리
- `AuthContext` + `AuthProvider`로 로그인 상태를 앱 전역에서 공유
- `localStorage` 기반 세션 유지 (새로고침 후 복원)
- `useAuth()` 커스텀 훅으로 컴포넌트에서 간편하게 접근
- 근거 파일: [frontend/src/contexts/AuthContext.jsx](../../frontend/src/contexts/AuthContext.jsx)

#### (J) Vite 개발 서버 프록시
- `/api` 경로를 `localhost:8000`으로 프록시 → 개발 중 CORS 없이 백엔드 연동
- 프로덕션(Nginx)과 동일한 URL 구조 유지 → 환경별 분기 코드 불필요
- 근거 파일: [frontend/vite.config.js](../../frontend/vite.config.js)

#### (G-2) 프론트엔드 서비스 레이어 + IndexedDB 폴백
- `noticeService.js`가 API 호출을 추상화, 실패 시 IndexedDB에서 로컬 데이터 반환
- 백엔드 장애 상황에서도 공지 목록 조회 가능
- 근거 파일: [frontend/src/services/noticeService.js](../../frontend/src/services/noticeService.js)

#### (L) WebGL GLSL 셰이더 직접 작성 (Aurora 배경)
- `ogl` 라이브러리로 WebGL 컨텍스트 관리
- Simplex Noise 기반 GLSL Fragment Shader를 직접 작성하여 동적 배경 이펙트 구현
- 근거 파일: [frontend/src/components/Aurora/Aurora.jsx](../../frontend/src/components/Aurora/Aurora.jsx)

### 3-3. 핵심 게임 기술

### 쓸만한거
FastAPI + SQLAlchemy + Authlib + pwdlib + Faker + pytest

### 4-2. 아키텍처

#### (A) RESTful API 설계
- HTTP 메서드를 의미에 맞게 사용: `POST`(생성), `GET`(조회), `PUT`(수정), `DELETE`(삭제)
- URL은 리소스 명사 중심: `/api/v1/users`, `/api/v1/monthly-scores`, `/api/v1/game-sessions`
- 적절한 HTTP 상태코드 반환: 201(Created), 400(Bad Request), 401(Unauthorized), 404(Not Found)
- 근거 파일: [backend/app/api/v1/](../../backend/app/api/v1/)

#### (B) API 버전 관리
- 모든 엔드포인트에 `/api/v1/` 접두사 적용
- 헬스체크(`/api/`)와 친구 API(`/api/friend-requests/`)는 버저닝 예외 — 이유: 초기 설계 잔재, 이슈로 인지하고 있음
- 근거 파일: [backend/main.py](../../backend/main.py) L47-55

#### (C) 레이어 분리 (관심사 분리)
```
main.py (진입점/라우터 등록)
├── app/api/v1/          # 라우터 레이어 (HTTP 핸들러)
├── app/schemas/         # Pydantic 스키마 (요청/응답 DTO)
├── app/core/            # 공통 로직 (security, config, jwt)
├── app/db/              # DB 세션 관리
└── models.py            # SQLAlchemy ORM 모델
```
- Request/Response DTO를 Pydantic 스키마로 분리 → 응답에서 `password` 필드 자동 제거
- DB 세션을 `deps.py`의 `get_db()` 의존성으로 추상화 → 모든 라우터에서 재사용
- 근거 파일: [backend/app/api/deps.py](../../backend/app/api/deps.py)

#### (D) 의존성 주입 (Dependency Injection)
- FastAPI의 `Depends(get_db)` 패턴으로 DB 세션 주입
- 테스트 시 `app.dependency_overrides[get_db]`로 테스트 DB로 교체 → 프로덕션 코드 수정 없이 테스트 격리
- 근거 파일: [backend/tests/conftest.py](../../backend/tests/conftest.py) L57-69

#### (E) 비정규화를 통한 읽기 성능 최적화
- `MonthlyScore` 테이블에 `nickname` 컬럼을 중복 저장 (비정규화)
- 랭킹 조회 시 users 테이블 JOIN 없이 빠른 응답
- 닉네임 변경 시 `PUT /users/{user_id}`에서 동기화 처리
- 근거 파일: [backend/app/api/v1/users.py](../../backend/app/api/v1/users.py) L113-116

#### (G-1) Redis Sorted Set 랭킹 캐시 + DB 폴백
- `monthly_scores` 조회 시 Redis `ZREVRANGEBYSCORE`로 먼저 읽기
- Redis 캐시가 비어있으면 DB에서 읽어 Sorted Set 재구성 (warm-up)
- Redis 장애 시 자동으로 DB 직접 조회로 폴백 → 단일 장애점 제거
- 점수 등록/수정/삭제 시 Redis Sorted Set도 동기 업데이트
- 근거 파일: [backend/app/api/v1/monthly_scores.py](../../backend/app/api/v1/monthly_scores.py) L42-138

#### (H) SQLAlchemy aliased JOIN (친구 관계 양방향 조회)
- `aliased(User)`로 requester/receiver를 각각 다른 테이블 별칭으로 처리
- 단일 쿼리에서 두 사용자의 닉네임을 동시에 조회
- 근거 파일: [backend/app/api/v1/friends.py](../../backend/app/api/v1/friends.py) L127-139

#### (K) Alembic DB 마이그레이션
- 스키마 변경을 코드로 관리 (버전 이력 추적)
- `alembic/env.py`에서 환경변수로 DB URL 주입 → 환경별 설정 분리
- 근거 파일: [backend/alembic/env.py](../../backend/alembic/env.py)

## 6. 발견된 개선 가능 포인트 (선택사항)

포트폴리오 어필 전 보완하면 좋은 항목들:

- `app/api/v1/` 모듈들이 `/code` 경로를 `sys.path`로 직접 추가 — 패키지 구조 정리 여지
- 친구 API가 `/api/friend-requests/`로 버저닝 미적용 — `/api/v1/` 통일 가능
- `chat.py`의 `_sessions` 딕셔너리가 서버 메모리에만 저장 — 재시작 시 세션 소멸 (Redis로 이전 가능)
- 이미 `docs/issue/2026-04-05-권한-검증-미비.md`에 기록된 권한 검증 이슈 존재
