# JWT 흐름 — 발급부터 API 처리까지

> 목적: JWT가 서버에서 발급되어 프론트엔드를 거쳐 다시 API로 돌아오는 전체 흐름을 단계별로 따라간다.

---

## 전체 흐름 한눈에 보기

```
[클라이언트]          [프론트엔드]              [백엔드]
  사용자 로그인 →  POST /api/v1/auth/login  →  토큰 발급
                  ←  { access_token }       ←
                     authStore에 저장
                     (localStorage 임시 보관)

  API 요청 →      api.js 인터셉터가           →  get_current_user()가
                  Authorization: Bearer       JWT 디코딩 → 사용자 조회
                  헤더 자동 부착              → 핸들러 실행
```

## 1단계: 백엔드에서 JWT 발급

**경로:** `backend/app/api/v1/auth.py` → `backend/app/core/security.py`

로그인 요청이 들어오면 백엔드는 비밀번호를 검증한 뒤 `create_access_token()`을 호출해 토큰을 만든다.

```python
# backend/app/core/security.py (핵심 로직)
from jose import jwt
from datetime import datetime, timedelta

def create_access_token(data: dict) -> str:
    to_encode = data.copy()
    expire = datetime.utcnow() + timedelta(minutes=settings.JWT_EXPIRE_MINUTES)
    to_encode.update({"exp": expire})
    return jwt.encode(to_encode, settings.JWT_SECRET_KEY, algorithm=settings.JWT_ALGORITHM)
```

- `data`에는 보통 `{"sub": str(user.id)}`가 들어간다. (`sub` = subject, JWT 표준 클레임)
- `JWT_SECRET_KEY`로 서명되어, 이 키를 모르면 위조 불가.
- `JWT_EXPIRE_MINUTES=10080` (7일) 으로 수명 설정.

발급된 토큰은 응답에 담겨 프론트엔드로 전달된다.

## 1.5단계: 백엔드가 응답에 토큰을 담아 전송

**경로:** `backend/app/api/v1/auth.py` → `_create_login_response()`

로그인(`/login`), 회원가입(`/register`), 구글 로그인(`/auth/google`) 세 엔드포인트 모두 이 함수를 공통으로 호출한다.

```python
# backend/app/api/v1/auth.py
def _create_login_response(user, response: Response) -> LoginResponse:
    # 1) Access Token 생성 — user_id, email, role을 payload에 담아 서명
    access_token = create_access_token({"sub": str(user.id), "email": user.email, "role": user.role})

    # 2) Refresh Token 생성 — UUID 기반 불투명 토큰, Redis에 저장
    refresh_token = create_refresh_token()
    redis_client.setex(f"refresh:{refresh_token}", REFRESH_TTL, str(user.id))

    # 3) Refresh Token은 HttpOnly 쿠키로 전송 (JS에서 접근 불가)
    response.set_cookie(
        key="refresh_token",
        value=refresh_token,
        httponly=True,   # XSS 방어
        secure=True,     # HTTPS 전용
        samesite="lax",
        max_age=REFRESH_TTL,
        path="/api/v1/auth",  # /api/v1/auth/* 요청에만 쿠키 전송
    )

    # 4) Access Token은 JSON body에 담아 반환
    return LoginResponse(
        message="Login successful",
        user_id=user.id,
        email=user.email,
        nickname=user.nickname,
        role=user.role,
        access_token=access_token,   # ← 프론트가 이 값을 authStore에 저장
        token_type="bearer",
    )
```

실제로 프론트엔드가 받는 HTTP 응답:

```
HTTP/1.1 200 OK
Set-Cookie: refresh_token=<uuid>; HttpOnly; Secure; SameSite=Lax; Path=/api/v1/auth

{
  "message": "Login successful",
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "bearer",
  "user_id": 42,
  "nickname": "hexsera",
  "role": "user",
  "email": "..."
}
```

**포인트:**
- Access Token → JSON body (프론트가 읽어 저장)
- Refresh Token → HttpOnly 쿠키 (브라우저가 자동 관리, JS 접근 불가)

## 2단계: 프론트엔드에서 토큰 수신 및 저장

**경로:** `Login.jsx` → `AuthContext.jsx` → `authStore.js`

```
login() 호출
  └─ AuthContext.login(userData, accessToken)
       ├─ setUser(userData)          // React 상태 업데이트
       ├─ localStorage('user', ...)  // 새로고침 대비 사용자 정보 저장
       └─ authStore.setAccessToken(token)
              ├─ Zustand 메모리에 저장  // 컴포넌트 구독 가능
              └─ localStorage('access_token', token)  // 새로고침 대비
```

**왜 Zustand를 쓰는가?**
- axios 인터셉터는 React 훅(`useContext`) 사용 불가.
- `useAuthStore.getState()`로 React 외부(인터셉터)에서도 토큰 읽기 가능.

**왜 localStorage에도 저장하는가?**
- 새로고침 시 Zustand 메모리는 초기화된다.
- 1단계 한정 타협: Refresh Token 없이 운영하므로 localStorage에 임시 보관.
- 보안 이슈(XSS)가 있지만, 2단계(Refresh Token + HttpOnly 쿠키)에서 개선 예정.

## 3단계: API 요청 시 토큰 자동 부착

**경로:** 어떤 컴포넌트든 `api.js` 인스턴스 사용 → 인터셉터 실행

### 인터셉터란?

인터셉터는 기능이라기보다 **패턴**에 가깝다. 요청/응답이 목적지에 도달하기 전에 반드시 거쳐가도록 등록해둔 처리 구간이다. axios가 강제로 가로채는 게 아니라, `api.get()`을 호출하면 axios 내부적으로 등록된 함수 목록을 순서대로 실행하고 나서 실제 요청을 보내는 파이프라인 구조다.

```
api.get('/users')
  → [요청 인터셉터] → 실제 HTTP 요청 전송
  ← [응답 인터셉터] ← 실제 HTTP 응답 수신
```

Django/FastAPI의 미들웨어, Spring의 필터, Express의 `app.use()`가 모두 같은 패턴이다. 이름만 다를 뿐 "모든 요청/응답이 공통 구간을 거쳐가게 만든다"는 설계다.

### 인터셉터가 없을 때 vs 있을 때

```js
// 인터셉터 없이 — 호출하는 쪽마다 토큰을 직접 붙여야 함
const token = localStorage.getItem('access_token');
await axios.get('/api/v1/users', {
  headers: { Authorization: `Bearer ${token}` }
});

// 인터셉터 있을 때 — api.js에 한 번만 등록, 이후 자동
await api.get('/users');  // 토큰은 인터셉터가 알아서 붙임
```

토큰을 붙이는 책임이 **호출하는 쪽마다** 있냐, **api.js 한 곳**에 있냐의 차이다.

### 코드 한 줄씩

```js
api.interceptors.request.use((config) => {
```
`api`(axios 인스턴스)의 요청 파이프라인에 함수를 등록한다. `api`로 요청이 나갈 때마다 이 함수가 실행된다. `config`는 요청 정보 전체(URL, 헤더, 바디 등)를 담은 객체다.

```js
  const token = useAuthStore.getState().accessToken;
```
Zustand 스토어에서 현재 저장된 토큰을 꺼낸다. `.getState()`를 쓰는 이유는 여기가 React 컴포넌트 바깥이라 훅(`useAuthStore()`)을 쓸 수 없기 때문이다.

```js
  if (token) config.headers.Authorization = `Bearer ${token}`;
```
토큰이 있을 때만 헤더에 추가한다. 비로그인 상태면 헤더를 건드리지 않는다.

```js
  return config;
```
수정한 `config`를 반환한다. 이걸 반환해야 axios가 이 설정으로 실제 요청을 보낸다. 반환하지 않으면 요청 자체가 진행되지 않는다.

### Authorization 헤더와 Bearer

`Authorization`은 개발자가 만든 이름이 아니라 **HTTP 표준(RFC)에 정의된 헤더**다. HTTP 헤더에는 두 종류가 있다:

- **표준 헤더** — RFC에 이름과 용도가 약속되어 있음. `Authorization`, `Content-Type`, `Accept` 등
- **커스텀 헤더** — 개발자가 임의로 만든 것. 관례적으로 `X-`로 시작. 이 프로젝트에서 기존에 쓰던 `X-API-Key`가 이것

표준 헤더를 쓰면 `HTTPBearer` 같은 프레임워크 내장 도구를 바로 쓸 수 있다는 장점이 있다.

`Authorization` 헤더의 값은 `<타입> <값>` 형식으로 약속되어 있다:

```
Authorization: Bearer eyJhbGci...
               ↑      ↑
               타입   값 (공백 하나로 구분)
```

이것은 별도의 key-value 구조가 아니라 **공백으로 구분된 문자열 하나**다. 받는 쪽(`HTTPBearer`)이 공백을 기준으로 split해서 타입과 값을 분리한다. RFC 6750에서 "Bearer 타입을 쓸 때는 공백으로 타입과 토큰을 구분해라"고 약속을 정해놓은 것이고, 서버와 클라이언트 양쪽이 그 약속대로 만들고 파싱한다.

타입 종류 예시:
- `Bearer` — JWT 같은 토큰 기반 인증
- `Basic` — 아이디:비밀번호를 Base64 인코딩한 것
- `Digest` — 해시 기반 인증

`Bearer` 없이 토큰만 보내면 백엔드의 `HTTPBearer`가 형식이 틀렸다고 판단해 **403**을 반환한다.

실제 네트워크에서 전송되는 헤더:
```
GET /api/v1/users/42 HTTP/1.1
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

## 4단계: 백엔드에서 JWT 검증

**경로:** FastAPI 라우터 → `get_current_user()` 의존성 → 핸들러

### HTTPBearer — 헤더 파싱 담당

`get_current_user()`가 토큰을 직접 꺼내기 전에, `HTTPBearer`가 먼저 실행된다.

```python
# backend/app/api/deps.py
bearer_scheme = HTTPBearer()  # FastAPI 내장 보안 유틸리티 클래스의 인스턴스
```

`HTTPBearer()`는 클래스를 호출해서 객체(인스턴스)를 만드는 것이다. 이 객체는 `__call__`이 정의되어 있어 함수처럼 호출할 수 있다(callable). FastAPI가 `Depends(bearer_scheme)`를 처리할 때 내부적으로 `bearer_scheme(request)`를 실행하고, `request`는 FastAPI가 자동으로 넘겨준다.

`HTTPBearer` 내부를 단순화하면:
```python
class HTTPBearer:
    async def __call__(self, request: Request):
        authorization = request.headers.get("Authorization")
        scheme, token = authorization.split(" ")  # "Bearer eyJ..." → 분리
        if scheme.lower() != "bearer":
            raise HTTPException(status_code=403)
        return HTTPAuthorizationCredentials(scheme=scheme, credentials=token)
```

`HTTPBearer`가 하는 일:
1. 요청 헤더에서 `Authorization` 헤더를 찾는다
2. 값이 `Bearer <token>` 형식인지 확인한다
3. `<token>` 부분만 추출해서 `HTTPAuthorizationCredentials` 객체로 감싸 반환한다
4. 헤더가 없거나 형식이 틀리면 그 자리에서 **403** 반환 (JWT 검증도 가기 전에 차단)

```
요청 헤더:  Authorization: Bearer eyJhbGci...
                                   ↓ HTTPBearer 파싱
credentials.scheme      →  "Bearer"
credentials.credentials →  "eyJhbGci..."   ← 이걸 jwt.decode()에 넘김
```

**`bearer_scheme = HTTPBearer()`를 분리해서 쓰는 이유:**
`Depends(HTTPBearer())`를 엔드포인트마다 쓰면 요청마다 객체가 새로 생성된다. 미리 `bearer_scheme`으로 분리하면 앱 시작 시 딱 한 번만 생성하고 재사용할 수 있다. 설정 변경도 한 곳만 수정하면 전체에 적용된다.

### Depends — 의존성 주입

`Depends`는 "이 함수를 실행하기 전에 저 함수를 먼저 실행하고, 그 결과를 여기에 넣어줘"라는 선언이다. 직접 함수를 호출하는 것과 결과는 같지만 세 가지 차이가 있다:

1. **중복 실행 방지** — 같은 요청 안에서 동일한 `Depends`가 여러 곳에 쓰이면 FastAPI가 한 번만 실행하고 결과를 재사용한다
2. **정리(cleanup) 보장** — `yield`를 쓴 의존성(`get_db`)은 요청이 끝나면 `finally` 블록을 FastAPI가 자동으로 실행한다. 직접 호출하면 예외 발생 시 `db.close()`를 내가 보장해야 한다
3. **Swagger 자동 반영** — `Depends(HTTPBearer())`를 쓰면 Swagger UI에 자물쇠 아이콘이 생기고 "인증 필요" 엔드포인트가 자동으로 표시된다

FastAPI는 `Depends`에 넘긴 함수의 파라미터 타입 힌트를 분석해서 자동으로 주입한다. `Request` 타입으로 선언하면 HTTP 요청 객체를, `Session` 타입이면 DB 세션을 넣어준다. 내 함수에도 `def my_func(request: Request)`로 선언하고 `Depends`에 넘기면 동일하게 동작한다.

### get_current_user — JWT 검증 담당

```python
# backend/app/api/deps.py
def get_current_user(
    credentials: HTTPAuthorizationCredentials = Depends(bearer_scheme),  # HTTPBearer가 먼저 실행됨
    db: Session = Depends(get_db),
) -> User:
    token = credentials.credentials      # HTTPBearer가 이미 "Bearer " 접두사 제거한 순수 토큰
    try:
        payload = jwt.decode(            # 서명 검증 + 만료 검증
            token,
            settings.JWT_SECRET_KEY,
            algorithms=[settings.JWT_ALGORITHM]
        )
        user_id = int(payload.get("sub"))    # 토큰 안의 사용자 ID 꺼내기
    except (JWTError, TypeError, ValueError):
        raise HTTPException(status_code=status.HTTP_401_UNAUTHORIZED, detail="Invalid token")
    user = db.query(User).filter(User.id == user_id).first()
    if not user:
        raise HTTPException(status_code=status.HTTP_401_UNAUTHORIZED, detail="User not found")
    return user                          # 핸들러에 사용자 객체 주입
```

실행 순서 정리:
```
요청 도착
  → Depends(bearer_scheme)  : HTTPBearer가 헤더 파싱 (형식 오류 → 403)
  → Depends(get_db)         : DB 세션 생성
  → get_current_user 본문   : JWT 디코딩, 만료/서명 검증 (실패 → 401)
                              DB에서 사용자 조회 (없으면 → 401)
  → 핸들러(user 주입)       : 비즈니스 로직 실행
```

### jwt.decode() — 위조 검증 원리

JWT는 점(`.`)으로 구분된 3개의 파트로 구성된다:

```
eyJhbGciOiJIUzI1NiJ9  .  eyJzdWIiOiI0MiJ9  .  SflKxwRJSMeKKF2QT4fwpMeJf36P
        헤더                    페이로드               서명(Signature)
```

- **헤더** — 알고리즘 정보 (`HS256` 등)
- **페이로드** — 실제 데이터 (`sub`, `exp` 등). Base64 인코딩이라 누구나 열어볼 수 있음
- **서명** — `헤더 + 페이로드`를 `JWT_SECRET_KEY`로 해싱한 값

누군가 페이로드를 `{"sub": "1"}` → `{"sub": "999"}`로 바꿨다면:
```
변조된 페이로드로 서명을 다시 계산하려면 JWT_SECRET_KEY 가 필요
→ 서버만 가지고 있으므로 공격자는 올바른 서명을 만들 수 없음
→ jwt.decode() 가 "페이로드는 바뀌었는데 서명이 안 맞네"를 감지
→ JWTError 발생 → 401 반환
```

`jwt.decode()`가 검증하는 항목:
1. **서명 위변조** — `JWT_SECRET_KEY`로 서명을 재계산해서 토큰의 서명과 비교
2. **만료(`exp`)** — 현재 시각과 비교해서 자동 확인

### DB 조회를 하는 이유

JWT 페이로드에 `user_id`, `email`, `role`이 이미 있어서 DB 조회 없이도 인증이 가능하다. 실제로 JWT의 원래 설계 철학이 "DB 없이 토큰만으로 검증"이다. 그러나 이 프로젝트에서는 DB 조회를 추가한다.

이유는 **JWT는 발급 시점의 스냅샷**이기 때문이다. 토큰 발급 이후 이런 일이 생기면 토큰만으로는 알 수 없다:
- 사용자가 탈퇴했다 → 토큰은 아직 유효, DB엔 없음
- role이 변경됐다 → 토큰엔 `"user"`, DB엔 `"admin"`
- 계정이 정지됐다 → 토큰은 모름

"토큰이 유효하다"와 "사용자가 현재도 유효하다"는 다른 말이다.

**토큰 수명과 DB 조회의 관계:**
- 토큰 수명이 짧을수록(15분 등) → 토큰만 믿어도 위험이 낮음 → DB 조회 생략 가능
- 토큰 수명이 길수록(이 프로젝트는 7일) → DB 조회가 현실적인 선택

또한 DB를 조회하면 토큰의 dict가 아닌 **User 객체**를 반환할 수 있어서, 핸들러에서 `current_user.nickname` 같이 모든 필드를 바로 쓸 수 있다는 실용적인 이점도 있다.

검증 실패 시 → `HTTP 401 Unauthorized` 반환

## 5단계: 401 수신 시 자동 로그아웃

**경로:** 백엔드 401 응답 → api.js 인터셉터 → 로그아웃

```js
// frontend/src/lib/api.js
api.interceptors.response.use(
  (res) => res,
  (error) => {
    if (error.response?.status === 401) {
      useAuthStore.getState().clearAccessToken();  // 토큰 제거
      window.location.href = '/login';             // 로그인 페이지 이동
    }
    return Promise.reject(error);
  }
);
```

토큰이 만료되거나 위조된 경우 사용자를 자동으로 로그아웃시킨다.

## 흐름 요약표

| 단계 | 위치 | 하는 일 |
|------|------|---------|
| 1 | `security.py` | `create_access_token()` — user_id를 담아 JWT 서명 후 발급 |
| 2 | `authStore.js` | 토큰을 Zustand + localStorage에 저장 |
| 3 | `api.js` (요청 인터셉터) | 모든 요청에 `Authorization: Bearer` 헤더 자동 부착 |
| 4 | `deps.py` | JWT 디코딩 → 서명/만료/사용자 검증 → 핸들러에 주입 |
| 5 | `api.js` (응답 인터셉터) | 401 수신 시 토큰 제거 + 로그인 페이지 이동 |

## 질문 포인트 (공부 시 확인할 것)

- `sub` 클레임이 무엇인지, JWT 표준 클레임(`iss`, `exp`, `sub`)의 의미
- `HS256` 알고리즘 — 대칭키 서명 방식, 서버만 키를 알고 있어야 하는 이유
- `Depends(bearer_scheme)`가 어떻게 헤더에서 토큰을 추출하는지
- 1단계의 보안 타협(localStorage XSS 위험)과 2단계 개선 방향(HttpOnly 쿠키 + Refresh Token)
