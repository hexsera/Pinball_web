# Redis 부하 테스트 설계

- **날짜**: 2026-04-04
- **도구**: k6
- **환경**: 실제 운영 서버 (HTTPS)
- **목표**: `GET /api/v1/monthly-scores` 엔드포인트에서 Redis 캐시 유무에 따른 성능 차이를 수치로 검증

---

## 테스트 대상 엔드포인트

| 엔드포인트 | 설명 |
|---|---|
| `GET /api/v1/monthly-scores` | 이번 달 전체 랭킹 조회 — Redis 캐시 주 경로 |

---

## 시나리오

### 시나리오 A — Redis 없음 (DB 직접 조회)

1. Redis 컨테이너 다운 (`docker compose stop redis-server`)
2. k6 부하 테스트 실행
3. 결과 기록

### 시나리오 B — Redis 있음 (캐시 히트)

1. Redis 컨테이너 재기동 (`docker compose start redis-server`)
2. 캐시 warm-up 확인 (첫 요청 시 자동 warm-up)
3. k6 부하 테스트 실행
4. 결과 기록

---

## 부하 단계 설계

단계적으로 동시 사용자 수를 올려 한계 지점을 찾는다.

| 단계 | 동시 사용자 수 | 지속 시간 | 목적 |
|---|---|---|---|
| 워밍업 | 5 | 30초 | 기준값 확인 |
| 경량 부하 | 20 | 60초 | 일반 트래픽 시뮬레이션 |
| 중간 부하 | 50 | 60초 | 부하 증가 시 변화 확인 |
| 고부하 | 100 | 60초 | 한계 지점 탐색 |
| 최고 부하 | 200 | 60초 | 에러 발생 지점 확인 |

---

## 측정 지표

| 지표 | 기준 |
|---|---|
| 응답 시간 p50 | Redis 있을 때 현저히 낮아야 함 |
| 응답 시간 p95 | DB 직접 조회 시 급격히 증가 예상 |
| 응답 시간 p99 | 최악 케이스 비교 |
| RPS (처리량) | Redis 있을 때 더 높아야 함 |
| 에러율 | Redis 없을 때 더 낮은 동시 사용자 수에서 에러 발생 예상 |

## 사전 준비

### 테스트 데이터

mock 데이터 스크립트(`backend/scripts/mock/seed_mock_data.py`)로 사용자 50명 + 월간 점수 50개가 이미 존재한다. 테스트 전 DB에 데이터가 충분한지 확인한다.

### k6 설치

```bash
# macOS
brew install k6

# Linux
sudo gpg --no-default-keyring --keyring /usr/share/keyrings/k6-archive-keyring.gpg \
  --keyserver hkp://keyserver.ubuntu.com:80 --recv-keys C5AD17C747E3415A3642D57D77C6C491D6AC1D69
echo "deb [signed-by=/usr/share/keyrings/k6-archive-keyring.gpg] https://dl.k6.io/deb stable main" \
  | sudo tee /etc/apt/sources.list.d/k6.list
sudo apt-get update && sudo apt-get install k6
```
