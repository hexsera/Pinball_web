## 논의 배경

실제 트래픽 상황에서 Redis 및 FastAPI 백엔드가 어떻게 동작하는지 검증이 필요하다.

`GET /api/v1/monthly-scores` 엔드포인트를 기준으로, Redis 캐시 유무에 따른 성능 차이를 수치로 검증한다.

## 현재 구조 파악

### `GET /api/v1/monthly-scores` 요청 흐름

```
요청 수신
  └─ Redis Sorted Set 조회 (zrevrangebyscore)
       ├─ 캐시 히트 → 즉시 응답
       └─ 캐시 미스 → DB 조회(warm_up_sorted_set) → Redis 재구성 → 응답
```

- Redis 장애 시 자동으로 DB 직접 조회로 폴백 (`except Exception: pass`)
- `POST /api/v1/monthly-scores` → DB 쓰기 후 Redis zadd 동기화

## 측정 지표

| 지표 | 설명 |
|---|---|
| 응답 시간 (latency) | 평균, p50, p95, p99 |
| 처리량 (throughput) | RPS (Requests Per Second) |
| DB 쿼리 횟수 | 동시 N 요청 시 DB hit 수 |
| 에러율 | 부하 증가 시 오류 발생 지점 |

### 응답 시간 (latency) — p50, p95, p99

100개의 요청을 빠른 것부터 느린 것 순으로 줄을 세웠을 때, 앞에서 몇 번째에 해당하는지를 나타낸다.

```
1번(가장빠름) ... 50번 ... 95번 ... 99번 ... 100번(가장느림)
```

- **p50**: 50번째. 절반은 이것보다 빠르고 절반은 느리다. 일반적인 사용자 경험에 가깝다.
- **p95**: 95번째. 상위 5% 안에 드는 느린 요청들의 대표값. 평균이 50ms인데 p95가 800ms라면, 대부분은 빠르지만 느린 요청은 꽤 심하게 느리다는 것을 알 수 있다. 평균만 보면 이 사실이 숨겨진다.
- **p99**: 99번째. 가장 극단적인 1개(100번째)를 제외한 사실상 최악의 케이스. 100번째(최댓값)를 쓰지 않는 이유는 네트워크 순간 끊김이나 서버 GC 같은 일회성 이상치에 너무 민감하게 반응하기 때문. p99는 이런 우연적 이상치를 제거하면서도 "실제로 경험할 수 있는 최악"에 가장 가까운 값이다.

---

### 처리량 (throughput) — RPS

서버가 1초에 처리할 수 있는 요청 수. RPS = Requests Per Second.
Redis가 있을 때 RPS가 더 높다면, 같은 시간에 더 많은 사용자를 처리할 수 있다는 의미.

### DB 쿼리 횟수 — DB hit

요청이 들어왔을 때 데이터베이스까지 실제로 조회가 발생한 횟수.

- Redis 있음: 캐시에서 바로 응답 → DB hit 거의 0
- Redis 없음: 요청 100개 → DB 조회 100번 발생

DB는 Redis보다 훨씬 느리고 자원을 많이 사용하므로, 동시 요청이 많아질수록 DB hit 수가 병목이 된다.

### 에러율 — 부하 증가 시 오류 발생 지점

부하를 점점 높였을 때 서버가 요청을 정상 처리하지 못하고 오류(타임아웃, 연결 실패 등)를 반환하기 시작하는 지점.

예: 동시 요청 50개까지는 에러 0%인데 200개부터 에러율 5%가 된다면 → 한계는 50~200 사이.
Redis 유무에 따라 이 한계치가 달라지는지를 비교하는 것이 핵심.
