## Nginx에서 사용할 수 있는 캐시 종류

### 1. 정적 파일 캐시 (Cache-Control 헤더)

Nginx가 정적 파일을 직접 서빙할 때 응답 헤더에 `Cache-Control`을 붙여주는 방식. Nginx가 파일을 메모리나 어딘가에 저장하는 게 아니라, 브라우저에게 "이 파일은 X시간 동안 캐시해도 된다"고 알려주는 것.

```nginx
location ~* \.(js|css|png|jpg|jpeg|mp3|wav|webp|woff2)$ {
    expires 1y;
    add_header Cache-Control "public, immutable";
}
```

- `expires 1y`: 브라우저에게 1년 동안 캐시해도 된다고 알림
- `immutable`: 만료 전에는 조건부 요청(If-Modified-Since)도 보내지 말라는 지시
- Vite 빌드 결과물(`index-BXRjaAlc.js`)처럼 파일명에 해시가 붙어 있으면 내용이 바뀌면 파일명 자체가 달라지므로 1년을 줘도 안전

| 설정 | 동작 |
|------|------|
| `expires 1y` | max-age=31536000 헤더 생성 |
| `immutable` | 만료 전 재검증 요청도 생략 |
| `no-cache` | 항상 서버에 재검증 요청 (304 가능) |
| `no-store` | 브라우저가 저장 자체를 안 함 |

---

### 2. 프록시 캐시 (proxy_cache)

Nginx가 업스트림 서버(FastAPI 등)의 응답을 Nginx 자신의 디스크/메모리에 저장하는 방식. 브라우저가 아닌 Nginx가 캐시를 보유.

```nginx
http {
    proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=api_cache:10m max_size=100m;

    server {
        location /api/ {
            proxy_pass http://fastapi:8000;
            proxy_cache api_cache;
            proxy_cache_valid 200 60s;
            add_header X-Cache-Status $upstream_cache_status;
        }
    }
}
```

- `proxy_cache_path`: 캐시 파일을 저장할 디스크 경로와 메모리 키 공간 설정
- `proxy_cache_valid 200 60s`: 200 응답을 60초 동안 보관
- `X-Cache-Status`: HIT/MISS 여부를 응답 헤더로 확인 가능
- 브라우저가 아닌 Nginx 서버 쪽에 캐시가 있으므로, 여러 브라우저/유저가 공유

---

### 3. 두 방식의 차이

| | 정적 파일 Cache-Control | proxy_cache |
|--|--|--|
| 캐시 위치 | 브라우저 | Nginx 서버 |
| 캐시 주체 | 각 사용자 브라우저 | 서버 1곳에서 공유 |
| 무효화 방법 | 파일명 변경(해시) or 만료 대기 | `proxy_cache_purge` 또는 만료 대기 |
| 적합한 대상 | JS, CSS, 이미지, 폰트 | API 응답, SSR HTML |
| 설정 위치 | Nginx `location` 블록 | Nginx `http` + `location` 블록 |

## 이 프로젝트에서 캐시를 쓰기 좋은 곳

### 현재 nginx.conf 상태

```nginx
location / {
    try_files $uri $uri/ /index.html;
}
```

`Cache-Control` 헤더가 없어 브라우저가 정적 파일을 캐시하지 않는 상태.

---

### 적용 추천 위치

#### (1) Vite 빌드 산출물 — 정적 파일 Cache-Control [적극 추천]

`dist/` 디렉토리의 파일 목록:
- `index-BXRjaAlc.js`, `index-Dww-4alG.css` — Vite가 내용 해시를 파일명에 포함시킴
- `newbg.jpg`, `pinball_back.png`, `*.mp3`, `*.wav` — 정적 에셋

파일명에 해시가 있으면 내용이 바뀔 때 파일명 자체가 달라지므로 `max-age=1y + immutable`을 줘도 안전. `index.html`은 해시 없이 항상 같은 이름이므로 캐시하면 안 됨(새 배포가 반영 안 됨).

```nginx
# 해시가 붙은 빌드 산출물 — 1년 캐시
location ~* \.(js|css)$ {
    expires 1y;
    add_header Cache-Control "public, immutable";
}

# 정적 에셋 (이미지, 오디오) — 1주일 캐시
location ~* \.(png|jpg|jpeg|webp|mp3|wav|woff2)$ {
    expires 7d;
    add_header Cache-Control "public";
}

# index.html — 캐시 금지 (항상 최신 버전 받아야 함)
location = /index.html {
    add_header Cache-Control "no-cache";
}
```

#### (2) API 응답 — proxy_cache [선택 사항]

이 프로젝트에서 Nginx는 `/api/*` 요청을 FastAPI로 그냥 통과시키는 역할만 함. API 응답 캐시는 이미 FastAPI에서 `Cache-Control` + `ETag`로 처리할 계획(공지사항 캐시 이슈 참고).

두 레이어에서 동시에 캐시하면 무효화가 복잡해지므로, API 캐시는 FastAPI에서 관리하는 걸로 역할을 분리하는 편이 깔끔함.

## 정리

| 대상 | 방법 | 설정 위치 | 우선순위 |
|------|------|-----------|----------|
| JS/CSS (해시 파일명) | `Cache-Control: immutable, max-age=1y` | Nginx | 높음 |
| 이미지/오디오 | `Cache-Control: public, max-age=7d` | Nginx | 높음 |
| index.html | `Cache-Control: no-cache` | Nginx | 높음 |
| API 응답 (공지사항) | `Cache-Control + ETag` | FastAPI | 별도 이슈 진행 중 |
| API proxy_cache | Nginx proxy_cache | Nginx | 낮음 (역할 중복) |
