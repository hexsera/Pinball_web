## 논의 내용

### 상태(State)란?

UI가 현재 기억하고 있는 데이터.

- 로그인한 유저 정보
- 버튼이 눌렸는지 여부
- 서버에서 불러온 목록 데이터

### 상태관리란?

이 데이터를 **어디에 저장하고, 어떻게 읽고, 어떻게 바꿀지** 다루는 것.

### 상태의 두 종류

| 종류 | 설명 | 예시 |
|------|------|------|
| **로컬 상태** | 컴포넌트 내부에서만 사용 | 모달 열림/닫힘, 입력 폼 값 |
| **전역 상태** | 여러 컴포넌트가 공유 | 로그인 정보, 장바구니 |

### Props란?

부모 컴포넌트가 자식 컴포넌트에게 전달하는 데이터. HTML 태그의 속성(attribute)과 비슷한 개념.

```jsx
// 부모가 전달
<ProfileIcon name="홍길동" age={25} />

// 자식이 받아서 사용
function ProfileIcon(props) {
  return <span>{props.name}</span>
}
```

**핵심 규칙**: props는 위에서 아래로만 흐른다. 자식이 부모로 올려보낼 수 없다.

### 왜 부모가 데이터를 갖고 있나?

React 설계 원칙: **여러 자식이 같은 데이터를 필요로 하면, 그 데이터는 공통 부모에 올려둔다.**

```
App ← user 데이터를 여기 둠 (공통 부모)
├── Header (user 필요)
└── Sidebar (user 필요)
```

결국 부모가 props를 전달하는 이유는 **"자식이 필요한 데이터를 부모가 갖고 있기 때문"** 이 전부.

### Props Drilling 문제

데이터가 필요한 컴포넌트가 트리 깊은 곳에 있을수록 공통 부모도 높아지고, 중간에 거쳐야 하는 컴포넌트가 많아진다.

```
App (user 보유 — 공통 부모)
 └── Layout (안 씀, 그냥 전달)
      └── Header (안 씀, 그냥 전달)
           └── ProfileIcon ← 여기만 필요
```

중간 컴포넌트들이 쓰지도 않는 props를 계속 전달해야 하는 문제 → **Props Drilling**

**문제점:**
- 중간 컴포넌트 코드가 불필요하게 복잡해짐
- `user` 구조가 바뀌면 중간 컴포넌트를 모두 수정해야 함
- 컴포넌트 재사용이 어려워짐

이를 해결하기 위해 상태관리 라이브러리 등장:
- **Redux**: 전통적인 대규모 앱용 전역 상태관리
- **Zustand**: 가볍고 단순한 전역 상태관리
- **Context API**: React 내장, 소규모 전역 상태에 적합
