# 구글 로그인 도입 적합성 분석 보고서

> 분석 대상: `react-oauth/google` + `Authlib` 조합 / Authorization Code Flow
> 작성일: 2026-03-22

---

## 1. 현재 인증 구조 요약

| 항목 | 현황 |
|------|------|
| 인증 방식 | 이메일/비밀번호 (자체 구현) |
| 세션 저장 | localStorage (`user_id`, `email`, `nickname`, `role`) |
| 토큰 | 없음 — 로그인 응답값을 그대로 localStorage에 저장 |
| 백엔드 인증 | API Key (`X-API-Key` 헤더) 기반 |
| User 테이블 | `password` 컬럼 `NOT NULL` + `birth_date` 필수 |

---

## 2. 패키지 궁합 분석

### react-oauth/google
- 프론트에서 구글 OAuth 팝업/리다이렉트를 처리하는 React 라이브러리
- 기본 동작: 구글로부터 **ID Token** (JWT) 또는 **Authorization Code** 반환
- `useGoogleLogin`의 `flow: 'auth-code'` 옵션으로 Authorization Code Flow 가능

### Authlib (백엔드)
- FastAPI용 OAuth 2.0 서버/클라이언트 라이브러리
- `AsyncOAuth2Client`를 이용해 구글 토큰 엔드포인트와 통신
- Authorization Code를 받아 Access Token + ID Token 교환 가능

### 궁합 평가: **호환 가능, 단 역할 분리 명확히 해야 함**

```
[브라우저]
  react-oauth/google → 구글로부터 Authorization Code 획득
        ↓ code 전송 (POST /api/v1/auth/google)
[백엔드 FastAPI]
  Authlib → 구글 Token 엔드포인트에 code 교환 요청
          → ID Token 검증 → 사용자 조회/생성 → 로그인 응답
```

두 라이브러리는 **각자의 역할**(프론트: code 획득, 백엔드: code 교환)을 분담하므로 기술적으로 잘 맞는다.

---

## 3. Authorization Code Flow 적합성

| 항목 | 판단 |
|------|------|
| 보안 | code + secret을 백엔드에서만 처리 → **Access Token이 브라우저에 노출되지 않아 안전** |
| 현재 구조와의 정합성 | 로그인 응답 포맷(`user_id`, `nickname`, `role`)을 그대로 유지 가능 |
| 세션 방식 호환 | AuthContext의 `login()` 함수를 그대로 재사용 가능 |
| SPA 적합성 | Vite/React SPA에서 `react-oauth/google`이 리다이렉트 없이 팝업 처리 지원 |

Implicit Flow (ID Token을 프론트에서 직접 검증)보다 Authorization Code Flow가 보안상 권장되며, 이 프로젝트 구조에도 더 잘 맞는다.

## 4. 프로젝트와의 충돌 지점

### 4-1. `password` 컬럼 NOT NULL 제약
- 현재 `User.password`가 `NOT NULL` — 구글 로그인 사용자는 비밀번호가 없음
- **해결 방안**: 컬럼을 `nullable=True`로 변경하거나, 더미 값(빈 문자열 해시 등) 저장
- Alembic 마이그레이션 필요

### 4-2. `birth_date` 필수 컬럼
- 구글 계정에서는 생년월일을 제공하지 않음
- 회원가입 플로우가 생년월일 3단계 입력으로 구성되어 있음
- **해결 방안**: 소셜 로그인 최초 진입 시 닉네임/생년월일 추가 입력 단계 추가 또는 `nullable` 변경

### 4-3. 닉네임 미제공
- 구글 계정의 `name` 필드를 기본 닉네임으로 활용 가능
- 단, 중복 닉네임 가능성이 있으므로 중복 검증 로직 필요

### 4-4. API Key 인증과 혼용
- 현재 대부분의 엔드포인트는 `X-API-Key` 헤더로 보호
- 구글 로그인 엔드포인트(`/api/v1/auth/google`)는 API Key 없이 접근 가능하도록 설계해야 함 (현재 `/login`, `/register`와 동일 패턴으로 처리 가능)

### 4-5. 기존 사용자와의 계정 통합 문제
- 동일 이메일로 이미 자체 가입한 사용자가 구글 로그인 시도 시 처리 방식 결정 필요
- **연동 허용** vs **오류 반환** 중 정책 선택 필요

---

## 5. 종합 판단

| 항목 | 평가 |
|------|------|
| 패키지 기술적 궁합 | 적합 |
| Authorization Code Flow 적합성 | 적합 (권장) |
| 현재 인증 구조 호환성 | 부분 충돌 (DB 스키마 수정 필요) |
| 도입 난이도 | 중간 — DB 마이그레이션 + 신규 엔드포인트 + 프론트 UI 추가 |
| 보안 영향 | 긍정적 (비밀번호 없는 로그인 옵션 추가) |

**결론**: 두 패키지의 조합은 이 프로젝트에 **적합**하다. Authorization Code Flow를 사용하면 기존 localStorage 기반 세션 구조를 유지하면서 구글 로그인을 추가할 수 있다. 다만, `password`/`birth_date` NOT NULL 제약 해소와 소셜 전용 사용자를 위한 최초 가입 플로우 설계가 선행되어야 한다.
