# 핀볼 게임 모바일 성능 문제 분석

## 개요

`Pinball.jsx`를 분석한 결과, 모바일에서 게임이 느린 원인은 크게 **렌더링 부하**, **물리 연산 부하**, **레이아웃 문제** 세 가지로 나뉩니다.

---

## 문제 1: afterRender에서 매 프레임마다 과도한 Canvas 드로잉

### 문제점
매 프레임마다 `afterRender` 이벤트에서 범퍼와 타겟 오브젝트에 **네온 글로우 효과**를 직접 Canvas에 그립니다. 범퍼 하나당 5번의 `ctx.arc` + `shadowBlur` 드로잉이 실행되고, 타겟도 마찬가지로 복잡한 별 모양 패턴을 매 프레임 그립니다.

### 왜 느린가
`shadowBlur`는 GPU가 아닌 CPU에서 처리되는 무거운 연산입니다. 게임은 보통 1초에 60번 화면을 다시 그리는데, 범퍼가 5개라면 매초 5×5×60 = **1,500번**의 shadowBlur 연산이 일어납니다. 모바일 CPU는 PC보다 성능이 낮기 때문에 이 연산이 쌓이면 프레임이 떨어집니다. 특히 `createRadialGradient`도 매 프레임 새로 생성하는데, 이 객체는 캐싱 없이 계속 만들어집니다.

### 해결 방법
- `shadowBlur` 사용을 제거하거나 오프스크린 Canvas에 미리 그려두고 `drawImage`로 복사하는 방식으로 바꿉니다.
- `createRadialGradient`를 컴포넌트 마운트 시 한 번만 생성해 캐싱합니다.

## 문제 2: Matter.js Render가 고정 해상도(700×1100)로 실행됨

### 문제점
`Render.create`에서 `width: 700, height: 1100`으로 고정 크기를 설정합니다. 모바일 화면은 보통 훨씬 작은데, 화면보다 **큰 Canvas를 CSS로 축소**해서 보여주는 방식이 됩니다. 게임 물리 연산과 렌더링은 여전히 700×1100 풀 사이즈로 돌아갑니다.

### 왜 느린가
Canvas 픽셀 수가 많을수록 GPU가 처리해야 할 양이 늘어납니다. 700×1100 = **770,000 픽셀**을 매 프레임 처리하는데, 모바일 화면이 실제로는 300×470 수준이라면 쓸데없이 2.5배 이상의 픽셀을 처리하는 셈입니다. 또한 Matter.js의 기본 렌더러는 `devicePixelRatio`를 고려하지 않아 Retina 디스플레이에서 추가 부하가 생길 수 있습니다.

### 해결 방법
- 모바일 화면 크기를 감지해 `Render.create`의 width/height를 동적으로 조정합니다.
- 또는 Matter.js 자체 렌더러 대신 CSS transform으로 스케일 다운하되, 물리 좌표계는 유지합니다.

## 문제 3: 매 충돌마다 Audio를 처음부터 재생 (hitSound 문제)

### 문제점
공이 무언가에 닿을 때마다 `hitSoundRef.current.currentTime = 0`으로 되감고 즉시 재생합니다. 핀볼 게임 특성상 충돌이 연속으로 발생하면, 이 오디오 재생 요청이 매 물리 프레임(~60fps)에서 반복됩니다.

### 왜 느린가
`Audio.play()`는 비동기 작업이고 내부적으로 오디오 디코딩과 버퍼 처리를 유발합니다. 모바일에서는 오디오 컨텍스트 생성과 관리 비용이 높으며, 짧은 시간 내에 수십 번 호출되면 오디오 스레드가 포화 상태가 되어 UI 스레드까지 영향을 받습니다. 특히 범퍼와 벽을 연속으로 치는 순간 성능이 급격히 떨어집니다.

### 해결 방법
- Web Audio API의 `AudioContext`와 `AudioBuffer`를 사용해 오디오를 재생합니다. 이 방식은 훨씬 가볍고 동시 재생도 잘 처리합니다.
- 쿨다운을 두어 같은 사운드가 100ms 이내에 다시 재생되지 않도록 제한합니다.

## 문제 4: 고정 픽셀 레이아웃으로 인한 모바일 스크롤 발생

### 문제점
최상위 `Box`의 너비가 `700px`로 고정되어 있습니다. 모바일 화면 너비(보통 375~430px)보다 크기 때문에 브라우저가 페이지를 가로로 축소하거나 스크롤이 생깁니다. 게임 캔버스와 터치 좌표가 어긋날 수 있습니다.

### 왜 느린가
브라우저가 700px 레이아웃을 모바일 화면에 맞게 강제 축소(viewport zoom)하면서 **레이아웃 재계산과 합성(composite) 비용**이 추가로 발생합니다. 또한 터치 이벤트 좌표와 실제 게임 좌표 사이에 변환 오차가 생겨 입력 지연처럼 느껴질 수 있습니다. 스크롤 가능 상태에서는 터치 이벤트가 스크롤과 충돌해 `touchAction: none` 설정에도 불구하고 일부 이벤트가 지연됩니다.

### 해결 방법
- 컨테이너를 `width: 100vw`, `max-width: 700px`로 변경하고 CSS `transform: scale()`로 내부 Canvas를 비율에 맞게 조정합니다.
- 또는 Matter.js Render 크기 자체를 `window.innerWidth` 기준으로 동적으로 계산합니다.

## 문제 5: `beforeUpdate` 이벤트에서 매 프레임 Body.setAngle / setAngularVelocity 호출

### 문제점
`beforeUpdate`는 물리 엔진이 매 스텝마다 호출하는 이벤트입니다. 여기서 플리퍼 두 개에 대해 `setAngle`, `setAngularVelocity`를 조건에 따라 호출하고, Plunger 위치도 `setPosition`으로 업데이트합니다.

### 왜 느린가
`setAngle`은 물리 바디의 회전 행렬을 다시 계산하게 만들며, 이를 매 프레임 강제 호출하면 물리 엔진의 슬립(sleep) 최적화가 무력화됩니다. 모바일에서 물리 엔진 틱이 빠르게 돌수록(기본 60fps) 이 비용이 누적됩니다. 특히 `isSleeping: false, sleepThreshold: Infinity`로 슬립을 완전히 비활성화했기 때문에 플리퍼가 항상 활성 바디로 처리됩니다.

### 해결 방법
- 키 상태가 변경된 프레임에서만 `setAngularVelocity`를 호출하도록 이전 상태와 비교하는 로직을 추가합니다.
- Matter.js Runner의 `delta` 값을 기반으로 물리 업데이트 빈도를 모바일에서 30fps로 낮추는 것도 고려합니다.
