### 1. `backend/tests/load/monthly_scores_load_test.js` 생성

**추가:**
```javascript
import http from 'k6/http';
import { check, sleep } from 'k6';

const BASE_URL = 'https://hexsera.com';

export const options = {
  stages: [
    { duration: '30s', target: 5   },  // 워밍업
    { duration: '60s', target: 20  },  // 경량 부하
    { duration: '60s', target: 50  },  // 중간 부하
    { duration: '60s', target: 100 },  // 고부하
    { duration: '60s', target: 200 },  // 최고 부하
    { duration: '10s', target: 0   },  // 종료
  ],
  thresholds: {
    http_req_duration: ['p(95)<2000'],  // p95 2초 이하 목표
    http_req_failed:   ['rate<0.05'],   // 에러율 5% 이하 목표
  },
};

export default function () {
  const res = http.get(`${BASE_URL}/api/v1/monthly-scores`);
  check(res, {
    'status is 200': (r) => r.status === 200,
  });
  sleep(1);
}
```

**변경 의도:** Redis 캐시 유무 두 시나리오에서 동일한 조건으로 부하를 재현하기 위해 작성. 동시 사용자 수를 5→20→50→100→200 단계적으로 올려 한계 지점을 탐색한다. p95 2초 이하, 에러율 5% 이하를 threshold로 설정.

### 3. `docs/history/2026-04-05-01-Redis-부하테스트-결과.md` 생성

**추가:** 테스트 조건, 결과 요약 표, 지표별 분석, k6 측정 원리 설명, 에러 원인 분석(DB 커넥션 풀 고갈), 결론을 포함한 결과 문서 작성.

핵심 분석 내용:
- 시나리오 A 최대 응답 30,582 ms는 SQLAlchemy 기본 커넥션 풀 타임아웃(30초)과 일치 → 동시 요청 15개(풀 5 + overflow 10) 초과 시 커넥션 대기 → 타임아웃 에러
- 시나리오 B는 Redis 메모리에서 직접 응답하여 커넥션 풀 경쟁 자체가 발생하지 않음
- `http_req_waiting ≈ http_req_duration` → 병목은 네트워크가 아닌 서버 처리 시간
