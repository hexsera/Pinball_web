## 각 포인트 상세

### [3] skill-creator가 있다는 걸 언제 아는가?

**OpenClaw 시점:**
- 게이트웨이 기동 시 설치된 스킬 디렉토리를 스캔
- 매 요청마다 시스템 프롬프트에 이걸 주입함:

```xml
<available_skills>
  <skill>
    <name>skill-creator</name>
    <description>Create or update AgentSkills...</description>
    <location>~/.../skills/skill-creator/SKILL.md</location>
  </skill>
  ...
</available_skills>
```

**LLM 시점:**
- 매 응답 직전, 시스템 프롬프트 안에서 이미 목록을 받아서 알고 있음
- 사용자가 물어보기 전부터 알고 있는 상태

---

### [1] skill-creator를 쓰겠다고 언제 결정하는가?

**LLM 시점 (전적으로 LLM이 결정):**

시스템 프롬프트 지침:
> "Before replying: scan `<available_skills>` description entries. If exactly one skill clearly applies → read SKILL.md. If multiple → choose most specific. If none → skip."

즉, 사용자 메시지를 받자마자 → 스킬 description들을 훑어서 → 매칭 여부 판단.

`skill-creator`의 description:
> *"Create or update AgentSkills. Use when: (1) building new features or apps, (2) designing/structuring/packaging skills..."*

사용자가 "스킬 만들어줘", "새 기능 추가하고 싶어" 같은 말을 하면 → 매칭 → 결정.

**사용자/OpenClaw 시점:**
- 이 결정 과정은 보이지 않음. LLM 내부에서 일어남.

---

### [2] skill-creator가 언제 호출되는가?

**LLM 시점:**
- 결정 후 `read` 툴을 호출해서 `SKILL.md`를 읽음
- 이게 실질적인 "호출" = 파일을 컨텍스트에 로드하는 시점

**OpenClaw 시점:**
- `read` 툴 요청을 받아서 파일 내용을 LLM에 반환
- OpenClaw 자체는 "스킬이 트리거됐다"는 개념을 따로 관리하지 않음. 그냥 파일 읽기 요청일 뿐.

**사용자 시점:**
- 직접 보이지 않음. 단지 응답이 평소보다 스킬에 맞게 구조화되어 나옴.

## 핵심 요약

| 질문 | 누가 | 언제 |
|---|---|---|
| 스킬 목록 인식 | LLM | 매 응답 직전 (시스템 프롬프트 수신 시) |
| 사용 결정 | LLM 단독 | 사용자 메시지 분석 후 즉시 |
| 실제 호출 | LLM → OpenClaw | read(SKILL.md) 툴 콜 시점 |

결론적으로 **사용자나 OpenClaw가 "이제 skill-creator 써라"고 트리거하는 게 아니라**, 항상 LLM이 스스로 판단해서 결정하는 구조야.
