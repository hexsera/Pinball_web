# 서버 접속 불가 원인 분석 및 해결

## 상황

hexsera.com 접속이 안 되는 문제 발생. 도커 컨테이너는 모두 정상 작동 중이었음.

## 네트워크 구조

```
인터넷 → 게이트웨이 → 메인공유기(192.168.0.1) → 서브공유기(192.168.0.55) → 서버(192.168.1.2)
```

메인공유기에서는 9090 포트만 서브공유기로 포워딩 (서브공유기 설정 페이지 접근용).  
서브공유기에서 80, 443을 서버(192.168.1.2)로 포워딩하는 구조.

## 시도한 것들

### 1. 도커 컨테이너 상태 확인 → 정상
- traefik, nginx, fastapi, postgres 모두 Up 상태
- 포트 80, 443 리스닝 확인됨

### 2. Traefik SSL 인증서 확인 → 정상
- acme.json에 Let's Encrypt 인증서 존재 (만료까지 89일 남음)
- localhost에서 직접 요청 시 200 응답 정상

### 3. DNS 확인 → 정상
- hexsera.com → 121.157.29.190 (서버 공인 IP와 일치)

### 4. 서브공유기 IP 변경 의심 → 아님
- 9090 포트로 서브공유기 설정 페이지 접근 성공 → IP 그대로 192.168.0.55

### 5. 서버 내부 IP 변경 의심 → 아님
- 서버 IP 확인 결과 192.168.1.2로 그대로
- 서브공유기 포트포워딩 설정도 정상

### 6. 외부에서 직접 IP 접근 테스트 → 원인 발견

```bash
curl http://121.157.29.190/
# → "Httpd/1.0", Last-Modified: 2018 → 공유기 자체 관리 페이지 응답
curl https://121.157.29.190/
# → 타임아웃 (443 포트 차단)
```

**메인공유기가 80 포트를 직접 먹고 있었음** — 서버로 포워딩되지 않고 공유기 관리 페이지가 응답하는 상태.

## 내가 생각한 원인
1. 내부 인터넷 망에서는 80 으로 접속시 메인 공유기 설정 페이지로 접속하는 문제가 생김
1-1. 외부 인터넷 망에서는 접근했을때 공유기 페이지가 나오지는 않았으나 여전히 웹사이트가 나오지 않았음
2. 이전에는 80, 443 을 서브 공유기로 포트포워딩 안해도 문제가 없었음.
3. 확실하지 않지만 이전에 80, 443 을 서브공유기로 포트포워딩 했다가 해당 포트포워딩 규칙을 삭제했는데 그것이 내부적으로 남아있었던 것 같음
4. 오랫동안 사용하지 않아서 포트포워딩 규칙 캐시가 사라짐
5. 포트포워딩 불가은

## 원인 (ai 해석. 불확실함)

메인공유기(iptime)의 **펌웨어 자동 업데이트** 후 설정 일부가 초기화된 것으로 추정.

- 외부에서 공유기 관리 페이지 접근이 기본값(허용)으로 초기화됨
- 80/443 포트포워딩 규칙이 사라짐
- 결과적으로 80 포트를 공유기가 직접 점유 → 서브공유기로 트래픽이 전달되지 않음

> 예전에 포트포워딩을 껐어도 작동했던 이유: 당시에는 공유기가 80포트를 점유하지 않았기 때문.

## 해결

메인공유기 설정에서 포트포워딩 추가:
- `80` → `192.168.0.55:80`
- `443` → `192.168.0.55:443`

## 재발 방지

- iptime 펌웨어 업데이트 후 포트포워딩이 초기화될 수 있으므로 설정 스크린샷 백업 권장
