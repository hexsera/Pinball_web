# Refresh Token 흐름 — 발급부터 자동 갱신까지

> 목적: Refresh Token이 어떻게 발급되고, Access Token이 만료됐을 때 어떻게 자동 갱신되는지 전체 흐름을 단계별로 따라간다.

---

## 왜 Refresh Token이 필요한가

1단계에서 Access Token 수명을 7일로 길게 잡았는데, 이건 보안 타협이었다. 토큰이 탈취되면 7일 동안 공격자가 API를 쓸 수 있다.

이상적인 구조:
- **Access Token** — 수명 15분. 짧아서 탈취돼도 피해 최소화
- **Refresh Token** — 수명 30일. Access가 만료되면 이걸로 새 Access를 발급받음

근데 Access가 15분이면 사용자가 15분마다 로그아웃되는 문제가 생긴다. Refresh Token이 이걸 막는다. 사용자는 로그아웃 없이 Access Token이 자동으로 갱신되는 것만 경험한다.

---

## 전체 흐름 한눈에 보기

```
[로그인]
  백엔드 → Access Token (JSON body, 15분)
         → Refresh Token (HttpOnly 쿠키, 30일) + Redis에 저장

[평소 API 요청]
  프론트 → Authorization: Bearer <Access Token>
  백엔드 → 검증 후 응답

[Access Token 만료 시]
  프론트 → API 요청 → 401 수신
         → /auth/refresh 자동 호출 (쿠키 자동 전송)
         → 새 Access Token 수신
         → 원래 요청 재시도 → 성공 응답
  (사용자는 이 과정을 인식하지 못함)

[로그아웃]
  프론트 → /auth/logout 호출
  백엔드 → Redis에서 Refresh Token 삭제 + 쿠키 만료
```

---

## 1단계: 로그인 시 Refresh Token 발급

**경로:** `auth.py` → `_create_login_response()` → `security.py`

### Refresh Token 생성

```python
# backend/app/core/security.py
import secrets

def create_refresh_token() -> str:
    return secrets.token_urlsafe(64)
```

Access Token(JWT)과 달리 Refresh Token은 **불투명 토큰(opaque token)** 이다. JWT는 페이로드에 정보가 담겨 있어 디코딩하면 읽을 수 있지만, 이건 그냥 랜덤 문자열이라 탈취해도 안에 아무 정보가 없다. 서버가 Redis에서 조회하는 방식으로만 검증한다.

### Redis에 저장

```python
# backend/app/api/v1/auth.py
REFRESH_TTL = settings.REFRESH_TOKEN_EXPIRE_DAYS * 24 * 3600  # 30일을 초로 환산

refresh_token = create_refresh_token()
redis_client.setex(f"refresh:{refresh_token}", REFRESH_TTL, str(user.id))
```

`setex(key, ttl_seconds, value)` — 저장과 동시에 TTL을 설정한다. 30일이 지나면 Redis가 자동으로 키를 삭제하므로 만료 청소 작업이 필요 없다.

키 구조: `refresh:{token_string}` → `value: user_id`
- `refresh:` 접두사로 다른 Redis 키와 충돌을 방지한다
- 조회 시 키 존재 여부만 확인하면 검증 완료

### 응답으로 전달

```python
response.set_cookie(
    key="refresh_token",
    value=refresh_token,
    httponly=True,   # JS에서 접근 불가 (XSS 방어)
    secure=True,     # HTTPS 전용
    samesite="lax",
    max_age=REFRESH_TTL,
    path="/api/v1/auth",  # /api/v1/auth/* 요청에만 쿠키 전송
)
```

Refresh Token은 JSON body가 아닌 **HttpOnly 쿠키**로만 전달한다. JS에서 `document.cookie`로 읽을 수 없어 XSS 공격으로 탈취가 불가능하다.

`path="/api/v1/auth"` 덕분에 `/api/v1/auth/refresh`, `/api/v1/auth/logout` 경로에만 쿠키가 자동 전송된다. 일반 API 요청에는 쿠키가 붙지 않는다.

## 2단계: 평소 API 요청 (Access Token 유효한 경우)

1단계 문서와 동일. 요청 인터셉터가 `Authorization: Bearer <Access Token>` 헤더를 자동으로 붙이고, 백엔드의 `get_current_user()`가 검증한다.

## 3단계: Access Token 만료 시 자동 갱신

**경로:** 401 수신 → `api.js` 응답 인터셉터 → `/auth/refresh` → 원 요청 재시도

### 프론트엔드 응답 인터셉터

```js
// frontend/src/lib/api.js
api.interceptors.response.use(
  (res) => res,
  async (error) => {
    const original = error.config;                          // 실패한 원래 요청 정보 보관
    if (error.response?.status === 401 && !original._retry) {
      original._retry = true;                              // 재시도 플래그 설정
      try {
        const { data } = await axios.post(                 // refresh 엔드포인트 호출
          '/api/v1/auth/refresh', {}, { withCredentials: true }
        );
        useAuthStore.getState().setAccessToken(data.access_token);  // 새 토큰 저장
        original.headers.Authorization = `Bearer ${data.access_token}`;  // 원 요청 헤더 교체
        return api(original);                              // 원 요청 재시도
      } catch {
        useAuthStore.getState().clearAccessToken();
        window.location.href = '/login';                   // 갱신 실패 시만 로그아웃
      }
    }
    return Promise.reject(error);
  }
);
```

코드 핵심 포인트:

- `error.config` — 실패한 요청의 설정 객체 (URL, 헤더, 바디, 메서드 전부 포함). 이걸 재사용해서 원 요청을 그대로 재시도한다. axios가 자동으로 만드는 객체다
- `original._retry` — 재시도 플래그. axios에 원래 없는 필드이고 우리가 임의로 추가한 것이다. 앞에 `_`를 붙이는 건 "내부적으로 쓰는 임시 필드"라는 관례적 표시일 뿐, 기능적으로 이름은 무관하다
- `withCredentials: true` — HttpOnly 쿠키를 요청에 함께 전송하도록 명시. 이게 없으면 쿠키가 안 붙어서 refresh 엔드포인트가 토큰을 못 읽는다
- `return api(original)` — 새 토큰으로 원 요청을 재시도. 컴포넌트는 이 재시도의 성공 응답만 받아서 실패 사실을 알지 못한다

### `_retry`가 무한 루프를 막는 원리

```
1. 원래 요청 → 401 수신
2. original._retry 없음 → if 조건 통과 → refresh 시도 → original._retry = true 세팅
3. api(original) 로 재시도 → 또 401 수신
4. 인터셉터 재진입 → !original._retry = !true = false → if 블록 진입 안 함
5. Promise.reject(error) → catch로 빠져서 로그아웃
```

`api(original)`로 재시도할 때 `original` 객체를 그대로 넘기기 때문에, 재시도 요청이 또 401을 받으면 인터셉터가 다시 실행되지만 `_retry`가 이미 `true`라 조건을 통과하지 못한다.

### 401은 토큰 만료만을 의미하지 않는다

401이 발생하는 경우:
- Access Token 만료
- Access Token 위조 (시그니처 불일치)
- Authorization 헤더 자체가 없음
- 탈퇴된 계정

현재 구현은 401이면 무조건 refresh를 시도한다. 토큰 위조나 헤더 누락으로 인한 401에도 refresh를 시도하지만, `_retry` 플래그 덕분에 무한 루프는 막히고 refresh 실패 시 로그아웃으로 빠져서 결과적으로 동작은 맞다.

엄밀하게 짜려면 백엔드가 에러 코드를 세분화해서 내려주는 방법이 있다:

```json
{ "detail": "token_expired" }
{ "detail": "token_invalid" }
```

프론트가 `error.response.data.detail === 'token_expired'` 일 때만 refresh를 시도하는 방식이다. 외부 공개 API에서는 HTTP 표준인 `WWW-Authenticate` 헤더를 쓰기도 한다:

```
WWW-Authenticate: Bearer error="invalid_token", error_description="Token has expired"
```

RFC 6750 (OAuth 2.0 Bearer Token 스펙)에 정의된 표준이다. 백엔드-프론트가 같은 팀이면 `detail` 필드로 에러 코드를 내려주는 게 더 간단해서 `WWW-Authenticate`까지 쓰는 경우는 외부 공개 API에서 주로 사용한다.

### 백엔드 refresh 엔드포인트

```python
# backend/app/api/v1/auth.py
@router.post("/auth/refresh")
def refresh_access_token(
    refresh_token: str = Cookie(default=None),  # 쿠키에서 자동으로 읽음
    db: Session = Depends(get_db),
):
    if not refresh_token:
        raise HTTPException(status_code=401, detail="Refresh token missing")

    user_id = redis_client.get(f"refresh:{refresh_token}")  # Redis 조회
    if not user_id:
        raise HTTPException(status_code=401, detail="Invalid or expired refresh token")

    user = db.query(User).filter(User.id == int(user_id)).first()
    if not user:
        raise HTTPException(status_code=401, detail="User not found")

    new_access = create_access_token({"sub": str(user.id), "email": user.email, "role": user.role})
    return {"access_token": new_access, "token_type": "bearer"}
```

- `Cookie(default=None)` — FastAPI가 요청 쿠키에서 `refresh_token` 값을 자동으로 꺼내 주입한다
- `redis_client.get(key)` — 키가 없거나 TTL이 만료됐으면 `None` 반환. 별도 만료 비교 로직이 필요 없다
- 검증 성공 시 새 Access Token만 JSON으로 반환한다 (Refresh Token은 재발급하지 않음)

## 4단계: 로그아웃

**경로:** `AuthContext.logout()` → `/auth/logout` → Redis 삭제 + 쿠키 만료

```python
# backend/app/api/v1/auth.py
@router.post("/auth/logout")
def logout(
    response: Response,
    refresh_token: str = Cookie(default=None),
):
    if refresh_token:
        redis_client.delete(f"refresh:{refresh_token}")  # Redis에서 즉시 무효화
    response.delete_cookie(key="refresh_token", path="/api/v1/auth")
    return {"message": "Logged out"}
```

```js
// frontend/src/contexts/AuthContext.jsx
const logout = async () => {
  try {
    await api.post('/auth/logout', {}, { withCredentials: true });
  } finally {
    setUser(null);
    setIsLoggedIn(false);
    localStorage.removeItem('user');
    useAuthStore.getState().clearAccessToken();
  }
};
```

- `redis_client.delete(key)` — 키가 없어도 에러 없이 동작한다(멱등성). 이미 만료됐어도 안전하게 호출 가능
- `finally` — API 호출 실패 여부와 관계없이 클라이언트 상태를 초기화한다. 서버가 죽어도 로컬에서는 로그아웃된다
- Redis에서 키를 삭제하면 Refresh Token이 즉시 무효화된다. 30일 수명이 남아 있어도 그 토큰으로 더 이상 갱신 불가

## 흐름 요약표

| 단계 | 위치 | 하는 일 |
|------|------|---------|
| 1 | `security.py` | `create_refresh_token()` — 랜덤 불투명 토큰 생성 |
| 1 | `auth.py` | Redis에 `refresh:{token}` → `user_id` 저장 (TTL 30일) |
| 1 | `auth.py` | Refresh Token을 HttpOnly 쿠키로 전송 |
| 2 | `api.js` (요청 인터셉터) | Access Token을 Bearer 헤더에 자동 부착 |
| 3 | `api.js` (응답 인터셉터) | 401 수신 시 `/auth/refresh` 자동 호출 후 원 요청 재시도 |
| 3 | `auth.py` `/auth/refresh` | 쿠키의 Refresh Token을 Redis 조회로 검증, 새 Access Token 반환 |
| 4 | `auth.py` `/auth/logout` | Redis 키 삭제 + 쿠키 만료 |
| 4 | `AuthContext.jsx` | 클라이언트 상태 초기화 |

## 왜 Refresh Token은 JWT가 아닌 불투명 토큰인가

### Refresh Token의 핵심 가치는 "즉시 무효화"

Refresh Token이 필요한 순간은 대부분 긴급 상황이다:
- 사용자 로그아웃
- 계정 정지 / 탈퇴
- 토큰 탈취 의심 → 강제 로그아웃
- 비밀번호 변경 → 모든 기기 로그아웃

JWT는 서버가 발급 후 기억하지 않는다. 취소하려면 블랙리스트를 따로 만들어야 하는데, 그 순간 매 요청마다 블랙리스트를 조회하게 되어 stateless의 장점이 사라진다.

불투명 토큰은 처음부터 서버(Redis)에 저장되어 있으니 `redis.delete(key)` 한 줄로 즉시 무효화된다.

### "Refresh Token을 JWT로 만들고 DB에 저장하면 안 되나?"

결론: 어차피 불투명 토큰과 동일하다.

JWT를 DB에 저장해서 검증하는 순간:
- JWT의 장점인 stateless(서버가 기억 안 해도 됨)가 사라짐
- 결국 매 요청마다 DB 조회 → 불투명 토큰과 똑같은 구조
- JWT는 불투명 토큰보다 문자열이 더 길기만 함 (헤더.페이로드.시그니처)

Refresh Token은 어차피 서버에 저장해야 하니, JWT로 만들 이유가 없다. 단순한 랜덤 문자열이 더 합리적인 선택이다.

| | Access Token (JWT) | Refresh Token (불투명) |
|---|---|---|
| 서버 저장 여부 | 저장 안 함 | Redis에 저장 |
| 검증 방식 | 시그니처만 확인 | 서버 조회 |
| 이유 | 매 요청마다 빠르게 | 어차피 조회하니 단순하게 |

### 탈취 시나리오와 한계

불투명 토큰도 탈취 자체를 막지는 못한다. 해커가 Refresh Token을 훔쳐서 Access Token 발급을 요청하면 서버는 그냥 발급해준다.

불투명 토큰의 가치는 **탈취를 인지했을 때 즉시 대응**할 수 있다는 것이다. JWT였다면 만료일(30일)까지 막을 방법이 없지만, 불투명 토큰은 즉시 삭제로 무효화된다.

더 고급 방어로 **Refresh Token Rotation**이 있다: 갱신 요청마다 Refresh Token을 새로 발급하고 이전 토큰을 폐기한다. 이미 폐기된 토큰으로 갱신 요청이 오면 탈취 의심으로 판단해 해당 유저의 모든 Refresh Token을 강제 폐기할 수 있다. 현재 구현에는 적용되어 있지 않다.

## 질문 포인트 (공부 시 확인할 것)

- JWT(Access Token)와 불투명 토큰(Refresh Token)의 차이 — 왜 Refresh는 JWT가 아닌 랜덤 문자열인가
- `withCredentials: true` 없이는 왜 쿠키가 전송되지 않는가 (CORS와 쿠키의 관계)
- `original._retry` 플래그가 없으면 어떤 무한 루프가 발생하는가
- `path="/api/v1/auth"` 쿠키 경로 제한이 왜 보안에 도움이 되는가
- Redis를 쓰는 이유 — PostgreSQL 테이블로 저장하면 어떤 단점이 있는가
