## 목표

- 핀볼 게임 음악/이미지 등 큰 정적 리소스를 첫 게임 시작 시 다운로드 후 캐시
- 파일이 바뀔 때만 재다운로드, 그 외에는 캐시 재사용
- 불필요한 트래픽 절감

## 현재 상태

- 정적 리소스는 `frontend/public/audio/`, `frontend/public/images/` 에 위치
- 빌드 시 `dist/`에 그대로 복사됨 (번들에 포함되지 않음 — 구조는 올바름)
- nginx.conf에 `Cache-Control` 헤더 없음 → 브라우저가 매번 재다운로드

### 주요 파일 크기

| 파일 | 크기 |
|------|------|
| audio/back_bgm.mp3 | 4.3MB |
| images/pinball_back.png | 2.2MB |
| audio/ball_hit.wav | 564KB |
| images/newbg.jpg | 564KB |
| audio/lifedown.wav | 368KB |
| 기타 | 소규모 |

## 구현 방향

### 1단계 — nginx Cache-Control 설정

```nginx
# index.html — 항상 최신 확인
location = /index.html {
    add_header Cache-Control "no-cache";
}

# JS/CSS 번들 — Vite가 해시를 파일명에 붙이므로 장기 캐시 안전
location ~* \.(js|css)$ {
    add_header Cache-Control "max-age=31536000, immutable";
}

# 정적 리소스 (audio, images) — 장기 캐시
location ~* \.(mp3|wav|ogg|png|jpg|webp|gif|svg)$ {
    add_header Cache-Control "max-age=31536000, immutable";
}
```

**캐시 무효화 전략:**
- JS/CSS: Vite 빌드 시 파일명에 해시 자동 삽입 → 변경 시 파일명이 달라져 자동 무효화
- audio/images: `public/`의 파일은 해시 미적용 → 파일 변경 시 파일명에 버전 수동 추가 필요 (예: `back_bgm_v2.mp3`)

### 2단계 — Lazy Load (게임 시작 시점에만 다운로드)

현재는 브라우저가 페이지 진입 시 자동 로드. 게임 시작 버튼 클릭 시점에 명시적으로 fetch하여 캐시에 올리는 방식으로 변경.

```js
// 게임 시작 버튼 클릭 핸들러에서
const preloadAssets = async () => {
  const assets = ['/audio/back_bgm.mp3', '/images/pinball_back.png', ...]
  await Promise.all(assets.map(url => fetch(url)))
}
```

첫 다운로드 이후에는 브라우저 캐시에서 즉시 로드.

## 적용 대상 nginx 파일

- 실제 배포: `nginx.conf` (루트) — Docker nginx 컨테이너에 마운트됨
- 로컬 테스트용: `nginx/nginx.conf`

## 동작 원리

### 정적 파일 GET 엔드포인트

별도 API 코드 없이 nginx가 자동으로 파일 경로를 GET 엔드포인트로 노출한다.

```
nginx root → /etc/nginx/html (dist/ 파일들이 복사된 곳)

dist/audio/back_bgm.mp3       →  GET /audio/back_bgm.mp3
dist/images/pinball_back.png  →  GET /images/pinball_back.png
dist/assets/main.a3f7c2.js    →  GET /assets/main.a3f7c2.js
```

FastAPI 등 백엔드 코드 수정 불필요. nginx.conf에 Cache-Control 헤더만 추가하면 됨.

### 음악/이미지가 핀볼 페이지 진입 시 로드되는 이유

JS 번들과 정적 파일은 로드 시점이 다르다.

- **JS 번들**: `import` 문으로 번들에 포함 → 첫 접속 시 전부 다운로드
- **audio/images**: JS 코드가 실행될 때 브라우저가 필요한 시점에 서버에 요청

```
핀볼 페이지 진입
    → React가 Pinball 컴포넌트 마운트
    → JS 코드 실행: new Audio('/audio/back_bgm.mp3')
    → 브라우저: GET /audio/back_bgm.mp3 요청
    → nginx가 파일 응답
```

캐시 미적용 시 핀볼 페이지 진입마다 위 흐름 반복.

### Cache-Control 적용 후 동작

```
첫 방문: 서버에서 파일 받음 → 브라우저 로컬에 저장
재방문: 로컬 캐시 유효 → 서버 요청 없이 즉시 로드
```

Network 탭에서 캐시 재사용 시 파일 옆에 `(disk cache)` 또는 `(memory cache)` 표시됨.
`max-age=31536000` = 브라우저에게 "1년간 유효"라고 알려주는 것.

## 실험 기록

### Before (캐시 적용 전)

**측정 방법:** 크롬 개발자 도구 Network 탭, `Ctrl+Shift+R` (강제 새로고침, 캐시 무시)

| 페이지 | requests | transferred | resources | Finish |
|--------|----------|-------------|-----------|--------|
| 메인 페이지 (`/`) | 9 | 1.9MB | 2.2MB | 463ms |
| 핀볼 페이지 (`/pinball`) | 16 | 1.9MB | 6.6MB | - |

**분석:**
- 메인과 핀볼의 transferred는 같지만 resources 차이(2.2MB vs 6.6MB)가 핀볼 진입 시 audio/image 파일이 추가 로드됨을 의미
- transferred < resources 격차는 일부 파일에 gzip 압축이 적용되고 있음을 의미
- `Ctrl+Shift+R` 기준이므로 현재 캐시 설정 없이 매 접속마다 이만큼 재다운로드 중

**측정 2: 브라우저 기본 캐시 동작 확인**

조건: 핀볼 페이지 이전 방문으로 audio/images 캐시된 상태 → 메인 페이지 `Ctrl+Shift+R` → 핀볼 페이지 이동

| requests | transferred | resources |
|----------|-------------|-----------|
| 12 | 569B | 4.4MB |

**분석:**
- transferred 569B — JS/HTML 등 일부만 서버에서 받음
- resources 4.4MB — audio/images는 브라우저 캐시에서 로드 (서버 요청 없음)
- nginx에 Cache-Control 미설정이지만 브라우저가 자체적으로 캐시를 유지하고 있음
- `Ctrl+Shift+R`은 현재 페이지 캐시만 무효화하고, 다른 페이지에서 캐시된 리소스는 유지됨
- nginx Cache-Control 설정 시 이 동작을 **명시적으로 제어**할 수 있게 됨 (만료 기간, 무효화 타이밍 등)
