## 논의 내용

### 주제 1: 공지사항 목록 페이지네이션

현재 `getNotices()`는 전체 데이터를 한 번에 받아온다.  
일정 개수씩 나눠 받아오는 페이지네이션 도입을 검토한다.

#### 방식 비교

**A안: 오프셋 페이지네이션 (Offset Pagination)**

```
GET /api/v1/notices?skip=0&limit=10   → 1페이지
GET /api/v1/notices?skip=10&limit=10  → 2페이지
```

- 구현 단순, 페이지 번호 UI와 자연스럽게 연결됨
- 단점: 데이터 추가/삭제 시 중복/누락 발생 가능 (공지사항은 빈도 낮아 실질적 문제 없음)
- 전체 개수(`total`)를 별도로 내려줘야 페이지 번호 계산 가능

API 응답 예시:
```json
{
  "items": [...],
  "total": 32
}
```

**B안: 커서 페이지네이션 (Cursor Pagination)**

```
GET /api/v1/notices?cursor=<last_id>&limit=10
```

- 데이터 추가/삭제에 안정적
- "더 보기" 버튼 또는 무한 스크롤 UI에 자연스럽게 연결됨
- 특정 페이지로 바로 이동 불가 (페이지 번호 UI 구현 어려움)

---

#### UI 방식 비교

**가. 페이지 번호 버튼** (`1 2 3 ... 다음`)
- 사용자가 원하는 페이지로 바로 이동 가능
- A안(오프셋)과 자연스러운 조합
- total 값 필요

**나. 이전/다음 버튼만** (`← 이전` `다음 →`)
- 구현 단순
- A안, B안 모두 호환

**다. 무한 스크롤 / 더 보기 버튼**
- B안(커서)과 자연스러운 조합
- 공지사항 게시판 성격과 다소 맞지 않을 수 있음

#### IndexedDB fallback 처리

현재 `noticeService.js`는 API 실패 시 IndexedDB에서 전체 목록을 반환한다.  
페이지네이션 도입 시 fallback도 같은 방식으로 잘라서 반환해야 일관성 유지된다.

```js
// 현재
export const getNotices = async () => {
  try {
    const res = await axios.get('/api/v1/notices');
    return res.data;
  } catch {
    await seedNoticesIfEmpty();
    return getAllNoticesFromDB(); // 전체 반환
  }
};

// 변경 후 (오프셋 방식 예시)
export const getNotices = async (skip = 0, limit = 10) => {
  try {
    const res = await axios.get('/api/v1/notices', { params: { skip, limit } });
    return res.data; // { items, total }
  } catch {
    await seedNoticesIfEmpty();
    const all = await getAllNoticesFromDB();
    return {
      items: all.slice(skip, skip + limit),
      total: all.length,
    };
  }
};
```

## 결정사항

### 주제 1: 공지사항 목록 페이지네이션

- **API 방식**: A안 — 오프셋 페이지네이션 (`?skip=0&limit=10`)
- **UI 방식**: 페이지 번호 버튼 (`1 2 3 ...`)
- **한 페이지 당 공지 수**: 10개
- **총 페이지 수 계산**: `Math.ceil(total / 10)` — API가 `{ items, total }` 구조로 응답해야 함
- **page 상태 관리**: URL 쿼리스트링 방식 (`?page=2`) — 뒤로가기/새로고침 시 페이지 유지, `useSearchParams` 사용
- **페이지 버튼 노출 개수**: 한 번에 5개씩 표시 (`< 1 2 3 4 5 >` 형태, 현재 페이지 기준 슬라이딩)

### noticeService.test.js — 수정 필요

현재 `getNotices` 테스트는 인자 없이 `GET /api/v1/notices`를 호출하는 것을 검증한다.  
페이지네이션 도입 후에는 `skip`, `limit` 파라미터가 추가되므로 아래 테스트가 깨진다.

```js
// 현재 (깨질 테스트)
it('GET /api/v1/notices를 호출한다', async () => {
  axios.get.mockResolvedValue({ data: [] });
  await getNotices();
  expect(axios.get).toHaveBeenCalledWith('/api/v1/notices'); // params 없어서 실패
});

it('서버 응답 data 배열을 반환한다', async () => {
  axios.get.mockResolvedValue({ data: mockData });
  const result = await getNotices();
  expect(result).toEqual(mockData); // 응답 구조가 { items, total, skip, limit }로 바뀌어서 실패
});
```

수정 방향:
```js
// 변경 후
it('GET /api/v1/notices를 skip, limit 파라미터와 함께 호출한다', async () => {
  axios.get.mockResolvedValue({ data: { items: [], total: 0 } });
  await getNotices(0, 10);
  expect(axios.get).toHaveBeenCalledWith('/api/v1/notices', { params: { skip: 0, limit: 10 } });
});

it('서버 응답의 { items, total } 구조를 반환한다', async () => {
  const mockData = { items: [{ id: 1, title: '공지1' }], total: 1 };
  axios.get.mockResolvedValue({ data: mockData });
  const result = await getNotices(0, 10);
  expect(result).toEqual(mockData);
});
```
