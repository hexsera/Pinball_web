## 목적

포트폴리오·학습 목적. 브라우저에서 캐시를 처리하는 방법을 공지사항 게시판에 직접 적용하며 익힌다.

## 현재 구조

### 페이지
- `NoticeListPage` — 목록 + 페이지네이션 (`skip`, `limit` 파라미터)
- `NoticeDetailPage` — 본문 (Tiptap HTML을 `dangerouslySetInnerHTML`로 렌더링, 이미지 포함 가능)
- `NoticeWritePage` — Admin 전용 작성

### API (`noticeService.js`)
- `GET /api/v1/notices?skip=&limit=` → `{ items, total }`
- `GET /api/v1/notices/{id}` → 공지 단건
- `POST /api/v1/notices`, `DELETE /api/v1/notices/{id}` — Admin 전용

### 현재 fetch 방식
- `useEffect` + `useState`로 직접 fetch (React Query 없음)
- API 실패 시 IndexedDB fallback (`noticeIndexDB.js`)

## 캐시 방식 정리

### 캐시 방식 두 가지

| | 타임아웃 방식 | ETag 방식 |
|--|--|--|
| 요청 발생 여부 | 시간 내엔 요청 안 보냄 | 매번 요청 보냄 |
| 데이터 신선도 | 시간 지나야 갱신 | 항상 최신 확인 가능 |
| 특징 | 네트워크 요청 자체를 생략 | 요청은 하되 바디 전송 생략 (304) |

### 캐시 레이어 세 가지

| | React Query | HTTP Cache-Control | ETag |
|--|--|--|--|
| 캐시 위치 | JS 메모리 (앱 안) | 브라우저 내장 캐시 (앱 밖) | HTTP 레벨 |
| 캐시 대상 | fetch/axios로 받은 데이터 (JS 객체) | HTTP 응답 자체 (헤더+바디 원문) | 응답 바디 변경 여부 |
| 탭 닫으면 | 사라짐 | 유지됨 | - |
| 서버 설정 필요 | 없음 | 있음 | 있음 (ETag 헤더 발급) |
| 무효화 제어 | 앱 코드에서 `invalidateQueries` | 브라우저가 자동 (앱 개입 불가) | 서버가 판단 |
| 주로 쓰는 곳 | API 응답 캐시 표준 | 정적 리소스 (JS, CSS, 이미지 파일) | 응답이 무거운 API |

**담당 역할 한 줄 정리**
- React Query — "언제 다시 fetch할지" 결정
- HTTP Cache-Control — "브라우저가 요청 자체를 보낼지" 결정
- ETag — "요청은 보내되, 서버가 바디를 보낼지" 결정

셋은 각자 다른 레이어에서 독립적으로 동작하고, 조합해서 쓸 수도 있음. React Query가 staleTime 만료 후 재요청할 때 ETag를 함께 보내도록 axios 인터셉터를 구성하는 식.

### 각각 왜 쓰는가

**React Query를 쓰는 이유**
캐시 무효화 타이밍을 앱 코드에서 직접 제어할 수 있기 때문. Admin이 글을 쓰거나 삭제한 순간 `invalidateQueries`로 즉시 캐시를 날릴 수 있음. 또한 여러 컴포넌트가 같은 데이터를 구독할 때 중복 요청을 자동으로 제거해줌. API 응답 캐시의 사실상 표준.

**HTTP Cache-Control을 쓰는 이유**
서버가 응답 헤더에 `max-age`를 선언하면 브라우저가 알아서 처리 — 앱 코드 변경 없이 캐시가 적용됨. 탭을 닫았다 열어도 캐시가 유지됨 (디스크 캐시). JS/CSS/이미지 파일처럼 배포 전까지 변경 없는 정적 리소스에 특히 적합. 단, 무효화를 앱이 제어할 수 없어서 변경이 잦은 API에는 부적합.

**ETag를 쓰는 이유**
응답이 무거운데 변경 여부를 먼저 확인하고 싶을 때. 요청은 항상 보내지만 데이터가 안 바뀌었으면 서버가 304만 응답해 바디 전송을 생략함. base64 이미지가 포함된 공지 상세처럼 응답 크기가 클수록 절약 효과가 큼.

### 어디서 설정하는가

| 목표 | 설정 위치 | 이유 |
|--|--|--|
| JS/CSS 파일 캐시 | Nginx | Nginx가 정적 파일을 직접 서빙 |
| API 응답 Cache-Control | FastAPI | Nginx는 API 요청을 FastAPI로 통과시키기만 함 |
| API 응답 ETag | FastAPI | 위와 같음 |
| 메모리 캐시 | React Query | JS 레벨에서 동작 |

공지사항 상세 API 캐시가 목표이므로 Cache-Control, ETag 모두 FastAPI에서 설정.

### 이 프로젝트 방향

API 응답 캐시는 React Query가 주력. 이유: Admin POST/DELETE 후 `invalidateQueries`로 즉시 무효화 가능. HTTP Cache-Control은 무효화가 어려워 API에는 부적합 (max-age 동안 브라우저가 무조건 캐시 사용). ETag는 학습 목적으로 추가 검토.

## 캐시 레이어 설계

### 1단계: React Query (메모리 캐시)

| 대상 | queryKey | 비고 |
|------|----------|------|
| 목록 | `['notices', skip]` | 페이지마다 독립 캐시 |
| 상세 | `['notices', id]` | 단건 |

- Admin POST/DELETE 후 `invalidateQueries(['notices'])` 한 번으로 목록 전체 + 상세 모두 무효화
- 공지사항은 변경 빈도가 낮아 Optimistic update 불필요
- IndexedDB fallback은 역할 중복 → 제거 또는 "완전 오프라인" 용도로 분리 결정 필요

### 2단계: HTTP Cache-Control (브라우저 캐시)

- 현재 FastAPI가 `Cache-Control` 헤더를 내려주지 않아 브라우저 캐시 불가
- GET 응답에 `Cache-Control: public, max-age=60` 추가 → 브라우저가 네트워크 요청 자체를 생략
- React Query의 staleTime과 별개로 동작 → 두 레이어의 차이를 직접 체감할 수 있어 학습 가치 높음

### 3단계: ETag (조건부 요청)

- 데이터 변경 없으면 서버가 304만 응답, 바디 전송 생략
- 구현 복잡도가 높아 1·2단계 완료 후 진행

## 구현 순서

```
1단계: React Query 도입 → 목록/상세 캐시 + invalidateQueries 무효화
2단계: FastAPI Cache-Control 헤더 추가 → 브라우저 캐시 레이어 체감
3단계: ETag → 304 흐름 (여유 있으면)
```

## 캐시 대상 확정

목표: 공지사항 데이터가 바뀌지 않으면 서버로 요청을 보내지 않는다 (트래픽 절감).

- 캐시 대상: `GET /api/v1/notices/{id}` (상세) → `GET /api/v1/notices` (목록) 순서로 진행
- POST/DELETE는 쓰기 요청이므로 캐시 대상 아님
- 목록 캐시는 나중에 다루기로 보류

### 상세를 먼저 하는 이유

- 본문에 이미지가 base64로 포함되어 응답이 무거움
- 무거운 응답 전체를 매번 다시 받는 것이 낭비 → 캐시 효과가 큼

## 이미지 저장 방식 논의

현재 Tiptap 에디터에서 이미지를 `canvas.toDataURL('image/jpeg', 0.8)`로 base64 변환 후 HTML 안에 통째로 저장.

이로 인해 **이미지만 따로 캐시하는 것은 불가능**. 브라우저는 API 응답 전체를 하나의 덩어리로 보기 때문.

일반적인 웹사이트는 이미지를 파일 스토리지(S3 등)에 저장하고 DB에는 URL만 기록해 이미지를 독립 리소스로 캐시할 수 있음. 이 프로젝트는 구현 편의상 base64를 선택한 것. → 별도 이슈로 기록: [2026-05-18-파일-스토리지-고려.md](../issue/2026-05-18-파일-스토리지-고려.md)

## 구현 확정

### 1. 무엇을 할 것인가

공지사항 목록(`GET /api/v1/notices`)과 상세(`GET /api/v1/notices/{id}`) 응답을 캐시한다. 상세는 base64 이미지가 포함되어 응답이 무거우므로 반복 전송을 줄이는 것이 목표. 공지사항은 1~2일에 하나 올라오는 빈도.

### 2. 어떤 방법을 쓸 것인가

| 대상 | 방법 | 값 |
|--|--|--|
| 목록 | Cache-Control max-age | 10분 (`max-age=600`) |
| 상세 | Cache-Control max-age + ETag | 1시간 (`max-age=3600`) + ETag |

- **Cache-Control max-age** — 시간 안에는 브라우저가 요청 자체를 생략
- **ETag** — max-age 만료 후 재요청 시 내용이 안 바뀌었으면 304만 응답, 바디 전송 생략
- 목록에 ETag를 안 쓰는 이유: 글 하나만 바뀌어도 ETag가 바뀌어 매번 새 바디를 받아야 해서 이점이 없음

### 3. 왜 이 방법인가

- 공지사항은 Admin만 수정/삭제하므로 변경 빈도가 낮아 max-age를 길게 잡아도 무방
- 이미지가 base64로 포함되어 응답이 무거움 → ETag로 바디 전송을 생략하면 절약 효과가 큼
- React Query는 같은 데이터를 여러 컴포넌트가 동시에 구독할 때 빛나는데, 공지사항은 그런 구조가 아니라 부적합
- "브라우저 캐시를 배운다"는 목적에 ETag + Cache-Control이 더 직결됨

### 4. 어디를 수정할 것인가

- **FastAPI** `backend/app/api/v1/notices.py`
  - `GET /` — `Cache-Control: max-age=600` 헤더 추가
  - `GET /{notice_id}` — `Cache-Control: max-age=3600` + `ETag` 헤더 추가
