# PostgreSQL 캐시 기능 논의

## 개요

Redis 없이 PostgreSQL 레벨에서 캐시 역할을 부여하는 방법 탐색.

## PostgreSQL 캐시 옵션 비교

| 방법 | 특징 | 단점 |
|------|------|------|
| **Materialized View** | 쿼리 결과를 디스크에 저장, 테이블처럼 조회 가능 | 실시간 아님, REFRESH 비용 존재 |
| **Unlogged Table** | WAL 미기록 → 쓰기 2~3배 빠름 | 재시작/크래시 시 데이터 소실 |
| **pg_prewarm** | 특정 테이블/인덱스를 shared_buffer에 미리 올림 | 쿼리 결과 캐싱 아님, 콜드 스타트 방지 용도 |
| **인덱스 튜닝** | 느린 쿼리 식별 후 인덱스로 해결 | 캐시가 아닌 근본 해결 접근 |

## 프로젝트 적용 후보

- **공지사항 목록**: Materialized View + 주기적 REFRESH
- **월간 랭킹**: `MonthlyScore`에 비정규화 중이므로 Materialized View로 집계 뷰 구성 가능

## 결정 사항

- [ ] 어떤 데이터에 캐시를 붙일지 확정
- [ ] Materialized View 구현 방식 및 REFRESH 주기 결정

## 목적 및 방향

취업 포트폴리오용 기술 경험이 목적. mock 데이터로 대용량 상황을 재현하고 Before/After 수치를 측정해 포트폴리오에 기록할 계획.

- 적용 후보: `game_visits` 테이블 (집계 쿼리 → Materialized View)
- 순서: mock 데이터 수십만 건 insert → EXPLAIN ANALYZE 측정 → MV 적용 → 재측정 → 수치 기록
