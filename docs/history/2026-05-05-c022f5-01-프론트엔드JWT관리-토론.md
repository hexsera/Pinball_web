# 프론트엔드 JWT 관리 토론

- 작성일: 2026-05-05
- 주제 ID: c022f5

## 배경

현재 인증 방식은 localStorage 기반 세션이며, 로그인 시 `{user_id, email, nickname, role}` 객체를 저장한다. JWT(JSON Web Token) 도입을 검토하면서 프론트엔드에서 토큰을 어디에, 어떻게 보관·갱신·검증할지 정한다.

---

## 1. 토큰 저장 위치

### 선택지 비교

| 방식 | 장점 | 단점 |
|------|------|------|
| localStorage | 구현 단순, 새로고침 후에도 유지 | XSS에 취약 (스크립트가 직접 읽음) |
| sessionStorage | 탭 종료 시 자동 만료 | 탭 간 공유 불가, XSS 노출은 동일 |
| 메모리(JS 변수) | XSS로 직접 탈취 어려움 | 새로고침 시 소실 → refresh 필요 |
| HttpOnly 쿠키 | JS에서 접근 불가 → XSS 방어 | CSRF 대응 필요, 백엔드 협조 필수 |

### 일반적인 실무 선택

**Access는 메모리, Refresh는 HttpOnly + Secure + SameSite 쿠키.**

- localStorage는 XSS 한 번에 전부 털리므로 OWASP/Auth0/Okta 모두 비권장
- 메모리에 두면 새로고침 때 사라지지만, Refresh 쿠키로 즉시 재발급해서 가린다
- 도메인이 다르면 SameSite=None + Secure + CSRF 토큰 병행

### 결정

→ **Access Token: 메모리 / Refresh Token: HttpOnly 쿠키** 방향으로 진행.

---

## 2. Access / Refresh 토큰 분리

- **Access Token**: 짧은 만료(예: 15분), API 요청 시 `Authorization: Bearer ...` 헤더에 첨부
- **Refresh Token**: 긴 만료(예: 7~30일), Access Token 갱신 전용
- Refresh는 HttpOnly 쿠키, Access는 메모리에서만 다룸

---

## 3. 토큰 갱신 흐름

### 핵심 개념

**"네트워크 요청을 한 곳에서 가로채서 토큰 문제를 알아서 해결한다."**

화면 코드 곳곳에서 토큰을 신경 쓰면 빼먹기 쉽기 때문에, HTTP 요청을 보내는 통로(axios)에 검문소 두 개를 둔다.

- **나가는 검문소 (요청 인터셉터)**: 모든 요청에 Access 토큰을 자동으로 부착
- **들어오는 검문소 (응답 인터셉터)**: 401(토큰 만료) 응답을 가로채 토큰 갱신

### 갱신 후 재시도

만료된 토큰으로 보낸 요청을 그냥 실패시키지 않는다.

1. 401이 왔다 → "토큰이 죽었구나"
2. Refresh 토큰으로 새 Access 토큰을 받아온다
3. 방금 실패했던 요청을 새 토큰으로 한 번 더 보낸다
4. 성공한 응답을 처음부터 성공한 것처럼 화면에 돌려준다

화면 입장에서는 "요청 한 번 보냈고 성공했다"로만 보인다. 만료가 있었는지조차 모른다.

### 동시 401 처리 (가장 실수하기 쉬운 부분)

화면을 열면 보통 요청이 여러 개 한꺼번에 나간다. 토큰 만료 직후라면 이 요청들이 거의 동시에 401을 받는다. 순진하게 만들면 갱신 요청이 5번, 10번 중복으로 나가고, Rotation을 쓰면 첫 번째만 성공하고 나머지가 다 실패해 로그아웃되는 사고가 난다.

해결:
- **첫 번째 401만 갱신을 시작한다**
- 그 사이 401을 받은 다른 요청들은 **"지금 갱신 중이니까 끝날 때까지 기다려"** 라고 줄을 세운다
- 갱신이 끝나면 줄 서 있던 요청들을 새 토큰으로 한꺼번에 다시 보낸다
- 갱신이 실패했으면 줄 선 요청들도 모두 실패 처리하고 로그아웃 화면으로 보낸다

비유: 티켓 창구에 한 명이 갱신하러 들어가 있고, 뒷사람들은 기다렸다가 새 티켓을 받아 한 번에 통과한다.

### 로그인/로그아웃은 인터셉터의 영역이 아니다

- **로그인**: `POST /api/v1/auth/login` 으로 최초 발급. 토큰이 없으니 인터셉터가 개입할 일이 없다.
- **인터셉터**: 이미 발급된 토큰을 부착·갱신·재시도
- **로그아웃**: `POST /api/v1/auth/logout` 으로 Refresh 쿠키 만료. 메모리의 Access도 비움.

비유:
- 로그인 = 출입카드 발급
- 인터셉터 = 매번 카드를 자동으로 찍고, 만료되면 갱신 창구에서 새 카드로 바꿔 다시 찍는 자동화 장치
- 로그아웃 = 카드 반납

## 4. "메모리 저장"과 "React 바깥 인터셉터"의 의미

### 메모리에 저장한다는 것

- **메모리 = JS 변수**. 별도의 "전역 메모리" 개념이 React에 있는 게 아니다.
- 새로고침/탭 닫기 시 **JS 실행 환경이 리셋되어 모든 변수가 사라진다.**
- localStorage/쿠키는 디스크에 저장되어 살아남지만, 메모리는 그렇지 않다.

### 새로고침 시 사용자가 로그아웃되지 않는 이유

```
[페이지 새로고침]
1. JS 메모리 초기화 → Access 사라짐
2. 앱 시작되자마자 자동으로 /api/v1/auth/refresh 호출
3. 브라우저가 HttpOnly 쿠키(Refresh)를 자동으로 같이 보냄
4. 서버가 Refresh 검증 → 새 Access 응답
5. 새 Access를 메모리에 저장 → 사용자는 로그인된 상태로 화면을 본다
```

이 동작은 **5단계(localStorage 의존 제거)** 에서 완성된다. 그 전(1단계)에서는 새로고침 시 즉시 로그아웃되는 문제를 막기 위해 **localStorage에 임시 저장**하는 타협을 둔다.

### React 안 vs React 바깥

| 구분 | React 안 | React 바깥 |
|------|---------|-----------|
| 위치 | 함수 컴포넌트, 훅 | 일반 모듈 파일, 함수, axios 인터셉터 |
| 실행 시점 | 컴포넌트 렌더링 시 | 모듈 import 시 / 요청 발생 시 |
| 훅 사용 | 가능 (`useState`, `useContext` …) | **불가** |

### axios 인터셉터가 "React 바깥"인 이유

axios 인스턴스는 앱 전체에서 한 번만 만들어 공유한다. 보통 이런 모듈 파일에 산다:

```js
// src/lib/api.js
import axios from 'axios';

const api = axios.create({ baseURL: '/api/v1' });

api.interceptors.request.use((config) => {
  // ← 여기는 React 바깥. useContext 못 씀.
  const token = /* 어디선가 꺼내야 함 */;
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

export default api;
```

이 파일은 컴포넌트가 아니다. 모듈이 처음 import될 때 인터셉터가 등록되고, 이후 앱이 끝날 때까지 살아 있다. **React의 렌더링 사이클과 무관하게 동작**하므로 훅을 못 쓴다.

화면 컴포넌트는 그냥 `api`를 import해서 쓴다:

```jsx
// src/pages/ScoreList.jsx
import api from '../lib/api';

function ScoreList() {
  useEffect(() => {
    api.get('/scores').then(...);   // 인터셉터를 거쳐 토큰이 자동 부착됨
  }, []);
}
```

컴포넌트는 토큰 부착이나 401 처리를 신경 쓰지 않는다.

### 토큰 저장소는 "양쪽에서 모두 접근 가능한 곳"이어야 한다

토큰을 React Context에만 두면 인터셉터(컴포넌트 아님)에서 못 꺼낸다. `useContext`는 컴포넌트 안에서만 호출 가능하기 때문이다. 따라서 토큰은 인터셉터에서도 꺼낼 수 있는 위치에 둬야 한다.

### 선택지

#### (A) 모듈 스코프 변수 — 가장 단순

```js
// src/lib/authToken.js
let accessToken = null;
export const setAccessToken = (t) => { accessToken = t; };
export const getAccessToken = () => accessToken;
export const clearAccessToken = () => { accessToken = null; };
```

- 장점: 매우 단순. 인터셉터/컴포넌트 어디서든 함수로 접근.
- 단점: 토큰이 바뀌어도 React가 모름 → 자동 리렌더링 안 됨. 로그인 상태 UI에 부적합.

#### (B) Zustand 스토어 — 양쪽 모두 지원

```js
// src/store/authStore.js
import { create } from 'zustand';
export const useAuthStore = create((set) => ({
  accessToken: null,
  setAccessToken: (t) => set({ accessToken: t }),
  logout: () => set({ accessToken: null }),
}));
```

- 컴포넌트: `const t = useAuthStore((s) => s.accessToken)` — 변경 시 자동 리렌더링
- 인터셉터: `useAuthStore.getState().accessToken` — React 바깥에서도 접근 가능
- 장점: 한 곳에서 일관 관리. user 정보까지 같이 담기 좋음.
- 단점: 라이브러리 1개 추가 (`zustand`).

#### (C) 순수 React Context — 부적합

- 컴포넌트 안에서는 좋지만 인터셉터에서 접근 불가. 단독으로는 못 씀.

### 결정

- **Zustand 스토어 채택.**
- 이유: 학습 비용이 매우 낮아 1단계 구현 시간을 크게 늘리지 않음. 5단계 이후 user 정보·로그인 상태 UI까지 일관되게 다룰 수 있어 결과적으로 시간 절약.
- 설치: `npm install zustand` 한 줄.

## 5. 만료 처리

### 일반적인 실무 선택

- **기본은 401 반응형 갱신**: 401이 오면 그때 갱신. 단순하고 서버 시계가 진리이므로 클럭 스큐에 강함.
- **선제 갱신은 보조 수단**: `exp`를 디코딩해 만료 60~120초 전이면 미리 갱신. 긴 폼 작성 중 401로 깨지는 UX 문제를 줄일 때 사용.
- **수명 가이드**:
  - Access: 5~15분
  - Refresh: 7~30일 (절대 만료) + 슬라이딩 갱신 또는 회전
- **클럭 스큐 여유**: 만료 비교 시 30~60초 마진

## 8. 미결 사항 — 개념 설명과 결정

### 8-1. Refresh Token 회전(Rotation) 정책

**개념**: Refresh를 쓸 때마다 새 Refresh를 같이 발급하고, 옛날 Refresh는 즉시 무효화한다. 일회용.

```
[갱신 1회차] Refresh #1 사용 → 새 Access + Refresh #2. #1 무효화
[갱신 2회차] Refresh #2 사용 → 새 Access + Refresh #3. #2 무효화
```

**효과**: 도둑이 Refresh #1을 훔쳐 먼저 쓰면, 정상 사용자가 #1을 다시 쓰려는 순간 "이미 사용됨" 에러가 발생. 서버는 이를 탈취 의심으로 판단해 그 사용자의 모든 Refresh를 통째로 무효화한다. **탈취를 막진 못해도 빨리 감지하는 장치.**

**→ 결정: 사용한다.**

---

### 8-2. 다중 탭 토큰 갱신 동기화

**문제**: Access를 메모리에 저장하면 탭마다 따로 보관된다. 탭 A가 갱신해도 탭 B는 옛 Access를 들고 있어 또 401 → 탭 B가 다시 갱신 시도 → Rotation 정책 때문에 옛 Refresh는 이미 무효 → 탭 B가 갑자기 로그아웃되는 사고가 난다. 로그아웃도 마찬가지로 한 탭에서 한 행동이 다른 탭에 전달되지 않는다.

**해결**: 브라우저가 제공하는 탭 간 통신 채널 사용.
- **BroadcastChannel API**: 같은 사이트 모든 탭에 메시지 방송
- **storage 이벤트**: 옛날 방식

한 탭에서 갱신/로그아웃 시 다른 탭에 방송 → 다른 탭이 자기 메모리 업데이트.

**→ 결정: 지금은 적용하지 않음. 추후 추가 가능 항목으로 보류.**

---

### 8-3. 모바일 PWA / 앱 래퍼 쿠키 전략

**용어**:
- **PWA (Progressive Web App)**: 홈 화면에 앱처럼 설치 가능한 웹사이트. 실제로는 브라우저가 전체화면으로 사이트를 띄움.
- **앱 래퍼 (WebView 앱)**: 네이티브 앱(.apk, .ipa) 안에 브라우저 창 하나를 띄워 웹사이트를 보여주는 방식. Cordova, Capacitor, React Native WebView 등.

**문제**:
1. **출처가 다르다**: WebView가 `file://`, `capacitor://localhost` 같은 비표준 출처로 동작 → SameSite=Lax 쿠키가 자동으로 안 붙음
2. **WebView마다 쿠키 저장소가 다르다**: 안드로이드/iOS WebView는 각자 쿠키 통을 가짐. 앱 업데이트나 캐시 정리로 통째로 날아갈 수 있음. 모바일 브라우저와도 공유 안 됨.
3. **HttpOnly 쿠키를 네이티브 코드에서 못 읽는다**: 네이티브가 인증 토큰을 직접 꺼낼 수 없어 푸시 알림 등 네이티브 기능과 협력이 어려움.

**대안**:
- 앱 환경에서는 쿠키 대신 토큰을 **iOS Keychain / Android Keystore** 같은 OS 보안 금고에 저장
- 웹은 HttpOnly 쿠키, 앱은 네이티브 저장소로 환경별 분기

**→ 결정: 지금은 고려하지 않음. 향후 모바일 환경 도입 시 검토할 항목으로만 기록.**

## 9. 현재 결정 사항 요약

| 항목 | 결정 |
|------|------|
| Access Token 저장 위치 | **메모리(Zustand 스토어)** |
| Refresh Token 저장 위치 | **HttpOnly + Secure + SameSite 쿠키** |
| 상태 관리 라이브러리 | **Zustand** (인터셉터에서도 `getState()`로 접근) |
| 토큰 갱신 방식 | **axios 인터셉터에서 401 가로채 자동 갱신 + 동시 요청 큐잉** |
| 만료 처리 | **401 반응형 갱신을 기본**, 필요 시 선제 갱신 보조 |
| Refresh Token 회전 | **사용** |
| 다중 탭 동기화 | **현재 미적용** (추후 추가 가능) |
| 모바일 PWA / 앱 래퍼 대응 | **현재 미적용** (모바일 도입 시 재검토) |
| 권한 판단 최종 책임 | **백엔드** (프론트 UI 분기는 편의용) |

## 10. 단계별 구현 계획

시간 제약을 고려해 **각 단계가 그 자체로 완결되어 운영 가능한 단위**가 되도록 우선순위를 정한다. 시간이 부족하면 어느 단계에서 멈춰도 시스템은 정상 동작한다.

### 10-1. 1단계 — 최소 동작 JWT (필수) (완료)

**목표**: localStorage 세션을 JWT 기반으로 교체. 가장 단순한 형태.

- 백엔드: `/api/v1/auth/login` 응답에 **Access Token만** 발급 (수명 길게, 예: 1~7일)
- 프론트: Access Token을 **메모리(JS 변수 또는 Context/Zustand)** 에 저장
- 새로고침 대비: 일단 **localStorage에도 임시 저장** (1단계 한정 타협)
- axios 인스턴스에 **요청 인터셉터**만 추가 → 모든 요청에 `Authorization: Bearer ...` 자동 부착
- 401 응답이 오면 단순히 로그아웃 + 로그인 페이지로 보냄 (갱신 없음)
- 사용자 정보는 토큰 디코딩 또는 `/api/v1/users/me` 호출

**완결성**: 로그인/로그아웃/인증 요청이 모두 동작. Refresh 없이 Access 수명을 길게 잡아 사용성 확보.

**남는 약점**: localStorage 보관 → XSS 취약. Access 수명 길어 탈취 시 피해 큼.

---

### 10-2. 2단계 — Refresh Token 도입 (완료)

**목표**: Access를 짧게(15분) 만들고 자동 갱신으로 사용자가 모르게 한다.

- 백엔드: 로그인 시 **Refresh Token을 HttpOnly + Secure + SameSite=Lax 쿠키**로 발급
- 백엔드: `/api/v1/auth/refresh` 엔드포인트 추가 (쿠키 검증 → 새 Access 발급)
- 프론트: **응답 인터셉터** 추가 → 401 받으면 refresh 호출 → 새 토큰으로 원 요청 재시도
- 프론트: Access 수명 단축 (15분)

**완결성**: 사용자는 한참 작업해도 갑자기 로그아웃되지 않음.

**남는 약점**: 동시 요청 처리 미흡 / Refresh 탈취 감지 불가 / 새로고침 후에도 localStorage 의존.

---

### 10-3. 3단계 — 동시 401 큐잉

**목표**: 페이지 로드 시 여러 요청이 동시에 401을 받을 때 refresh가 중복 호출되는 문제 해결.

#### "동시 401"이 왜 발생하는가

페이지 하나를 새로고침하면 그 화면에 필요한 데이터들을 위해 API 요청 여러 개가 거의 동시에 출발한다. 예를 들어 메인 페이지 진입 시:

```
거의 동시에 출발:
- GET /api/v1/users/me
- GET /api/v1/scores/monthly
- GET /api/v1/scores/high
- GET /api/v1/friends
```

이 요청들은 모두 메모리에 들어 있는 **같은 Access Token**을 헤더에 달고 나간다. 그 토큰이 마침 직전에 만료됐다면 4개 요청이 전부 401을 받고 돌아온다.

#### 큐잉이 없으면 무슨 일이 벌어지는가

응답 인터셉터(2단계)는 401을 보면 "갱신하자"로 동작한다. 4개가 동시에 401을 받으면:

```
요청 A 401 → refresh 호출 (1번째)
요청 B 401 → refresh 호출 (2번째)
요청 C 401 → refresh 호출 (3번째)
요청 D 401 → refresh 호출 (4번째)
```

갱신이 4번 중복으로 나가는 것 자체도 낭비지만, **4단계 Rotation과 만나면 사고가 난다**. Rotation은 "Refresh를 한 번 쓰면 즉시 무효화"이므로:

```
1번째 refresh: Refresh #1 사용 → 새 토큰 발급. #1 무효화
2번째 refresh: Refresh #1 사용 → "이미 쓴 토큰" → 탈취 의심으로 전체 로그아웃
```

정상 사용자가 새로고침 한 번에 강제 로그아웃된다.

#### 큐잉 처리 방식

프론트 axios 응답 인터셉터에 **진행 플래그 + 대기 큐** 두 가지를 둔다.

```
요청 A 401 → "내가 첫 번째다" → 플래그 ON, refresh 시작
요청 B 401 → "갱신 중이네" → 큐에 줄 서기
요청 C 401 → 큐에 줄 서기
요청 D 401 → 큐에 줄 서기

[refresh 완료, 새 Access 받음]

→ A 재시도 + 큐의 B, C, D를 새 토큰으로 한 번에 재시도
→ 4개 모두 성공
```

refresh 실패 시에는 큐의 요청들도 모두 실패 처리하고 로그아웃 화면으로 보낸다.

#### 프론트 단독 작업

3단계는 **프론트만 손대면 끝난다**. 백엔드 코드는 2단계에서 만든 그대로다.

이유: "내 탭 메모리 안의 같은 토큰을 단 4개 요청이 동시에 출발했다"는 사실은 브라우저 한 개 안에서만 알 수 있는 정보다. 백엔드 입장에서는 그냥 "어떤 클라이언트가 만료된 토큰으로 4번 요청했네" 정도이고, 누가 보냈는지 묶을 단서도 없다. 따라서 "갱신 1번으로 묶어라"는 조정은 요청을 만들어 보내는 쪽, 즉 프론트의 axios 인터셉터에서만 가능하다.

#### 구현 요지

```js
// src/lib/api.js (개념)
let isRefreshing = false;
let waitingQueue = [];

api.interceptors.response.use(null, async (error) => {
  if (error.response?.status !== 401) throw error;

  if (isRefreshing) {
    // 갱신 중이면 큐에서 대기
    return new Promise((resolve, reject) => {
      waitingQueue.push({ resolve, reject, config: error.config });
    });
  }

  isRefreshing = true;
  try {
    await refreshAccessToken();
    // 큐에 쌓인 요청들을 새 토큰으로 한 번에 재시도
    waitingQueue.forEach(({ resolve, config }) => resolve(api(config)));
    waitingQueue = [];
    return api(error.config);
  } catch (e) {
    waitingQueue.forEach(({ reject }) => reject(e));
    waitingQueue = [];
    logout();
    throw e;
  } finally {
    isRefreshing = false;
  }
});
```

**완결성**: 다중 요청 환경에서 안정적으로 동작. 실서비스 품질 도달. 4단계 Rotation 도입의 전제 조건이기도 함.

---

### 10-4. 4단계 — Refresh Token 회전 (Rotation)

**목표**: Refresh 탈취를 빠르게 감지.

- 백엔드: refresh 호출마다 **새 Refresh를 같이 발급, 옛 Refresh는 무효화**
- 백엔드: 이미 사용된 Refresh가 다시 들어오면 → 해당 사용자의 모든 Refresh 무효화 + 알림
- 프론트: 변경 사항 없음 (쿠키 자동 갱신)

**완결성**: 보안 수준 향상. 탈취 사고 발생 시 피해 차단 가능.

---

### 10-5. 5단계 — 새로고침 대응 정리 (localStorage 의존 제거)

**목표**: 1단계에서 타협한 localStorage 보관을 제거.

- 프론트: 앱 시작 시 **자동으로 `/api/v1/auth/refresh` 호출** → Refresh 쿠키가 있으면 Access를 메모리에 복원
- localStorage에서 토큰 제거
- 로그인 상태 판별은 "메모리에 Access가 있는가"로 통일

**완결성**: XSS 위협 표면 최소화. 보안 권장 형태 도달.

---

### 10-6. 6단계 — 선제 갱신 (UX 개선)

**목표**: 긴 폼 작성 중 401로 요청이 한 번 깨지는 미세한 UX 문제 제거.

- 프론트: 요청 직전 토큰의 `exp`를 보고 만료 60~120초 전이면 미리 갱신
- 클럭 스큐 마진 30~60초

**완결성**: 사용자가 401 → 재시도 지연을 한 번도 체감하지 않음. 있으면 좋고 없어도 무방.

---

### 10-7. 7단계 — 다중 탭 동기화

**목표**: 여러 탭에서 작업하는 사용자의 토큰 상태 일관성.

- 프론트: BroadcastChannel API로 갱신/로그아웃 이벤트 방송
- 다른 탭이 받아 메모리 동기화

**완결성**: 탭 다수 사용 시나리오에서 갑작스런 로그아웃 사고 방지.

---

### 10-8. 8단계 — 모바일 PWA / 앱 래퍼 대응

**목표**: 모바일 앱 도입 시 쿠키 한계 우회.

- iOS Keychain / Android Keystore에 토큰 저장
- 환경 분기: 웹은 쿠키, 앱은 네이티브 저장소

**완결성**: 모바일 환경에서도 동일 인증 로직 사용 가능.

---

### 권장 정지 지점

- **시간 매우 부족**: 1단계까지. localStorage 세션 시절보다 크게 나쁘지 않고 JWT 인프라 확보.
- **시간 약간 있음**: 3단계까지. 실서비스 품질의 자동 갱신.
- **권장 목표**: 5단계까지. 보안 권장 사항 충족. 이후는 "있으면 좋은 것".
- 6~8단계: 필요·여유 생길 때마다 추가.
