## 논의 배경

이번달 랭킹(`GET /api/v1/monthly-scores`) 조회 성능 개선을 위해 Redis 도입을 논의함.

현재 구조:
```

  └── PostgreSQL: monthly_scores 테이블 날짜 범위 필터 + score 내림차순 정렬 (매 요청마다 풀스캔)
```

## 논의 내용

### Redis Sorted Set vs TTL 캐시

두 가지 방식을 비교 검토함.

| 방식 | 특징 | 적합한 상황 |
|------|------|------------|
| **TTL 기반 캐시** | 랭킹 결과를 JSON으로 Redis에 저장, 만료 시 DB 재조회 | 구현 단순, 약간의 딜레이 허용 가능 |
| **Redis Sorted Set** | Redis가 랭킹 데이터를 직접 관리, 점수 변경 즉시 반영 | 실시간 반영 필수, 대규모 트래픽 |

### Redis Sorted Set은 캐시가 아님

- 캐시: 원본(DB)의 임시 복사본, TTL로 만료
- Redis Sorted Set: Redis 자체가 주 저장소 역할, DB와 동기화되는 별도 데이터 계층 (이중 쓰기 패턴)

## 결정 사항

### Phase 1: TTL 기반 캐시 (즉시 구현)
- `GET /api/v1/monthly-scores` 응답을 Redis에 캐시
- TTL: 30초
- 캐시 키: `monthly_scores:current_month`
- 캐시 무효화 없음 — TTL 만료 시 자연 갱신

**캐시 무효화 제거 이유:**
- 100명이 동시에 게임하면 점수 등록마다 캐시가 무효화되어 캐시 효과가 사라짐
- 랭킹은 30초 딜레이가 UX에 큰 영향 없음
- TTL만으로 충분히 실용적

### Phase 2: Redis Sorted Set (추후 전환)
- Phase 1의 Redis 인프라를 그대로 재활용
- 저장 방식만 JSON blob → Sorted Set으로 변경
- Phase 1에서 캐시 무효화 지점을 잘 설계해두면 전환 용이

## 영향 범위

| 엔드포인트 | 역할 | 캐시 영향 |
|-----------|------|----------|
| `GET /api/v1/monthly-scores` | 랭킹 조회 | 캐시에서 반환 (TTL 30초) |
| `POST /api/v1/monthly-scores` | 점수 upsert | 영향 없음 (캐시 무효화 안 함) |
| `PUT /api/v1/monthly-scores/{user_id}` | 점수 수정 | 영향 없음 (캐시 무효화 안 함) |
| `DELETE /api/v1/monthly-scores/{user_id}` | 점수 삭제 | 영향 없음 (캐시 무효화 안 함) |
