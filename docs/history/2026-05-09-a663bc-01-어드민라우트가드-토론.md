# 어드민 라우트 가드 토론

- 작성일: 2026-05-09
- 주제 ID: a663bc

## 배경

프론트엔드에서 어드민 페이지(`/admin`, `/admin/users`, `/admin/statistics`)에 진입할 때, **로그인한 사용자의 role이 `admin`인 경우에만** 접근을 허용하고자 한다. 사용자에게 "라우트 가드"는 이번이 첫 도입이며, 백엔드 어드민 API 측 검증도 현재는 적용되어 있지 않을 가능성이 높다.

관련 선행 결정:
- [2026-05-05-c022f5-01-프론트엔드JWT관리-토론.md](2026-05-05-c022f5-01-프론트엔드JWT관리-토론.md)
  - Access Token은 메모리(Zustand)에만 저장
  - JWT payload에 `{sub, email, role, nickname}` 포함 (Stateless 검증)
  - **권한 판단의 최종 책임은 백엔드. 프론트 UI 분기는 편의용.**

## 용어 정리

### 라우트 가드(Route Guard)

라우트 element를 보호 컴포넌트로 감싸 조건 미충족 시 다른 라우트로 리다이렉트하는 패턴.

```jsx
<Route path="/admin" element={
  <AdminRoute>
    <AdminUserPage />
  </AdminRoute>
} />
```

`AdminRoute` 내부에서 "Access Token 보유 여부", "role === 'admin'" 등을 검사하고, 조건 불충족 시 `<Navigate to="..." replace />`로 다른 페이지로 보낸다.

### 가드의 역할 분담

- **프론트 가드 = UX 보호**: 권한 없는 사용자가 `/admin`에 접근했을 때 적절한 페이지로 안내. 데이터 자체를 막는 장치는 아니다.
- **백엔드 검증 = 보안의 본체**: 어드민 API가 admin role을 검증해야 데이터 노출이 차단된다. 프론트 가드만 있으면 화면은 가려져도 API 직접 호출로 데이터를 빼낼 수 있다.

## 개념 토대 — 왜 프론트 가드는 "검증"이 아니라 "안내"인가

이번 작업은 사용자에게 라우트 가드 도입이 처음이므로, 이후 결정의 근거가 되는 개념을 명확히 정리한다.

### 1. 프론트는 민감 정보를 다루지 않는다 (원칙)

- SPA의 JS 번들은 누구나 다운로드 가능 → 리버스 엔지니어링으로 코드 전부 노출 가능
- 브라우저 DevTools로 메모리 상태, 네트워크 요청, 토큰 값 모두 들여다볼 수 있음
- 전제: **"프론트에 있는 것은 비밀이 아니다."**

### 2. 프론트 가드는 접근 차단(UX)만 담당한다

- 권한 없는 사용자에게 어드민 화면을 보이지 않게 하는 안내 장치
- 사용자가 토큰을 위조해 가드를 통과해도 문제없음 — 어차피 데이터가 없으므로
- 예: 일반 user가 실수로 `/admin`을 쳐도 `/`로 자연스럽게 돌려보냄

### 3. 민감 정보는 API 응답으로만 전송된다

- 사용자 목록, 통계, 권한 정보 등 보호 대상 데이터는 백엔드 API 응답에만 존재
- 프론트는 "받아서 화면에 그리는" 역할만 수행

### 4. 백엔드에서 JWT 위조 검증 + 권한 체크

- **서명 검증**: `JWT_SECRET_KEY`로 토큰이 우리가 발급한 것임을 증명 (위조 차단)
- **권한 체크**: payload의 `role`이 `'admin'`인지 확인
- 이 두 가지가 통과해야 데이터 응답

### 5. 결과적으로 프론트 가드 검증 방식은 "payload 디코딩만으로 충분"

| 방식 | 서버 호출 | 즉시성 | 일반적 채택 |
|------|-----------|--------|-------------|
| (A) payload 디코딩 | 0회 | 즉시 | ✅ 표준 |
| (B) 라우트 진입마다 서버 검증 | 매번 1회 | 라운드트립 지연 | 드묾 (특수 사례 한정) |

서명 검증에 필요한 `JWT_SECRET_KEY`는 서버에만 있어야 하므로 프론트는 서명 검증 자체를 할 수 없다. 그러나 위조된 토큰으로 가드를 통과해도 백엔드 검증이 막아주므로, 프론트는 서명 검증을 **할 필요도 없다.**

### 핵심 한 줄

> **"클라이언트는 신뢰하지 않는다. 토큰의 서명만 신뢰한다."**
> 이것이 JWT stateless 인증의 본질이며, 프론트 가드는 이 모델 위에서 UX 안내 역할만 맡는다.

### 비유

은행 창구에 비유하면:

- **프론트 가드** = 대기실 입구의 안내원. "어드민 전용 라운지는 저쪽이에요"라고 안내. 안내원을 무시하고 들어가도 막을 권한은 없다.
- **백엔드 검증** = 금고 앞 직원. 카드(JWT) 진위 확인 + 권한 확인. 여기를 못 통과하면 절대 돈을 꺼낼 수 없다.
- **위조 시나리오** = 가짜 신분증으로 라운지에는 들어가도, 금고 앞 검증에서 출금 불가.

라운지가 뚫려도 금고는 안 뚫린다 — 이것이 두 층 분리의 핵심.

### 2. 가드 컴포넌트 형태

#### 결정 포인트와 채택안

| 포인트 | 후보 | 채택 |
|--------|------|------|
| 가드 분할 | 단일형 / 분리형 / 일반화(`roles` props) | **단일형 `<AdminRoute>`** |
| 사용 방식 | element 감싸기 / Outlet 패턴 | **Outlet 패턴** (자식 라우트 묶음 보호) |
| 파일 위치 | `components/` / `components/guards/` / `pages/admin/` | **`frontend/src/components/AdminRoute.jsx`** |
| JWT 디코딩 | `jwt-decode` 라이브러리 / 직접 구현 | **`jwt-decode` 라이브러리** |

#### 채택 근거

- **단일형**: 이번이 첫 가드 도입이고 보호 대상이 어드민 한 묶음뿐. 분리/일반화는 향후 일반 사용자 전용 라우트(`/user/account` 등)나 다중 role이 생길 때 그때 도입. 미리 추상화하지 않는다.
- **Outlet 패턴**: `/admin`, `/admin/users`, `/admin/statistics` 3개 라우트를 한 번에 묶어 보호. 반복 줄어듦.
- **`components/` 직배치**: 가드 1개 시점에 `guards/` 폴더는 과함. 늘어나면 그때 옮긴다.
- **`jwt-decode`**: base64url 패딩, 예외 처리 등의 미묘한 함정을 라이브러리가 처리. 번들 크기 약 1KB로 가볍다.

#### 사용 형태 (개념)

```jsx
// frontend/src/App.jsx
<Route element={<AdminRoute />}>
  <Route path="/admin" element={<AdminUserPage />} />
  <Route path="/admin/users" element={<AdminUserPage />} />
  <Route path="/admin/statistics" element={<AdminStatisticsPage />} />
</Route>
```

```jsx
// frontend/src/components/AdminRoute.jsx (개념)
import { Navigate, Outlet } from 'react-router-dom';
import { jwtDecode } from 'jwt-decode';
import { useAuthStore } from '../store/authStore';

export default function AdminRoute() {
  const accessToken = useAuthStore((s) => s.accessToken);
  if (!accessToken) return <Navigate to="/login" replace />;

  try {
    const { role } = jwtDecode(accessToken);
    if (role !== 'admin') return <Navigate to="/" replace />;
  } catch {
    return <Navigate to="/login" replace />;
  }

  return <Outlet />;
}
```

> 위 코드는 형태 합의용 스케치. role 소스(1번 항목)와 새로고침 타이밍(4번 항목) 결정에 따라 세부 조정.

### 3. 비인가 사용자 리다이렉트 정책

#### 채택안

- **비로그인 / 일반 user 모두 `/`로 통일 리다이렉트.**
- **토스트·알림 메시지 노출 없음.** 조용히 보낸다.

#### 근거

- 가드는 본질적으로 "안내" 역할이므로 사용자 추궁이나 강조 알림이 어울리지 않는다.
- 비로그인/일반 user를 분기해 다른 곳으로 보낼 실익이 크지 않다 — `/`에 도달하면 자연스럽게 로그인 상태에 맞는 화면이 보이므로 사용자 혼란이 적다.
- 단순한 정책이 유지보수에도 유리.

#### 가드 코드 반영 (확정형)

```jsx
// frontend/src/components/AdminRoute.jsx
import { Navigate, Outlet } from 'react-router-dom';
import { jwtDecode } from 'jwt-decode';
import { useAuthStore } from '../store/authStore';

    if (role !== 'admin') return <Navigate to="/" replace />;
  } catch {
    return <Navigate to="/" replace />;

  return <Outlet />;
}
```

## 결정 사항

### D-1. 작업 방향

**어드민 페이지에 프론트 라우트 가드를 도입하고, 검증은 프론트에서 JWT payload를 디코딩해 `role === 'admin'`을 확인하는 방식으로 한다.**

- 서버에 라우트 진입마다 검증을 묻지 않는다 (라운드트립 비용 대비 실익 없음).
- 백엔드 어드민 API의 admin 권한 검증은 별도 항목으로 점검·적용한다 (5번 토론 항목).
- 위조 토큰으로 가드를 통과하더라도 데이터 노출이 없도록 백엔드 검증을 반드시 갖추는 것이 전제.

### D-2. 가드 컴포넌트 형태

- **단일형 `<AdminRoute>`** 컴포넌트로 시작 (로그인 + role 검사 한꺼번에).
- **Outlet 패턴**으로 어드민 라우트 묶음 전체를 보호.
- 파일 위치: **`frontend/src/components/AdminRoute.jsx`** (가드 폴더 분리는 가드가 늘어나면 그때).
- JWT 디코딩: **`jwt-decode`** 라이브러리 사용.

### D-3. 비인가 사용자 리다이렉트 정책

- **비로그인 / 일반 user / 토큰 디코딩 실패 — 모두 `/`로 통일 리다이렉트.**
- 토스트·알림 노출 없음 (조용히 보냄).

### B. 구현 전 알아야 할 사실

#### B-1. Access Token 저장 위치

- 메모리(Zustand 스토어)에만 저장. localStorage/쿠키에는 없음.
- **스토어 위치**: `frontend/src/store/authStore.js` (예상 — 실제 파일명·구조 확인 후 사용)
- **인터셉터(React 바깥)에서 토큰 꺼낼 때**: `useAuthStore.getState().accessToken`
- **컴포넌트(React 안)에서 토큰 꺼낼 때**: `useAuthStore((s) => s.accessToken)` — 변경 시 자동 리렌더링됨. 가드는 컴포넌트이므로 이 형태를 쓴다.

> ⚠️ **시작 전 확인 필수**: `frontend/src/store/authStore.js`의 실제 export 이름, 필드명(`accessToken`이 맞는지)을 먼저 읽어볼 것. 이 인계서의 추정과 다를 수 있음.

#### B-2. JWT payload 구조

서버는 다음 필드를 payload에 넣어 발급한다 (선행 토론 12장 결정):

```
{ sub, email, role, nickname, exp }
```

- `sub`: user_id (문자열)
- `role`: `'user'` 또는 `'admin'`
- 가드에서 사용하는 필드: **`role` 단 하나**

#### B-3. 가드 = UX 안내

- **이 가드는 보안의 본체가 아니다.** 사용자가 토큰을 위조해 통과해도 무관.
- 데이터 보호는 백엔드 API의 admin 의존성이 담당(이번 작업 범위 외, 후속 항목).
- 따라서 가드에서 JWT 서명 검증을 하지 않는다. **payload 디코딩만 한다.**
- 본문 "개념 토대" 섹션 참고.

#### B-4. 새로고침 직후의 한순간 — 알고 넘어가기

앱 시작 시 메모리 토큰은 비어 있고, `/api/v1/auth/refresh`로 복원된다. 그 복원이 완료되기 전에 사용자가 곧바로 `/admin`에 들어와 있는 상태라면, 가드는 "토큰 없음"으로 판정해 `/`로 보낼 수 있다.

**이 문제는 이번 작업에서 해결하지 않는다** (후속 작업으로 분리). 즉 현재 가드는 "토큰이 메모리에 있을 때만 정확히 동작"하는 단순 형태로 둔다. 새로고침 시 잠깐 잘못된 리다이렉트가 나는 건 알면서 두는 트레이드오프.

> 만약 구현 중 이 문제를 같이 풀고 싶어진다면 **사용자에게 먼저 물어볼 것.** (CLAUDE.md / memory 참조: "계획서 범위 외 수정 금지")

### C. 구현 단계

#### C-1. 패키지 설치

```bash
source ~/.nvm/nvm.sh && cd frontend && npm install jwt-decode
```

#### C-2. `frontend/src/components/AdminRoute.jsx` 신규 작성

```jsx
import { Navigate, Outlet } from 'react-router-dom';
import { jwtDecode } from 'jwt-decode';
import { useAuthStore } from '../store/authStore';

export default function AdminRoute() {
  const accessToken = useAuthStore((s) => s.accessToken);
  if (!accessToken) return <Navigate to="/" replace />;

  try {
    const { role } = jwtDecode(accessToken);
    if (role !== 'admin') return <Navigate to="/" replace />;
  } catch {
    return <Navigate to="/" replace />;
  }

  return <Outlet />;
}
```

체크포인트:
- import 경로의 store 파일명·export 이름이 실제와 맞는지 확인.
- `react-router-dom`, `jwt-decode`, store 모두 이미 다른 곳에서 쓰이고 있을 것 (store는 확실, 나머지는 import 형태 확인).

#### C-3. `frontend/src/App.jsx` 수정

**현재** ([App.jsx:33-35](../../frontend/src/App.jsx#L33-L35)):

```jsx
<Route path="/admin" element={<AdminUserPage/>} />
<Route path="/admin/users" element={<AdminUserPage/>} />
<Route path="/admin/statistics" element={<AdminStatisticsPage/>} />
```

**변경 후**:

```jsx
<Route element={<AdminRoute />}>
  <Route path="/admin" element={<AdminUserPage/>} />
  <Route path="/admin/users" element={<AdminUserPage/>} />
  <Route path="/admin/statistics" element={<AdminStatisticsPage/>} />
</Route>
```

상단에 `import AdminRoute from './components/AdminRoute';` 추가.

> ⚠️ `/admin`과 `/admin/users` 중복 매핑은 **유지**한다. 사용자 의도일 가능성이 있어 후속 작업에서 별도 검토하기로 했음. 임의로 정리하지 말 것.

### D. 검증 시나리오

브라우저에서 직접 확인할 것:

1. **비로그인 상태로 `/admin` 접속** → `/`로 리다이렉트되어야 함
2. **일반 user role로 로그인 후 `/admin` 접속** → `/`로 리다이렉트되어야 함
3. **admin role로 로그인 후 `/admin` 접속** → 어드민 페이지 정상 표시
4. **admin 상태로 `/admin/users`, `/admin/statistics` 접속** → 모두 정상
5. **admin 상태에서 어드민 내부 이동** → 매번 정상 (가드가 매 진입마다 통과해야 함)
6. **admin 상태로 페이지 새로고침** → 토큰 복원 전 잠깐 `/`로 튕길 수 있음. **이는 알려진 한계로 이번 작업 범위 밖.**

### E. 테스트 작성

기존 프로젝트가 Vitest 기반 테스트를 사용 중 (`frontend/src/test/`).

**최소 테스트 권장**: `frontend/src/test/AdminRoute.test.jsx`

테스트 케이스:
- accessToken이 없으면 `/`로 Navigate
- accessToken이 있고 role이 'admin'이면 자식 라우트(Outlet) 렌더
- accessToken이 있고 role이 'user'면 `/`로 Navigate
- 잘못된 토큰(디코딩 실패)이면 `/`로 Navigate

`useAuthStore`는 모킹, `jwtDecode`도 필요 시 모킹.

**테스트 우선순위**: 사용자가 TDD를 강제하지 않았으므로 필수는 아님. 다만 기존 테스트 컨벤션을 따라 작성하는 것을 권장. 시간 제약이 있으면 생략 가능하나 사용자에게 확인할 것.
