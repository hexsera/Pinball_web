# ACL 설계 논의

## 목적

RBAC 유무와 관계없이, ACL 본연의 개념과 구현 방법을 이해한다.

---

## ACL 핵심 개념

```
Resource → [(Actor, Permission)]
게시글#42 → [(홍길동, read), (홍길동, write), (김철수, read)]
게시글#99 → [(홍길동, read)]
```

리소스마다 접근 허용 목록을 따로 가진다. "이 리소스에 대해 이 사람이 뭘 할 수 있는가"가 핵심.

---

## 가장 단순한 ACL 데이터 모델

```
acl_entries
┌───────────────┬─────────────┬──────────────┬────────────┐
│ resource_type │ resource_id │ actor_id     │ permission │
├───────────────┼─────────────┼──────────────┼────────────┤
│ "post"        │ 42          │ 1 (홍길동)   │ "write"    │
│ "post"        │ 42          │ 2 (김철수)   │ "read"     │
│ "file"        │ 7           │ 1 (홍길동)   │ "delete"   │
└───────────────┴─────────────┴──────────────┴────────────┘
```

체크 로직:
```python
def can(actor_id, permission, resource_type, resource_id) -> bool:
    return db.query(AclEntry).filter_by(
        actor_id=actor_id,
        permission=permission,
        resource_type=resource_type,
        resource_id=resource_id,
    ).exists()
```

---

## ACL이란?

가장 직관적인 권한 관리 방법. 파일 시스템(유닉스)에서 기원.  
리소스 하나하나마다 "누가 뭘 할 수 있는가"를 명시적으로 저장한다.

```bash
# 리눅스 파일 ACL 예시
getfacl somefile.txt
# user:홍길동:rw-
# user:김철수:r--
```

웹 서비스에서는 파일 경로 대신 API 리소스가 대상이 된다.  
`/api/posts/42` → `("post", 42)` 로 쪼개서 저장하는 이유:  
문자열 파싱 없이 `resource_id` 만으로 바로 조회 가능.

---

## ACL 데이터 모델 (직접 설계)

```
acl_entries
┌────┬──────────┬───────────────┬─────────────┬────────────┐
│ id │ actor_id │ resource_type │ resource_id │ permission │
└────┴──────────┴───────────────┴─────────────┴────────────┘
```

- `id`: 같은 유저가 여러 리소스에 대한 항목을 가질 수 있어 별도 PK 필요
- `actor_id`: 권한을 부여받는 사용자
- `resource_type`: 리소스 종류 ("post", "file" 등)
- `resource_id`: 리소스 식별자
- `permission`: 허용할 조작 ("read", "write", "delete" 등)

---

## permission 값 관리

문자열로 저장하면 개발자마다 `"write"` / `"Write"` / `"쓰기"` 불일치 문제 발생.

해결책: 코드 레벨 + DB 레벨 둘 다 강제.
- 코드: enum 타입
- DB: enum 타입 또는 check constraint

코드에만 enum을 쓰면 부족한 이유: mock 데이터 삽입 등 DB에 직접 접근하는 경우 코드를 거치지 않아 실수 가능.

---

## ACL 항목 생성 시점

리소스 생성 API에서 자동으로 acl_entries 레코드를 함께 생성.  
기본 규칙 예시: 게시글 작성자에게 write, delete 허용.

주의: 리소스 INSERT와 acl_entries INSERT는 반드시 하나의 트랜잭션으로 묶어야 함.  
그렇지 않으면 리소스는 존재하는데 ACL이 없는 상태가 발생할 수 있음.

---

## 레코드 수 문제와 해결 패턴

게시글 10만 개 * 유저 1만 명이면 acl_entries가 폭발적으로 늘어남.  
"모든 유저가 읽을 수 있는" 같은 기본 케이스를 레코드로 다 저장하면 비효율적.

### 패턴 1: default-allow

ACL에 없으면 기본 정책 적용. 예외만 저장.  
- ACL에 없으면 → 기본 허용
- ACL에 있으면 → 명시적 규칙 적용

### 패턴 2: `*` actor

`actor_id`에 특정 유저 대신 `"*"` 를 넣어 "모든 사람"을 표현.

```
actor_id | resource_type | resource_id | permission
"*"      | "post"        | 42          | "read"      ← 모든 사람 읽기 허용
1        | "post"        | 42          | "write"     ← 작성자만 수정 허용
```

비밀글로 전환 시 `*` 레코드만 삭제하면 됨. 유저 수만큼 레코드 불필요.

## 체크 로직과 우선순위

`*` actor 도입 후 체크 순서가 중요해짐.  
**명시적 규칙(특정 유저)이 기본 규칙(`*`)보다 우선한다.**

```
"*"      | "read"      ← 모든 사람 읽기 허용
2 (김철수)| "deny_read" ← 김철수는 읽기 거부
```

위 상황에서 `*` 부터 확인하면 김철수도 허용되어버림.  
올바른 순서:

1. 본인 레코드 먼저 확인 → 있으면 해당 규칙 적용
2. 없으면 `*` 레코드 확인

```python
def can(actor_id, permission, resource_type, resource_id) -> bool:
    # 1. 본인 레코드 먼저
    entry = db.query(AclEntry).filter_by(
        actor_id=str(actor_id),
        resource_type=resource_type,
        resource_id=resource_id,
        permission=permission,
    ).first()
    if entry:
        return entry.permission != "deny_read"

    # 2. * 레코드 확인
    return db.query(AclEntry).filter_by(
        actor_id="*",
        resource_type=resource_type,
        resource_id=resource_id,
        permission=permission,
    ).exists()
```

## ACL 테이블 구조 선택

**테이블 하나** vs **리소스마다 별도 테이블** 두 가지 방법이 있음.

| | 테이블 하나 | 테이블 여러 개 |
|---|---|---|
| 장점 | 종합 조회 쉬움, 관리 단순 | 조회 속도 빠름 |
| 단점 | 테이블 커짐, 느려질 수 있음 | 파편화, 종합 조회 어려움 |

실무에서는 **테이블 하나 + 인덱스** 조합을 주로 씀.  
`resource_type`, `resource_id`, `actor_id` 조합에 인덱스를 걸어 속도 문제 해결.

테이블 구조 선택이 **리소스 삭제 시 고아 레코드 처리 방식**에도 영향을 줌.

| | 테이블 하나 | 테이블 여러 개 |
|---|---|---|
| ON DELETE CASCADE | 불가 (resource_id가 여러 테이블 가리킴) | 가능 |
| 고아 레코드 처리 | 코드에서 직접 삭제 | DB가 자동 처리 |

## ACL 항목 변경 API

```
POST   /api/acl     ← 권한 부여 (resource_type, resource_id, actor_id, permission)
DELETE /api/acl/:id ← 권한 회수
PUT    /api/acl/:id ← 권한 수정 (바뀔 permission)
```

## 소유자(owner) 개념

ACL 수정 API는 아무나 호출하면 안 됨 — 소유자만 권한을 부여/회수할 수 있어야 함.

소유자를 acl_entries에 일반 레코드로 넣으면 실수로 삭제될 수 있음.

**방법 1: 코드에서 소유자 레코드 보호**
- DELETE/PUT 시 해당 actor_id가 소유자면 차단
- 다른 리소스에도 적용할수록 코드가 늘어남

**방법 2: is_owner 컬럼 추가**
- acl_entries에 `is_owner` 컬럼으로 구분
- 테이블 관리가 복잡해짐

**방법 3: 리소스 테이블에 owner_id 컬럼 추가 (권장)**
```
posts
┌────┬──────────┬──────────┐
│ id │ content  │ owner_id │
└────┴──────────┴──────────┘
```
소유자는 ACL 대상이 아니라 리소스 자체에 귀속.  
ACL 수정 API에서 `owner_id == current_user` 체크 한 번으로 보호.

## ACL 정리 (본인 언어로)

ACL은 리소스에 누가 어떤 접근 권한을 가지고 있냐에 대한 방법.  
필드는 사용자 id, 리소스 타입, 리소스 id, 권한으로 이루어져 있고  
권한은 보통 enum 값으로 관리한다.  
모든 사용자에게 공개하는 처리는 사용자 수만큼 레코드를 추가해야 하는 문제가 있어 `*` 같은 특수값을 쓴다.  
직급(역할) 같은 그룹 단위 권한 처리는 어렵다 — 해당 역할에 속한 사람 수만큼 레코드를 넣어야 하기 때문. 이 한계를 해결하는 게 RBAC.
