|------------|----------|-----------|------|

### 식별된 위험

1. **무인증 정보 조회**: `GET /users`, `GET /users/{id}`로 비로그인 상태에서 사용자 정보 노출
2. **무인증 사용자 생성 + role 임의 지정**: `POST /users`로 누구나 admin 가입 가능
3. **본인/admin 체크 부재**: `PUT`/`DELETE`는 인증은 하지만 "수정·삭제 대상이 본인인가? 또는 호출자가 admin인가?"를 묻지 않음 → 수평 권한 상승 취약

## 참고: user API 권한 설계의 표준 패턴

이번 작업에서 "전부" 도입하지는 않더라도, 향후 정리·확장의 기준으로 삼을 수 있도록 표준 패턴을 기록한다.

### 핵심 원칙 3가지

1. **Authentication ≠ Authorization** — 로그인 여부 확인(인증)과 "이 행동을 할 권한이 있나"(권한)는 분리된 두 층이다.
2. **본인 자원에 대한 자동 권한** — 사용자는 자기 데이터에 대해서는 기본 권한 보유.
3. **admin은 모든 자원에 대한 권한** — admin은 자기/타인 구분 없이 통과.

이 셋이 합쳐져 가장 흔히 표현되는 규칙이:

> **"본인 자원이거나, admin이거나"**

### 표준 권한 매트릭스

| 엔드포인트 | 비로그인 | 일반 user (본인) | 일반 user (타인) | admin |
|-----------|----------|------------------|------------------|-------|
| `POST /users` (가입) | ✅ | — | — | ✅ |
| `GET /users` (목록) | ❌ | ⚠️ 제한적 | ⚠️ 제한적 | ✅ 전체 |
| `GET /users/{id}` | ❌ | ✅ 본인 풀필드 | ⚠️ 공개 필드만 | ✅ 전체 |
| `PUT /users/{id}` | ❌ | ✅ 본인 | ❌ | ✅ |
| `DELETE /users/{id}` | ❌ | ✅ 본인 | ❌ | ✅ |
| `GET /users/me` | ❌ | ✅ | — | ✅ |

해석:
- **가입**은 비로그인이 호출해야 하므로 인증 면제. 단, **`role`은 서버가 강제로 `'user'`로 박아야 함** (요청에서 받지 않음).
- **목록 조회**는 보통 admin 전용 또는 검색 필드 강하게 제한.
- **단건 조회**는 본인은 풀필드, 타인은 nickname 등 공개 필드만 (또는 admin만 허용).
- **수정·삭제**는 본인 또는 admin.
- **`/users/me`** 라는 별도 엔드포인트로 "내 정보"를 분리하는 것이 관례.

### 표준 패턴 1 — 권한 의존성 3종 세트

라우터는 권한 로직을 가지지 않고, 의존성 함수를 끼우기만 한다.

```
# deps.py (의사코드)

def get_current_user(token):
    payload = verify_jwt(token)
    if invalid: raise 401
    return payload   # { sub, email, role, ... }

def require_admin(current_user = Depends(get_current_user)):
    if current_user.role != 'admin':
        raise 403
    return current_user

def require_self_or_admin(user_id, current_user = Depends(get_current_user)):
    is_self  = (current_user.sub == user_id)
    is_admin = (current_user.role == 'admin')
    if not (is_self or is_admin):
        raise 403
    return current_user
```

라우터 적용 (의사코드):

```
POST   /users                                     # 인증 없음, role은 서버가 'user'로 강제
GET    /users           require_admin
GET    /users/me        get_current_user
GET    /users/{id}      require_self_or_admin(id)  # 또는 require_admin
PUT    /users/{id}      require_self_or_admin(id)
DELETE /users/{id}      require_self_or_admin(id)
```

장점: 라우터 본문이 비즈니스 로직만 남고, 권한 규칙은 한 곳에서 관리.

### 표준 패턴 2 — `/users/me` 분리

본인 정보는 `{id}` 경로가 아닌 `/me` 경로로 다룬다.

```
GET    /users/me     → 토큰의 sub로 자동 조회. user_id 파라미터 없음
PUT    /users/me     → 본인 정보 수정
DELETE /users/me     → 본인 탈퇴

GET    /users/{id}, PUT/DELETE /users/{id} → admin 전용 또는 제한적
```

이유:
- 클라이언트가 자기 id를 알 필요 없음 — 코드 단순화.
- 권한 분기가 사라짐 — `/me`는 무조건 "토큰 주인", `/{id}`는 admin 전용.
- URL이 의도를 드러냄.
- GitHub, Stripe, Slack 등 주요 서비스가 채택하는 패턴.

### 표준 패턴 3 — 가입 경로 분리

```
POST /auth/register   → 회원가입 (인증 면제, role 'user' 강제)
POST /users           → admin이 사용자 생성 (admin 전용)
```

이유:
- "가입"은 인증(auth) 영역, 사용자 관리(users)와 구분.
- `POST /users`는 admin 운영용으로 자연스럽게 분리됨.

### 표준 패턴 4 — 응답 스키마 권한별 분기

같은 데이터라도 호출자 권한에 따라 노출 필드를 다르게.

```
def get_user(id, current_user):
    user = db.find(id)
    if current_user.is_admin or current_user.id == user.id:
        return UserFullResponse(user)    # email, role, birth_date 등 전부
    else:
        return UserPublicResponse(user)  # nickname 등 공개 필드만
```

또는 타인 조회 자체를 차단(403/404)하는 정책도 흔함 — 정책 선택의 문제.

### 현재 프로젝트와의 격차 요약

| 표준 항목 | 현재 상태 |
|-----------|-----------|
| 권한 의존성 분리 (`require_admin` 등) | ❌ `get_current_user`만 있음 |
| `/users/me` 엔드포인트 | ❌ 없음 |
| 가입 경로 분리, role 강제 | ❌ `POST /users`로 role 임의 지정 가능 |
| 응답 스키마 권한별 분기 | ❌ 없음 |
| `GET /users` 보호 | ❌ 누구나 접근 |

### 2. 권한 검증 구현 방식

#### 채택: 의존성 함수 신설

`backend/app/api/deps.py`에 권한 의존성 함수를 신설하고, 라우터는 `Depends(...)`로 끼우기만 한다.

#### 처리 흐름 (모든 보호 엔드포인트 공통)

```
[요청 도착]
  ↓
1. JWT 추출 + 서명 검증           (get_current_user — 기존)
  ↓
2. 권한 의존성에서 role/소유 검사  (require_admin 또는 require_self_or_admin — 신설)
  ↓
3. 라우터 본문 진입 — 비즈니스 로직만
```

#### 신설 의존성

```python
# backend/app/api/deps.py

def require_admin(current_user = Depends(get_current_user)):
    if current_user["role"] != "admin":
        raise HTTPException(403, "Admin required")
    return current_user

def require_self_or_admin(
    user_id: int,
    current_user = Depends(get_current_user),
):
    is_self  = int(current_user["sub"]) == user_id
    is_admin = current_user["role"] == "admin"
    if not (is_self or is_admin):
        raise HTTPException(403, "Forbidden")
    return current_user
```

#### 라우터 적용

| 엔드포인트 | 의존성 |
|------------|--------|
| `POST /users` | (없음) |
| `GET /users` | `Depends(get_current_user)` |
| `GET /users/{user_id}` | `Depends(get_current_user)` |
| `PUT /users/{user_id}` | `Depends(require_self_or_admin)` |
| `DELETE /users/{user_id}` | `Depends(require_self_or_admin)` |

#### 채택 사유

- 권한 규칙이 라우터 시그니처 한 줄로 드러나 가독성 ↑
- 같은 규칙을 여러 라우터에서 재사용 가능, 변경 시 한 군데만 수정
- 의존성 단독 테스트 / mock 갈아끼우기 용이
- FastAPI OpenAPI 스펙에 인증 요구사항이 자동 반영
- 라우터 본문은 비즈니스 로직만 남음 (관심사 분리)

### 3. `POST /users`의 role 임의 지정 처리

#### 결정: 이번 작업에 포함

#### 사유

권한 시스템의 신뢰 기반 확보. `require_admin`을 도입해도 누구나 admin으로 가입 가능하면 가드는 무력화된다. 가드 작업과 짝지어 함께 처리.

#### 위험 (도입 전 현재 상태)

```json
POST /api/v1/users
{
  "email": "evil@example.com",
  "password": "...",
  "nickname": "evil",
  "birth_date": "...",
  "role": "admin"        ← 회원가입 한 번에 admin
}
```

요청 body의 `role` 값을 그대로 DB에 저장하고 있어, 외부에서 admin 계정 생성이 가능.

#### 처리 방식

- **요청 스키마(`UserCreateRequest`)에서 `role` 필드 제거.**
- **라우터에서 `role='user'` 고정 대입.**
- 영향: 정상 회원가입은 그대로 동작 (클라이언트가 보내든 안 보내든 무시되고 'user' 고정).
- admin 계정 생성 경로는 별도(DB 직접 / admin이 어드민 페이지에서 다른 사용자의 role 변경 등) — 본 작업 범위 외.

### D-1. 적용 권한표

#### 무엇을 정했나

users 라우터 5개 엔드포인트의 권한 규칙:

- `POST /users` — 비로그인 허용 (회원가입)
- `GET /users` — 로그인 사용자면 누구나
- `GET /users/{id}` — 로그인 사용자면 누구나
- `PUT /users/{id}` — 본인 또는 admin
- `DELETE /users/{id}` — 본인 또는 admin

#### 왜 이렇게 정했나

**원칙**: 표준 패턴인 "본인 자원이거나, admin이거나" + "Authentication ≠ Authorization"의 두 층 분리 구조를 따른다(본 문서 "참고: 표준 패턴" 섹션 참조). 다만 시간 제약상 응답 스키마 분기, `/users/me` 분리, 회원가입 경로 분리 등은 후속 작업으로 미루고 **"권한 검증을 일단 모든 엔드포인트에 도입"** 만 이번 범위로 한정.

엔드포인트별 사유:

- **`POST /users`** — 회원가입은 비로그인이 호출해야 하는 본질적 특성. 인증 면제는 표준. 권한 통과는 무조건이지만 `role` 임의 지정 차단(D-2)을 짝지어 신뢰 기반 확보.
- **`GET /users`** — 처음에는 "admin 전용" 보수적 안을 추천했으나, 호출처 점검(본 문서 "호출처 점검 결과" 섹션)에서 친구 검색([FriendPage.jsx:154](../../frontend/src/pages/FriendPage/FriendPage.jsx#L154))이 일반 user로 호출하고 있어 admin 전용으로 잠그면 친구 기능이 깨짐을 확인. 친구 검색 보존을 우선하고 응답 풀필드 노출 문제는 후속 작업으로 분리하기로 합의.
- **`GET /users/{id}`** — 점수판 닉네임 표시, 친구 프로필 조회 등 일반 용도가 있어 보수적 잠금(본인/admin)은 과함. 로그인 가드만 거는 유연한 정책.
- **`PUT /users/{id}` / `DELETE /users/{id}`** — "본인 자원이거나, admin"의 표준. 일반 user가 타인 정보를 변경/삭제할 수 있는 현 구조는 명백한 수평 권한 상승 취약점이라 이번 작업에서 반드시 잠근다.

#### 트레이드오프 (수용한 위험)

- **`GET /users`, `GET /users/{id}` 응답에 이메일 등 풀필드가 그대로 나간다.** 로그인한 사용자라면 누구나 다른 사용자의 이메일을 조회 가능. 이는 권한 검증 작업의 본질 범위 밖이고, 응답 스키마 분기는 별도 후속 작업이라 분명히 분리.
- **친구 검색의 권한 폭이 넓다.** 일반 user가 nickname 파라미터 없이 호출해도 막지 않음. 이번 작업에서는 "권한 잠금 단순화"를 우선했고, 검색 파라미터 강제 등은 후속 작업.

#### 영향 범위

- 백엔드: `backend/app/api/v1/users.py`만 수정.
- 프론트엔드: 영향 없음 (호출처 점검에서 깨질 곳 없음 확인).

### D-2. POST /users의 role 서버 강제

#### 무엇을 정했나

- `UserCreateRequest` 스키마에서 `role` 필드 제거.
- 라우터에서 `role='user'`로 고정 대입.

#### 왜 이렇게 정했나

현재 구조는 클라이언트가 회원가입 요청 body의 `role` 필드에 `"admin"`을 넣어 보내면 그대로 admin으로 가입된다. **권한 시스템 자체의 신뢰 기반이 무너지는 결함.**

이번 작업으로 `require_admin` 등 권한 의존성을 도입하더라도, "**누구나 admin이 될 수 있다**"면 가드는 무력. 따라서 **권한 시스템과 짝으로 함께 처리**하는 것이 논리적으로 분리 불가능한 한 묶음.

#### 트레이드오프

- **admin 계정 생성 경로가 일시적으로 좁아진다.** 회원가입으로는 더 이상 admin 생성 불가. 새 admin이 필요할 때는 DB 직접 수정, 환경 변수 시드, 또는 기존 admin이 어드민 페이지에서 다른 사용자의 role을 변경하는 경로만 남는다. 이는 보안적 정상 구조이며, 별도 admin 생성 API가 필요하면 후속 작업으로 분리.

#### 영향 범위

- 스키마: `backend/app/schemas/user.py`의 `UserCreateRequest`.
- 라우터: `backend/app/api/v1/users.py`의 `create_user`.
- 프론트엔드: 회원가입 폼이 role을 보내도 무시되므로 영향 없음(별도 확인 필요).

### D-3. 권한 검증 구현 방식 — 의존성 함수 신설

#### 무엇을 정했나

`backend/app/api/deps.py`에 두 의존성 함수 신설:

- `require_admin` — admin role 강제
- `require_self_or_admin(user_id)` — 토큰 sub와 경로 user_id 일치 또는 admin

라우터는 `Depends(...)`로 끼우기만 한다.

#### 왜 이렇게 정했나

inline if문 방식과 비교했을 때 의존성 분리가 갖는 이점:

1. **권한 규칙이 라우터 시그니처 한 줄로 보인다** — 코드 안 읽고도 권한 파악 가능.
2. **재사용** — 같은 규칙을 여러 라우터에서 사용. 변경 시 한 곳만 수정.
3. **테스트 분리** — 의존성 단독 테스트 가능. 라우터 테스트에서는 mock 갈아끼우기 용이.
4. **OpenAPI 자동 반영** — FastAPI가 의존성을 보고 인증 요구사항을 스펙에 박아준다. inline if문은 못 잡음.
5. **관심사 분리** — 라우터 본문은 비즈니스 로직만, 권한은 별도 계층.

또한 사용자가 직관적으로 그린 처리 흐름(JWT 위조 검증 → role/소유 검사 → 비즈니스 로직)이 그대로 의존성 체인에 매핑됨 — 사고 모델과 코드 구조가 일치.

#### 트레이드오프

- **새 함수 2개 추가**로 deps.py에 코드 줄 수 증가. inline 대비 코드 양은 비슷하나 개념 분리가 명확.

#### 영향 범위

- 신규: `backend/app/api/deps.py`에 함수 2개 추가.
- 수정: `backend/app/api/v1/users.py` 5개 라우터 시그니처에 `Depends(...)` 추가/변경.
- 기존 `get_current_user`는 그대로 유지 (선행 토론 c022f5의 12장 Stateless 결정 그대로).

### B. 구현 전 알아야 할 사실

#### B-1. 기존 코드 구조

- `backend/app/api/deps.py:22-36`에 `get_current_user`가 이미 존재. JWT 서명 검증 후 payload(dict)를 반환. **DB 조회 없는 Stateless 방식.** 이번 작업은 이 함수를 그대로 활용.
- payload 구조: `{"sub": "<user_id 문자열>", "email": ..., "role": ..., "nickname": ..., "exp": ...}`
- `sub`은 **문자열**이므로 정수 비교 시 `int()` 변환 필요.

#### B-2. 현재 스키마 상태 (`backend/app/schemas/user.py`)

- `UserCreateRequest` 에 `role: str` 필드가 있음 — 이번 작업으로 **제거**.
- **참고**: `UserRegisterRequest`라는 role 없는 스키마가 이미 존재한다. 그러나 라우터(`POST /users`)가 사용 중인 것은 `UserCreateRequest` 쪽. 이번 작업에서는 **`UserCreateRequest`에서 role 필드를 제거하는 방향**으로 통일한다 (별도 회원가입 스키마로 분리하는 것은 후속 작업의 회원가입 경로 분리와 함께 다룬다).
- `UserUpdateRequest`의 `role: Optional[str]`은 이번 작업에서 **건드리지 않는다**. 후속 작업 항목으로 분리.

#### B-3. 권한 모델 핵심

- "**본인 자원이거나, admin이거나**"
- 본인 판정: `int(payload["sub"]) == user_id`
- admin 판정: `payload["role"] == "admin"`

#### B-4. 미충족 시 응답 정책 (D-4)

- 비인증(토큰 없음/위조): `get_current_user`가 이미 **401** 반환.
- 인증은 됐으나 권한 부족: 신규 의존성에서 **403** 반환.

#### B-5. 기존 라우터 코드와의 차이점 — 주의

`PUT`, `DELETE` 라우터는 이미 `current_user: User = Depends(get_current_user)` 를 받고 있다. 그러나 **타입 힌트가 `User`로 되어 있으나 실제 반환은 dict.** 이는 Stateless 전환 후 잔존한 표기 오류로 보임. 이번 작업에서 의존성을 `require_self_or_admin`으로 바꾸면서 타입 힌트도 `dict`로 정정한다 (또는 그냥 `=Depends(...)` 형태로 받고 라우터 본문에서 사용 안 하면 타입 힌트 생략).

라우터 본문에서 현재 `current_user`를 실제로 쓰는 곳이 없는지 한 번 더 확인하고, 안 쓰면 변수명을 `_auth` 같은 underscore 변수로 받아 lint 경고 회피.

### C. 구현 단계

#### C-1. `backend/app/api/deps.py` 수정 — 의존성 함수 2개 추가

기존 파일 끝에 추가:

```python
def require_admin(current_user: dict = Depends(get_current_user)) -> dict:
    """admin role만 허용"""
    if current_user.get("role") != "admin":
        raise HTTPException(status_code=status.HTTP_403_FORBIDDEN, detail="Admin required")
    return current_user


def require_self_or_admin(
    user_id: int,
    current_user: dict = Depends(get_current_user),
) -> dict:
    """경로의 user_id와 토큰 sub가 일치하거나 admin인 경우만 허용"""
    is_self = int(current_user.get("sub", 0)) == user_id
    is_admin = current_user.get("role") == "admin"
    if not (is_self or is_admin):
        raise HTTPException(status_code=status.HTTP_403_FORBIDDEN, detail="Forbidden")
    return current_user
```

체크포인트:
- `Depends`, `HTTPException`, `status`는 파일 상단에 이미 import되어 있음 — 추가 import 불필요.
- `int(current_user.get("sub", 0))` — sub이 빠진 비정상 토큰을 0으로 폴백해 비교를 무력화. (정상 토큰이면 sub은 항상 존재)

##### C-3-2. `POST /users` — role 강제 대입

```python
@router.post("", response_model=UserResponse)
def create_user(
    user: UserCreateRequest,
    db: Session = Depends(get_db)
):
    """회원가입"""
    existing_user = db.query(User).filter(User.email == user.email).first()
    if existing_user:
        raise HTTPException(
            status_code=status.HTTP_400_BAD_REQUEST,
            detail="Email already registered"
        )

    db_user = User(
        email=user.email,
        nickname=user.nickname,
        password=hash_password(user.password),
        birth_date=user.birth_date,
        role="user",          # ← 서버가 강제 대입 (변경 부분)
    )
    db.add(db_user)
    db.commit()
    db.refresh(db_user)

    return db_user
```

##### C-3-5. `PUT /users/{user_id}` — 본인 또는 admin

```python
@router.put("/{user_id}", response_model=UserResponse)
def update_user(
    user_id: int,
    user_update: UserUpdateRequest,
    db: Session = Depends(get_db),
    _auth: dict = Depends(require_self_or_admin),  # ← 변경 (기존 get_current_user → require_self_or_admin)
):
    user = db.query(User).filter(User.id == user_id).first()
    if user is None:
        raise HTTPException(
            status_code=status.HTTP_404_NOT_FOUND,
            detail=f"User with id {user_id} not found"
        )

    if user_update.email and user_update.email != user.email:
        existing_user = db.query(User).filter(User.email == user_update.email).first()
        if existing_user:
            raise HTTPException(
                status_code=status.HTTP_400_BAD_REQUEST,
                detail="Email already registered"
            )

    update_data = user_update.model_dump(exclude_unset=True)
    for field, value in update_data.items():
        if value is not None:
            setattr(user, field, value)
            if field == "nickname":
                month = db.query(MonthlyScore).filter(MonthlyScore.user_id == user_id).all()
                for i in month:
                    i.nickname = value

    db.commit()
    db.refresh(user)
    return user
```

> 변수명을 `current_user`에서 `_auth`로 바꾼 것은 라우터 본문에서 사용하지 않기 때문(타입 정정 + lint 경고 회피). 본문 로직 자체는 그대로.

##### C-3-6. `DELETE /users/{user_id}` — 본인 또는 admin

```python
@router.delete("/{user_id}", response_model=DeleteResponse)
def delete_user(
    user_id: int,
    db: Session = Depends(get_db),
    _auth: dict = Depends(require_self_or_admin),  # ← 변경
):
    user = db.query(User).filter(User.id == user_id).first()
    if user is None:
        raise HTTPException(
            status_code=status.HTTP_404_NOT_FOUND,
            detail=f"User with id {user_id} not found"
        )

    db.query(GameVisit).filter(GameVisit.user_id == user_id)\
        .update({"user_id": None}, synchronize_session=False)

    db.query(Friendship).filter(
        (Friendship.requester_id == user_id) | (Friendship.receiver_id == user_id)
    ).delete(synchronize_session=False)

    db.query(MonthlyScore).filter(MonthlyScore.user_id == user_id)\
        .delete(synchronize_session=False)

    db.delete(user)
    db.commit()

    return DeleteResponse(
        message="User deleted successfully",
        deleted_user_id=user_id
    )
```
