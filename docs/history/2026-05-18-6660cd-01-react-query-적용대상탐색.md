## 목적

공지사항 캐시 논의 중 React Query가 공지사항에는 부적합하다는 결론이 나왔다. React Query는 같은 데이터를 여러 컴포넌트가 동시에 구독할 때 가장 효과적이므로, 프로젝트 내에서 그 조건에 맞는 적용 대상을 찾는다.

## React Query 가 할 수 있는 것

### 1. 중복 요청 제거
같은 queryKey를 가진 요청이 여러 컴포넌트에서 동시에 발생하면 하나로 합쳐서 보낸다. 헤더, 사이드바, 본문이 같은 API를 동시에 쓰는 경우 특히 유용.

### 2. 자동 리페치
- 탭을 다시 포커스했을 때 (`refetchOnWindowFocus`)
- 네트워크가 재연결됐을 때 (`refetchOnReconnect`)
- 일정 주기마다 (`refetchInterval`) — 실시간성이 필요한 데이터에 적합

### 3. staleTime / gcTime 으로 캐시 수명 제어
- `staleTime` — 이 시간 동안은 캐시를 신선하다고 보고 재요청 안 함
- `gcTime` — 컴포넌트가 언마운트된 후 캐시를 메모리에 얼마나 유지할지

### 4. 캐시 무효화 (`invalidateQueries`)
특정 이벤트(글쓰기, 삭제, 수정) 후 관련 캐시를 즉시 날려서 다음 접근 시 최신 데이터를 받아오게 한다.

### 5. 로딩 / 에러 상태 자동 관리
`isPending`, `isError`, `error` 등을 자동으로 제공. `useEffect` + `useState`로 직접 관리하던 보일러플레이트를 제거.

### 6. 페이지네이션 / 무한스크롤
- `keepPreviousData` — 페이지 이동 시 이전 데이터를 유지해 깜빡임 방지
- `useInfiniteQuery` — 무한스크롤 구현에 특화

### 7. Optimistic Update
서버 응답을 기다리지 않고 UI를 먼저 업데이트한 뒤, 실패 시 롤백. 좋아요, 팔로우 같이 즉각적인 피드백이 필요한 곳에 적합.

### 8. 의존성 기반 쿼리
다른 쿼리의 결과가 나와야 실행되는 쿼리를 `enabled` 옵션으로 제어.
```js
useQuery({ queryKey: ['profile'], enabled: !!userId })
```

## React Query 가 가장 빛나는 조건

- 같은 데이터를 **여러 컴포넌트가 동시에 구독**할 때
- **쓰기 후 즉시 무효화**가 필요할 때
- **자동 리페치**로 실시간성을 유지해야 할 때
- `useEffect` + `useState` 보일러플레이트가 반복될 때

## 적용 후보

### 1. UserInfo 페이지 (`/user/account`)

`GET /users/{id}` 로 유저 정보를 가져오고 `PUT /users/{id}` 수정 후 화면을 갱신하는 흐름. 현재는 수정 성공 후 `setUserInfo`로 수동 업데이트하고 있는데, React Query를 쓰면 `invalidateQueries`로 자동 갱신 가능. `birth_date`처럼 토큰에 없는 데이터를 API로 가져오므로 캐시 가치 있음.

### 2. FriendPage (`/user/friend`)

승인/거절 후 현재는 `fetchPendingRequests()`, `fetchFriendList()`를 수동으로 재호출하고 있음. `invalidateQueries`로 자동 갱신하기에 딱 맞는 패턴. 친구 목록 조회 후 각 친구의 월간 점수를 `Promise.all`로 병렬 호출하는 부분도 `useQuery`로 캐시하면 목록이 안 바뀌는 한 재요청 불필요.

| 현재 | React Query 적용 후 |
|--|--|
| 승인/거절 후 수동으로 fetch 재호출 | `invalidateQueries`로 자동 갱신 |
| 친구 점수를 매번 새로 받아옴 | `useQuery`로 캐시, 변경 없으면 재사용 |
| 로딩/에러 state를 직접 관리 | `isPending`, `isError` 자동 제공 |

### 3. HomePage 월간 랭킹 (`/`) ✅ 완료 (2026-05-19)

현재 로딩 상태가 없어서 데이터 오기 전에 랭킹 테이블이 빈 채로 보임.

**적용할 것**
- `isPending` — 로딩 중 스켈레톤이나 스피너 표시
- `staleTime` — 짧은 시간 안에 재방문 시 캐시 재사용, 재요청 생략
- `refetchInterval` — 주기적 자동 재요청. 월간 랭킹 특성상 실제로는 과한 설정이지만, 학습 목적으로 Network 탭에서 동작을 직접 체감해보기 위해 적용

### 4. AdminUserMain (`/admin/users`)

유저 수정/삭제 후 테이블을 수동으로 재호출하고 있고, 재로딩 시 1페이지로 돌아가는 UX 문제가 있음. React Query로 여러 가지 개선 가능.

| 문제 | React Query 적용 |
|--|--|
| 수정/삭제 후 `fetchUsers()` 수동 재호출 | `invalidateQueries`로 자동 갱신 |
| 재로딩 시 테이블이 1페이지로 돌아감 | `placeholderData: keepPreviousData`로 페이지 위치 유지 |
| 수정 다이얼로그 열 때 `GET /users/{id}` 중복 호출 | React Query 캐시에서 꺼내와 불필요한 요청 제거 |
| 재진입 시 매번 전체 유저 재요청 | `staleTime` 설정으로 짧은 시간 내 재진입 시 캐시 재사용 |

## 할 일

- 프로젝트 전체 API 호출 현황을 파악한다
- 여러 컴포넌트에서 동시에 구독하는 데이터가 있는지 확인한다
- 자동 리페치가 필요한 데이터가 있는지 확인한다
- React Query 적용이 적합한 대상을 선정한다
