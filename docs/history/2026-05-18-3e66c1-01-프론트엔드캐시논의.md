## 목적

취업용 학습 목적. 프로젝트에 직접 적용하면서 프론트엔드 캐시를 익힌다.

## 캐시의 핵심 목적

클라이언트-서버 간 트래픽 절감. 두 가지 방식:
- 불필요한 요청 자체를 없애기 — 클라이언트가 이미 가진 데이터 재사용
- 요청은 하되 전송량을 줄이기 — 서버가 304(변경 없음)만 응답, 바디 전송 생략

## 캐시 도입 적합 조건 (일반)

| 조건 | 설명 |
|------|------|
| 자주 읽히지만 잘 안 바뀜 | 읽기 빈도 높고 쓰기 빈도 낮을수록 효과 큼 |
| 생성 비용이 비쌈 | JOIN, 집계 쿼리 등 DB 부담 연산 |
| 같은 요청이 반복됨 | 여러 사용자 동일 요청(공용) 또는 한 사용자 단시간 반복(클라이언트) |
| 정적 리소스 | JS, CSS, 이미지 — 배포 전까지 변경 없음 |

반대로 캐시 하면 안 되는 곳: 실시간성 중요 데이터, 사용자별 민감 데이터, 쓰기가 읽기만큼 잦은 데이터

## 타겟 선정: 공지사항

### 후보 비교

| 후보 | 적합도 | 이유 |
|------|--------|------|
| 월간 랭킹 | 적합 | 조회 빈도 높음, 약간 오래된 데이터 허용 가능. 단 Redis라 서버 부담은 적음 |
| 어드민 전체 유저 목록 | 애매 | 유저 변동 빈도에 따라 다름. 캐시보다 수동 refetch 패턴이 더 적합할 수 있음 |
| **공지사항** | **최적** | 데이터 변경 드묾 + HTML 콘텐츠(이미지 포함 가능)로 무거움 + Admin만 쓰기/삭제라 무효화 타이밍 명확 |

### 공지사항 기능 현황

- 백엔드: `GET /api/v1/notices` (목록), `GET /api/v1/notices/{id}` (상세), POST/DELETE (Admin)
- 프론트엔드: `noticeService.js`에서 fetch 직접 호출. API 실패 시 IndexedDB fallback
- 콘텐츠: Tiptap 에디터로 작성한 HTML 저장. 이미지 포함 가능(Canvas 리사이징, 최대 800px)

### IndexedDB fallback과의 관계

현재 IndexedDB는 API 실패 시 mock 데이터 대체 용도. React Query 도입 시 역할 중복 발생 → 정리 방향 논의 필요.

## 학습 로드맵

```
1단계: React Query 도입 + 공지사항 목록/상세에 staleTime 적용
2단계: queryKey 설계 ['notices'] vs ['notices', id]
3단계: invalidateQueries — 글쓰기/삭제 후 목록 캐시 무효화
4단계: HTTP Cache-Control 헤더 — 이미지, 정적 리소스
5단계: 빌드 해시 + CDN 캐시 무효화 전략
```
